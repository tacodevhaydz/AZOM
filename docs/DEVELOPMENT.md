# Project Overview

SimHub plugin for MOZA Racing hardware providing two-way telemetry: streams game data (speed, RPM, gear, lap times, fuel, tyre wear, etc.) to the wheel dashboard display, drives wheel/dashboard RPM and flag LEDs, and allows configuring wheelbase settings. Uses a custom binary serial protocol reverse-engineered from the [boxflat](https://github.com/Lawstorant/boxflat) project.

## Building from Source

The project targets .NET Framework 4.8 (x86) and uses the `Microsoft.NETFramework.ReferenceAssemblies.net48` NuGet package so it can cross-compile on Linux without Mono. The built DLL runs on Windows under SimHub.

### Building on Windows

#### Prerequisites

- [VS Code](https://code.visualstudio.com/) with the [C# Dev Kit](https://marketplace.visualstudio.com/items?itemName=ms-dotnettools.csdevkit) extension
- .NET SDK 8.0+ ([download](https://dotnet.microsoft.com/download))

#### Steps

1. **Open the project folder** in VS Code.

2. **Build** from the VS Code terminal:

   ```
   dotnet build -c Release
   ```

3. **Copy the DLL** to your SimHub folder:

   Copy `bin/x86/Release/MozaPlugin.dll` into your SimHub installation directory.

   Or set the `SIMHUB_PATH` environment variable to have it copied automatically on build:

   ```
   set SIMHUB_PATH=C:\Program Files (x86)\SimHub
   dotnet build -c Release
   ```

   PowerShell:
   ```powershell
   $env:SIMHUB_PATH = "C:\Program Files (x86)\SimHub"
   dotnet build -c Release
   ```

4. **Restart SimHub.** The plugin appears under Settings > Plugins as "MOZA Control".

### Cross-Compiling on Linux

You can build the plugin entirely on Linux. The .NET SDK can target .NET Framework 4.8 using the `Microsoft.NETFramework.ReferenceAssemblies.net48` NuGet package (already included in the `.csproj`).

#### Prerequisites

- .NET SDK 8.0+ (`dotnet-sdk` package from your distro or [Microsoft repos](https://dotnet.microsoft.com/download))

#### Steps

1. **Install the .NET SDK** if you don't have it:

   ```bash
   # Arch Linux
   sudo pacman -S dotnet-sdk

   # Ubuntu/Debian
   sudo apt install dotnet-sdk-8.0

   # Fedora
   sudo dnf install dotnet-sdk-8.0
   ```

2. **Build:**

   ```bash
   dotnet build -c Release
   ```

   The output DLL will be in `bin/x86/Release/MozaPlugin.dll`.

3. **Copy the built DLL to your Windows SimHub installation:**

   ```bash
   # Example: copy to a Windows machine via scp
   scp bin/x86/Release/MozaPlugin.dll user@windows-pc:"C:/Program Files (x86)/SimHub/"

   # Or copy to a shared folder, USB drive, etc.
   cp bin/x86/Release/MozaPlugin.dll /mnt/shared/SimHub/
   ```

4. **Restart SimHub** on Windows.

#### Notes

- The `Microsoft.NETFramework.ReferenceAssemblies.net48` NuGet package provides the .NET Framework 4.8 reference assemblies, so you do **not** need Mono or Windows installed.
- The build produces a standard .NET Framework DLL that runs natively on Windows under SimHub.
- SimHub DLLs in `libs/SimHub/` are reference-only (`Private=false`) and not copied to output.
- The build produces a single output DLL with no additional runtime dependencies to deploy.

### Running Tests

The test project (`MozaPlugin.Tests/`) uses xUnit and targets `net9.0`. It includes the testable source files directly from the main project (via `<Compile Include>`) rather than using a `<ProjectReference>`, because the main project's net48 DLL has WPF and SimHub dependencies that cannot load under .NET 9 on Linux. Minimal stubs in `MozaPlugin.Tests/Stubs/` satisfy the two external types referenced by included source files (`GameReaderCommon.StatusDataBase` and `SimHub.Logging.Current`).

```bash
dotnet test -c Release
```

This works on both Linux and Windows.

### CI/CD

- **Build**: Every push to `main` and every PR is built automatically via GitHub Actions.
- **Release**: Pushing a `v*` tag (e.g., `v0.2.0`) builds a Release, generates a changelog, and publishes a GitHub Release with the DLL (device definitions are embedded in the DLL).
- **SimHub dependency updates**: A daily workflow checks for new SimHub releases and creates a PR to update `libs/SimHub/`.

## Architecture

### Component Layers

**Logging & Diagnostics** (`Diagnostics/`) — Shared infrastructure for plugin logging and the Diagnostics-tab export bundle:
- `MozaLog` — Static wrapper around `SimHub.Logging.Current` (`Info`/`Debug`/`Warn`/`Error`). Every plugin log call goes through this class instead of `SimHub.Logging.Current` directly so that, in addition to forwarding to SimHub's log4net pipeline, every `[Moza]` line is appended to a 5 000-entry in-process ring buffer with a millisecond local-time stamp and level. `Snapshot()` returns the buffer as a single string for the Diagnostics export — sidesteps SimHub's flush cadence and per-version log-path / file-extension differences (`Logs/SimHub.<n>.txt` on current builds). Lives in `namespace MozaPlugin` so all source files reach it without a using statement. Tests link the same file in via `MozaPlugin.Tests.csproj`; a `Stubs/SimHubLogging.cs` no-op satisfies the SimHub dependency under .NET 9.
- `SerialTrafficCapture` — Singleton ring buffer of timestamped serial frames in both directions, used by the Diagnostics tab's capture toggle. `RecordTx(label, frame)` and `RecordRx(label, frame)` are called from `MozaSerialConnection.WriteFrame` (after a successful port write) and `ReadLoop` (just before `MessageReceived?.Invoke`) respectively. Each `MozaSerialConnection` carries a `CaptureLabel` property (default "wheelbase"; AB9 manager flips its connection's label to "ab9") so a single export can distinguish frames from each pipe. Capacity is 200 000 entries — about 30–60 minutes of dense telemetry — with oldest-drop ring trimming so a long capture never exhausts memory. `Start()` clears the buffer and arms the per-frame fast path; `Stop()` returns the snapshot and disables the gate; `Format(entries)` renders to one-line-per-frame text with local timestamps and uppercase hex bytes.

**Plugin Entry Point** (`MozaPlugin.cs`) — Implements SimHub's `IPlugin`, `IDataPlugin`, `IWPFSettingsV2`. Manages lifecycle (Init/DataUpdate/End), connection state, auto-reconnect (5s timer), setting polling (2s timer), `TelemetrySender` for dashboard data streaming, and `DashboardProfileStore` for profile management. Uses phased per-device initialization: `TryConnect()` sends only lightweight detection probes, then `DetectDevices()` applies saved settings and reads back device state as each device (base, wheel, dashboard, handbrake, pedals, hub) is confirmed present. This is the orchestrator that wires everything together. **Reload safety:** `Init()` is wrapped in try/catch; on partial-init failure `CleanupPartialInit()` mirrors `End()` so SimHub's reload-without-restart path can't leak timers, serial ports, or HID readers. `Instance = this` is published only after every resource is wired so background callbacks can't observe a half-built plugin. `End()` symmetrically detaches `_connection.MessageReceived -= OnMessageReceived` and `_subscribedProfileStore.CurrentProfileChanged -= OnProfileChanged` (the subscribed store is tracked separately so `ClearSettings()` — which builds a fresh `MozaPluginSettings` — can detach the orphaned subscription before re-subscribing on the new store). `OnMessageReceived` and `PollStatus` short-circuit on `IsShuttingDown` so in-flight callbacks delivered after `End()` started can't touch disposed state. Property delegates registered with `AttachDelegate` are null-guarded (`_data?.X ?? default`) so SimHub's property polling is immune to partial-init / mid-teardown windows.

**Serial Protocol Layer** (`Protocol/`) — Binary protocol over USB serial at 115200 baud (VID `0x346E`):
- `MozaSerialConnection` — Device auto-discovery (WMI on Windows, probe-based under Wine/Proton or when System.Management can't load into SimHub's AppDomain), background read/write threads, frame assembly, full 0x7E byte stuffing (read + write — every 0x7E byte after the start+len header is doubled on the wire; without this, payload 0x7Es in zlib blobs / seq counters desync the read stream after ~15s and trigger the wheel-alive watchdog). Probe path is target-scoped via `MozaProbeTarget` (`BaseAndHub` for the wheelbase connection, `Ab9` for the AB9 shifter). `BaseAndHub` runs a two-pass probe: bases first (group 0x2B → expect response group 0xAB), then hubs (group 0x64 dev 0x12 → expect 0xE4) so hub-only setups are discovered. `Ab9` runs the AB9 identity probe (group 0x09 dev 0x12 → expect 0x89) — needed because `DeviceAb9` and `DeviceHub` share id 0x12, so a hub probe alone can't tell them apart. **Response disambiguation:** `ProbeMozaDevice` reads ≥3 wire bytes after the reply and verifies `buf[2] == expectedRespGroup` for the kind being probed; a 0x7E reply with the wrong group is logged at Debug and rejected. Pre-2026-04-28 the probe accepted any 0x7E reply, which mis-claimed AB9 ports as hubs (group 0x64 cmd 03 happens to elicit a 0x7E from the AB9). WMI enumeration failures are logged at Debug (`WMI enumeration failed: <Type>: <Message>`) — silent before, hiding AB9 detection failures on Windows installs where `System.Management` couldn't be loaded
- `MozaCommand` — Message builder with big-endian int encoding
- `MozaCommandDatabase` — 150+ pre-built command definitions, including PitHouse-style extended wheel identity probes (`wheel-presence` group 0x09, `wheel-device-presence` 0x02, `wheel-device-type` 0x04, `wheel-capabilities` 0x05, `wheel-mcu-uid` 0x06 (12-byte STM32 UID), `wheel-hw-sub` 0x08 cmd=02, `wheel-identity-11` 0x11 cmd=04) fired by `MozaDeviceManager.SendPithouseIdentityProbe()` at wheel detect for full 12-frame PitHouse parity
- `MozaResponseParser` — Decodes responses using bit-7 toggle, nibble swap, and wildcard matching. Unwraps Display sub-device identity responses (0xC3 group + inner 0x8X response byte) into `display-*` command names (`display-model-name`, `display-hw-version`, `display-sw-version`, `display-serial`, `display-mcu-uid`, `display-device-type`, `display-capabilities`, `display-presence`, `display-device-presence`, `display-identity-11`). Silent-drops SerialStream control frames (0xC3 + `7C/FC 00` session-open/data wrappers + 0x7C 0x23 dashboard-activate advertisements) so they don't spam unmatched-log
- `MozaProtocol` — Constants (start byte `0x7E`, device IDs, checksum formula: `(13 + sum) % 256`). Two checksum helpers: `CalculateChecksum()` (raw sum, for legacy/test use) and **`CalculateWireChecksum()`** (raw sum + `count(0x7E in body positions 2..len-1) × 0x7E`). Production code uses `CalculateWireChecksum` on both send and verify paths so frames with 0x7E in body validate correctly — before this fix (pre-2026-04-22), ~20% of zlib-bearing session chunks (configJson state, dashboard uploads) failed verify and were silently dropped. The static probe builder (`MozaSerialConnection.BuildProbe`) also uses the wire variant; current probe payloads contain no 0x7E so the values coincide, but the wire variant is forward-correct if a future probe template needs a 0x7E byte. Also exposes `WheelEchoPrefixes` + `IsWheelEcho(data)`: the set of `(group, device, payload-prefix)` tuples the wheel echoes back on LED/brightness/page-config writes. `MozaPlugin.OnMessageReceived` short-circuits recognized echoes — marks wheel alive + skips unmatched-log noise. Mirrors `sim/wheel_sim.py:_WHEEL_ECHO_PREFIXES`
- `MozaHidReader` — Reads physical input positions (steering, throttle, brake, clutch, paddles, handbrake) and button states from Moza HID devices using HidSharp. Enumerates devices by name regex (matching boxflat/foxblat patterns), opens per-device receiver threads, and writes normalized values to `MozaData`. Tracks axis usages (GenericDesktop page) and button usages (Button page 0x0009). Handbrake button events are routed to a dedicated `HandbrakeButtonPressed` field — detected either by device name (standalone HBP USB device) or by button index 121 (base-attached handbrake). Also matches the Universal Hub HID device (`gudsen universal hub`). Independent of the serial protocol — provides live axis and button data for UI display at 30 Hz

**Device Management** (`MozaDeviceManager.cs`) — High-level read/write API. Handles wheel device ID cycling (IDs 23→21→19) since ES wheels respond on different IDs. `ResetWheelDetection()` must be called on disconnect so detection probes are re-sent on reconnect. `ReadSettingsPaced()` runs a batch of reads on a background task with an extra ~10ms gap between enqueues — used for large startup bursts (30+ wheel setting reads) that the wheel would otherwise drop, without throttling steady-state telemetry.

**Data Model** (`Telemetry/MozaData.cs`) — Thread-safe storage (~80 volatile fields, including string identity fields) for all device values including base, wheel, dashboard, handbrake, pedals, hub, and HID physical input positions. Per-button `WheelButtonDefaultDuringTelemetry[14]` flags request the live-telemetry path to replace incoming 'off' (0,0,0) button colors with that button's configured static color from `WheelButtonColors` (see `MozaLedDeviceManager.Display()`). (steering angle, pedal axes, paddle axes, handbrake). `IsConnected` is true when any Moza device (base or hub) is confirmed on the serial bus — use this as the "can I send commands?" guard. `IsBaseConnected` is the narrower flag for base-specific features (FFB, temps). Hub port power fields track connection status of each hub port. Includes HID button state tracking: `ButtonStates[128]` bool array for wheel buttons with a `ButtonCount` high-water mark, and a separate `HandbrakeButtonPressed` flag for the handbrake's button-mode output. `UpdateFromCommand()` maps parsed responses to fields; `UpdateFromArray()` handles color/timing byte arrays with per-branch length checks (colors require 3+ bytes, identity strings accept any length). Extended identity fields captured from PitHouse-style probes: `WheelHwSubVersion`, `WheelSubDeviceCount`, `WheelDevicePresence`, `WheelMcuUid[12]` (STM32 UID — likely PitHouse's dashboard-sync cache key), `WheelDeviceType`, `WheelCapabilities`, `WheelIdentity11`. Display sub-device parallels: `DisplayModelName`, `DisplayHwVersion`, `DisplaySwVersion`, `DisplaySerialNumber`, `DisplayMcuUid[12]`, `DisplayDeviceType`, `DisplayCapabilities`, `DisplayIdentity11`, `DisplaySubDeviceCount`, `DisplayDevicePresence` — populated when `ConfigJsonClient`/response parser receives the 10 response variants wrapped in 0x43 group from the display sub-module.

**Dashboard Telemetry Streaming** (`Telemetry/`) — Streams game data to the wheel dashboard display via a multi-tier binary protocol:
- `TelemetrySender` — Timer-based sender with a multi-phase startup matching PitHouse's observed sequence. Also pushes an empty-state tile-server blob on session 0x03 after tier definition via `SendTileServerState()` (matches PitHouse — captures show wheel never pushes back on 0x03, direction is host→wheel only). Exposes `TileServerState : TileServerState?` for UI diagnostics panel binding. Sequence: (0) **host session opens** — first sends type=0x00 end markers on sessions 0x01..0x10 to reclaim any stale sessions left by a prior SimHub crash, then directly opens 0x01 (mgmt) and 0x02 (telem = FlagByte) with type=0x81, waiting up to 500ms for each fc:00 ack. Proceeds with PitHouse defaults if acks don't arrive. The previous 48-port probe was removed — the write-queue burst it caused saturated the line long enough for the wheel-alive watchdog to reset the handshake mid-flight. Session 0x03 (aux) is opened fire-and-forget later. (1) **device-initiated session intake** — real wheel opens sessions 0x04/0x06/0x08/0x09/0x0A on its own side ~40–400 ms after host opens 0x01/0x02 on direct wheelbase, ~2.1 s when behind Universal Hub; sender acks each incoming type=0x81 with `openSeq = data[6]|data[7]<<8` echoed in an fc:00 ACK, routes subsequent 7c:00 data to per-session handlers. (2) **dashboard upload (session 0x04)** — waits up to 3 s for device open, then if the active mzdash's MD5 doesn't match any `WheelState.EnabledDashboards[].hash` (skip-if-unchanged), sends sub-msg 1 (path registration) + sub-msg 2 (file content) + type=0x00 end marker via `FileTransferBuilder`/`DashboardUploader`, with waits for device echoes between sub-messages. After the END marker the sender sleeps 500 ms so the wheel's post-upload dir-listing refresh chunks on session 0x04 arrive while the preamble handler is still subscribed. Hub-attached wheels are detected at port-probe time via `MozaSerialConnection.HubProbeSucceeded` (set true when the hub probe — group 0x64, dev 0x12, cmd 0x03 — returned a reply on the chosen COM port). When set, `SendHubSlotEnumeration()` fires PitHouse's 5-slot Form B burst (`7E 03 64 12 01 NN 00 [chk]` for NN=1..5) right after sessions open, ~300 ms before tier-def. Hub responds with `7E 03 E4 21 01 NN VV [chk]` per slot (VV = device-type code on that port; 0x00 = empty). Burst is gated to hub-only setups so direct-wheelbase doesn't waste 5 frames. (3) **configJson RPC (session 0x09)** — `ConfigJsonClient` parses the device's state blob (TitleId, enableManager dashboards, imageRefMap, etc.) as it arrives; on first complete blob the sender replies once with a `configJson()` canonical library list so the wheel's `configJsonList` picks up our profile set. (4) **channel-catalog quiet wait + tier definition + Display probe** — `WaitForChannelCatalogQuiet(200, 2000)` blocks until the wheel's pre-tier-def tag-0x04 channel burst on session 0x02 has been idle for 200 ms (max 2 s); `SendTierDefinition` intersects the active `MultiStreamProfile` with `_catalogParser.Catalog` via `FilterProfileToCatalog` (drops channels the wheel didn't advertise, drops empty tiers, falls back to unfiltered if everything is dropped), then sends the tier def. Format depends on `ProtocolVersion` (resolved from `MozaPluginSettings.TelemetryFirmwareEra`): v2 = sub-msg 1 preamble + compact numeric tier defs via `TierDefinitionBuilder`, v0 = URL subscription via `BuildV0UrlSubscription`. Then probes Display sub-device via 0x43 identity. (5) **preamble** (~1s) — acks incoming 7c:00 channel data with fc:00, buffers for channel catalog parsing, detects Display from 0x87 response; for v0 re-sends URL subscription (PitHouse double-sends). At preamble end the catalog is parsed into `_catalogParser.Catalog` (full URLs) but the `OnMessageDuringPreamble` handler **stays subscribed** so session 0x04 dir-refresh, 0x09 configJson updates, and 0x0a RPC replies keep flowing during the active phase. (6) **active** — `0x40` channel config burst (channel enables, dashboard state queries 28:00/28:01, mode 28:02), `0x41` enable signal (~30 Hz), `7d:23` telemetry per tier (30ms/500ms/2000ms), `0x2D` sequence counter, mode frames, dash/wheel keepalives (0x43 to 0x14/0x15/0x17), display config (7C:27) + dashboard activate (7C:23) interleaved per page, session acks. Two configurable settings: `MozaFirmwareEra` (Auto / `TierDefV2_Upload8B` / `TierDefV2_Upload6B` / `TierDefV0_Upload8B` / `TierDefV0_Upload6B` — drives both tier-def variant and `FileTransferWireFormat` for dashboard uploads, see `Telemetry/MozaFirmwareEra.cs`) and `FlagByteMode` (0=zero-based, 1=session-port, 2=two-batch; v2 only). On sub-msg 1 ack timeout the upload wire format auto-flips once (Legacy2025_11 ↔ New2026_04) for one retry — the `Auto` era relies on this to recover from a wrong default guess. Telemetry deferred until wheel is detected (`StartTelemetryIfReady`), which dispatches `Start()` to a background thread so the serial read thread stays free to deliver fc:00 ack responses during port probing. `Start()` is serialized via an `Interlocked.CompareExchange` guard on `_startInProgress` — concurrent calls (e.g. rapid Test-button double-click routing through the ThreadPool) short-circuit with a warning log instead of racing, which previously leaked orphan `System.Timers.Timer` instances still subscribed to `OnTimerElapsed` and multiplied the tick rate. `Start()` also checks `_state == TelemetryState.Idle` between phases so a concurrent `Stop()` exits cleanly. (See `TelemetryState` enum at the top of `Telemetry/TelemetrySender.cs` for the lifecycle states; replaced the prior 3-boolean soup `_enabled` / `_preambleComplete` / `_dashSwitchMuted`.) `OnTimerElapsed` is guarded by `_tickInProgress` (also `Interlocked.CompareExchange`): `System.Timers.Timer` fires on ThreadPool and queues concurrent invocations if a handler overruns its interval, which races `_tickCounter`/`_slowCounter`/`_modeCounter` and fires non-coalesced one-shot frames (heartbeat, display_cfg) 2–3× the intended rate. Dropping overlapping ticks is safe because stream-lane traffic is latest-wins coalesced and re-covered by the next tick. `Pause()` / `Resume()` toggle the timer without tearing down session state — used by the UI Test Stop button when telemetry remains enabled so the wheel goes quiet within ~1 ms instead of continuing at tier-0 rate from the still-running timer. `FlushPendingOutput()` exposes the connection's `FlushPendingWrites()` so the Stop button also drops any in-flight stream slots + OS write buffer. `Stop()` resets `FramesSent` + session state AND unsubscribes the session handler so disable→re-enable performs a full fresh startup. **`Stop()` also records `_lastStopUtcTicks` (a static field) at the end** — `StartInner` reads this on entry and `Thread.Sleep`s up to `MinSilenceAfterStopMs = 11000` (11s) of host silence between the last Stop and any new prime/open-request emission. The wheel has a ~10–14 s internal timeout on sess=0x09 dashboard-binding state during which it ignores every host emission; without this gate, rapid Stop+Start cycles (SimHub game-switch plugin reload, manual disable+re-enable, dashboard switches via `RestartForSwitch`) would land inside the timeout window and the wheel never re-engages configJson. Static so it survives plugin instance recycle in the same process. Cold-start skips the gate (`_lastStopUtcTicks == 0` initially). See [`docs/protocol/findings/2026-05-08-wheel-sess09-timeout.md`](protocol/findings/2026-05-08-wheel-sess09-timeout.md). Dashboard switches go through `RestartForSwitch()` (`PreStopDrainMs` 300ms drain → Stop → Start with the silence gate enforcing the 11s wait), called from `MozaPlugin.OnDashboardSwitched` after `ApplyTelemetrySettings()` stages the new profile. As secondary defense against dropped chunks under Wine SerialPort R/W contention, `TickRetryS09IfNotEstablished` runs once per Active tick re-emitting `SendSessionPrime(0x09) + SendConfigJsonOpenRequest(0x09)` at 1s intervals (10-round budget) until `_sessions.GetOrCreate(0x09).DeviceInitiated == true`. `Stop()` also drains every outstanding `SendRpcCall` waiter (sets the `ManualResetEventSlim` so blocked callers return immediately with a null reply) instead of leaving them parked on `Wait()` until their own per-call timeout fires. `Dispose()` is idempotent (`Interlocked.Exchange` gate on `_disposed`) and disposes the two TelemetrySender-owned waiter slims (`_ackReceived`, `_mgmtResponseEvent`) plus the helper coordinators (`_uploader`, `_rpc`, `_dashboardDownloader`) which each own and dispose their own wait events; subsequent `Start`/`SendRpcCall` calls bail with a warning instead of touching disposed state. Exposes `DisplayDetected`/`DisplayModelName` for UI gating, `WheelChannelCatalog`, `WheelState` (most-recent session 0x09 dashboard state), `Session04DirListingRefreshed`, and `SendRpcCall(method, arg, timeoutMs)` for session 0x0a JSON RPC round-trips (e.g. `completelyRemove`). `CanonicalDashboardList` is populated by the plugin from the built-in profile set. Supports test patterns for diagnostics.
- `TelemetryFrameBuilder` — Assembles complete Moza telemetry frames (`7E [N] 43 17 7D 23 ... [flag] 20 [data] [checksum]`) by bit-packing channel values from a `GameDataSnapshot`. Flag byte is `TierFlagBase + tierIndex`, where the base depends on `FlagByteMode`. We send 28:00/28:01 dashboard state queries during channel config but don't yet read the responses — we could use them to auto-detect which dashboard is loaded on the wheel
- `TelemetryEncoder` — Encodes game values to compressed wire formats (25+ types: `float_001`, `percent_1`, `tyre_temp_1`, `int30`, etc.) with appropriate scaling and clamping
- `TelemetryBitWriter` — LSB-first bit packer that writes variable-width values into a byte buffer
- `GameDataSnapshot` — Flat struct populated from SimHub's `StatusDataBase` API, decoupling frame building from SimHub types. Normalizes units (e.g. throttle 0–100 → 0.0–1.0)
- `DashboardProfile` / `MultiStreamProfile` — Channel definitions grouped into tiers by update frequency. Each channel maps a Moza telemetry URL to a `SimHubField` enum, compression type, bit width, a `SimHubProperty` string (full SimHub property path resolved per-frame via `PluginManager.GetPropertyValue`), and a `SimHubPropertyScale` multiplier. Empty `SimHubProperty` = fall back to the snapshot-based `SimHubField` reader. Sentinel paths starting with `@internal/` are plugin-computed (see `MozaPlugin.ResolveInternalChannel` — currently `@internal/SteeringWheelAngle` sources from `MozaHidReader.GetCurrentAngleDegrees`)
- `DashboardProfileStore` — Parses `.mzdash` dashboard files (extracts `Telemetry.get()` URLs via regex), loads channel metadata from embedded `Data/Telemetry.json` (including optional `simhub_property` + `simhub_scale` defaults), and seeds each channel's `SimHubProperty`/`SimHubPropertyScale` at build time. `UrlFieldMap` still maps URL suffixes to `SimHubField` for the legacy fallback. `ApplyUserMappings(profile, overrides)` overlays per-channel user overrides by URL. `GetDashboardKey(path, profile)` produces a stable identity string (`builtin:<name>` or `file:<filename>:<sha1-first-8>`) used to key user mappings per dashboard in `MozaPluginSettings.TelemetryChannelMappings`. `IsInternalChannel(simHubProperty)` flags plugin-locked channels so user overrides skip them and the UI hides them from the mapping grid
- `PropertyCoercion` — `Coerce(object?, path)` converts SimHub property values (double/float/int/uint/long/byte/bool/TimeSpan/string — including Gear strings "R"/"N") to the double fed into the encoder. Null or unsupported types return 0 and log a one-time warning per path. Also exposes `ParseGear(string?)` used by both the resolver path and `GameDataSnapshot`
- `KnownSimHubProperties` — Curated list of common `DataCorePlugin.GameData.*` paths powering the editable-ComboBox autocomplete in the Channel mappings grid
- `TierDefinitionBuilder` — Builds tier definition messages in two formats. `BuildTierDefinitionMessage()` generates version 2 (compact numeric): flag bytes, channel indices, compression codes, and bit widths per tier. `BuildV0UrlSubscription()` generates version 0 (URL-based): sentinel + config + tag 0x04 channel URLs + end marker — the wheel firmware resolves compression internally. Also provides `BuildProbeBatch()` for FlagByteMode=2. CRC-32 (ISO 3309) on ALL chunks including final (verified against captures). `ChunkMessage()` and `Crc32()` shared between both versions. Version 2 compression codes confirmed from capture: 0x00=bool, 0x04=uint16_t, 0x07=float, 0x0D=int30, 0x0E=percent_1, 0x0F=float_6000_1, 0x14=uint3, 0x17=float_001
- `FileTransferBuilder` — Session 0x04 file-transfer wire format for `.mzdash` uploads. Produces two sub-messages (path registration, file content push) matching the 2025-11 firmware protocol (`docs/protocol/dashboard-upload/path-b-session-04.md`). Handles 8-byte transfer header, TLV paths (`0x8C` local UTF-16LE, `0x84` remote UTF-16LE), MD5 integrity, UTF-16BE destination path, 12-byte pre-zlib header (CRC32 LE + `08 00 00 00` + uncompressed size BE), and zlib deflate. The older session 0x01 FF-prefixed upload path has been retired — current firmware only acts on session 0x04 writes.
- `DashboardUploader` — Thin facade over `FileTransferBuilder`. `BuildUpload(mzdashContent, dashboardName, token, timestampMs)` returns both sub-messages + the chosen MD5 + a `PickToken()` helper. `WheelUploadCoordinator` (wired into TelemetrySender via `_uploader`) drives the wire-level dance (wait for device's FT-session open → chunk sub-msg 1 → wait echo → chunk sub-msg 2 → wait ack → end marker → wait device end). The coordinator also owns the FT-eligible session set, sub-msg wait events, dir-listing reassembler, and the 2025-11 ↔ 2026-04 wire-format auto-fallback.
- `ConfigJsonClient` — Session 0x09 RPC handler. `OnChunk()` feeds incoming device→host data into `SessionDataReassembler` and, once the 9-byte compressed envelope + zlib stream completes, parses the state JSON via `WheelStateParser`. Logs one Info line per decoded state push showing TitleId, displayVersion, resetVersion, configJsonList count, enabled/disabled/imageRefMap/imagePath counts, rootDirPath. Emits a drift warning (once per distinct missing-field shape) when firmware omits any of the 11 expected 2025-11 top-level fields. Exposes `LastState : WheelDashboardState` for UI binding. `BuildConfigJsonReply(canonicalList)` builds the host→device reply `{"configJson()":{"dashboardRootDir":"","dashboards":[...],"fontRootDir":"","fonts":[],"imageRootDir":"","sortTags":0},"id":11}` wrapped in the same 9-byte envelope — wheel stores the list in its next state blob's `configJsonList` field. `TelemetrySender.MaybeSendConfigJsonReply()` fires once per session on first state arrival using `CanonicalDashboardList` (populated from `DashboardProfileStore.BuiltinProfiles` + active dashboard name).
- `WheelStateParser` / `WheelDashboardState` — Schema-tolerant parser that accepts BOTH firmware shapes (2025-11 `enableManager`/`dashboards`/`configJsonList`/`displayVersion`/`rootPath`, and legacy 2026-04 `enabledManager`/`updateDashboards`/`imagePath`). `WheelDashboardState` captures all 11 2025-11 top-level fields (TitleId, DisplayVersion, ResetVersion, SortTag, RootDirPath, ConfigJsonList, EnabledDashboards, DisabledDashboards, ImageRefMap, ImagePath, FontRefMap, RootPath) + full per-dashboard metadata (Title, DirName, Hash, Id, CreateTime, LastModified, PreviewImageFilePaths, ResourceImageFilePaths, IdealDeviceInfos). `Parse(bytes, out missingFields)` overload reports top-level fields absent from the parsed state for drift diagnostics. See `docs/protocol/dashboard-upload/config-rpc-session-09.md` for the full schema table.
- `TileServerStateBuilder` — Builds session 0x03 host→wheel tile-server state blobs. `BuildEmptyStateJson()` produces the 775-byte empty-ATS/ETS2 JSON PitHouse sends on connect. `Compress()` wraps with zlib. `BuildEnvelope(compSize, uncompSize)` emits the 12-byte `FF 01 00 [comp+4 u32 LE] FF 00 [uncomp u24 BE]` wrapper observed in captures. `BuildFullBlob(json)` combines the three into a ready-to-chunk payload for `TierDefinitionBuilder.ChunkMessage(payload, session=0x03, ref seq)`.
- `TileServerStateParser` — Inbound decoder for session 0x03 if the wheel ever pushes blobs back (not observed in any of 5 captures — session 0x03 is host→wheel only in observed protocol). `OnChunk()` accumulates chunks, recognises the 12-byte envelope, decompresses zlib, parses JSON into `TileServerState { Root, Version, Games[ats/ets2]: { Populated, Name, MapVersion, TileSize, ... } }`. The accumulator is capped at 1 MiB and dropped wholesale on overflow so a malformed envelope (sentinel mismatch never trims the buffer) cannot grow unbounded. Exposed via `TelemetrySender.TileServerState`. Dormant in normal operation; kept for forward compatibility.
- `SessionDataReassembler` — Reassembles 7c:00 type=0x01 chunks. CRC-aware trailer strip (tries full-chunk CRC first, then 1-byte flag-prefix variant), `TryDecompress()` pops the 9-byte `[flag][comp_size][uncomp_size][zlib]` envelope. Falls back to `TryDecompressByMagic` (scan for `78 9c` / `78 da` zlib magic + trial decompress each hit) when the offset-based header fails — survives envelope corruption from embedded 0x7E bytes in mzdash JSON payloads, mirroring `sim/wheel_sim.py`'s approach. Seq-aware overload `AddChunk(int seq, byte[], string tag)` detects gaps (e.g. when a chunk drops on Wine SerialPort R/W contention) by tracking the last-accepted seq; on gap it clears the buffer and returns false so the caller can re-handshake (e.g. re-issue `SendConfigJsonOpenRequest` so the wheel re-emits its state burst). Without this guard a single dropped chunk silently corrupts the zlib stream and breaks the surrounding handshake for the rest of the session.
- `RpcCallChannel` — Owns session 0x0a JSON RPC machinery (host→wheel `{method(): arg, id}` calls, replies routed by integer `id`). Extracted from TelemetrySender 2026-05-08; `TelemetrySender.SendRpcCall` is now a thin pass-through to `RpcCallChannel.Call`.
- `ChannelCatalogParser` — Owns the inbound 7c:00 byte buffer and the parsed wheel channel-URL list. Replaces the prior `_incomingSessionBuffer`/`_wheelChannelCatalog`/`_lastCatalogParseLen` fields scattered across TelemetrySender. Exposes `AppendChunk` / `TryParse` / `BufferLength` / `Catalog` / `LastActivityMs` / `HasCompleteCatalog` / `Reset`. The parser handles the wheel's three URL encoding forms (full text, `0x01` prefix-compressed, `0x5C 0x31` abbreviation-compressed) and back-references plus dash-switch re-indexing.
- `WheelUploadCoordinator` — Owns the wheel-side mzdash upload session lifecycle: FT-eligible session tracking (`NoteDeviceInit`), sub-msg state machine, dir-listing reassembly (`NoteInboundChunk`), end-marker observation (`NoteEndMarker`), `RunBackgroundUpload` entry point, the 2025-11 ↔ 2026-04 wire-format auto-fallback, and skip-if-already-loaded MD5 check. Hosts `MzdashContent` / `MzdashName` / `UploadDashboard` / `UploadSessionOverride` properties (TelemetrySender exposes thin pass-throughs).
- `SessionRegistry` — Tracks session byte → role assignment for 2025-11 firmware (0x01 mgmt, 0x02 telemetry, 0x03 aux config, 0x04 file transfer, 0x06/0x08/0x0A keepalive, 0x09 configJson). Records whether each session was opened by host or device (type=0x81 source).

### Adding New Device Settings

When adding a new setting that is written to the device, it must also be saved/restored with the profile system:

1. **`MozaCommandDatabase.cs`** — Add command definition (name, device, read/write groups, command ID, payload size, type)
2. **`MozaDeviceManager.cs`** — Add device type mapping in `GetDeviceId()` if it's a new device
3. **`MozaProtocol.cs`** — Add device ID and read/write group constants if needed
4. **`MozaData.cs`** — Add volatile field(s) and `UpdateFromCommand` case(s)
5. **`MozaPlugin.cs`** — Add to `StatusPollCommands` (if it's a detection probe) and the appropriate per-device read array (e.g. `BaseSettingsReadCommands`, `NewWheelSettingsReadCommands`, `HandbrakeSettingsReadCommands`, etc.) so it's read after that device is detected. Add detection logic in `DetectDevices()` if needed, update `ApplyProfile()` (restore from profile → `_data` + device write), and update the device's `ApplySaved*Settings()` method so the value is written on detection
6. **`MozaProfile.cs`** — Add property (with -1 sentinel default), copy in `CopyProfilePropertiesFrom()`, capture in `CaptureFromCurrent()`
7. **`SettingsControl.xaml`** — Add UI controls
8. **`SettingsControl.xaml.cs`** — Add refresh logic and event handlers (handler must update `_data`, write to device, and call `SaveSettings()`). `SaveSettings()` is debounced (500ms) so rapid slider drags don't thrash the disk

Every setting that writes to the device on UI change must round-trip through profiles. If it's not in `MozaProfile`, it will be lost on game/profile switch.

### Adding a New Telemetry Channel

Most new channels only need step 1 — set `simhub_property` (and optional `simhub_scale`) in `Telemetry.json` and the resolver pulls the value directly via `PluginManager.GetPropertyValue`. Steps 2–4 are only required when the channel should ALSO be readable via the legacy `SimHubField` snapshot path (e.g. for tests that don't have a `PluginManager`, or when the property needs pre-scaled access via `GameDataSnapshot`).

1. **`Data/Telemetry.json`** — Ensure the channel's URL, compression type, and package_level are defined. Add an optional `simhub_property` (full SimHub property path, e.g. `DataCorePlugin.GameData.OilPressure`) to pre-assign a default mapping, and `simhub_scale` (number) if the SimHub value's unit doesn't match the compression's expected range (e.g. `0.01` for 0–100 → 0–1; `57.2957795` for radians → degrees). Use `simhub_property: "@internal/<key>"` to lock the channel to a plugin-computed value (add the corresponding `case` in `MozaPlugin.ResolveInternalChannel`).
2. *(optional)* **`DashboardProfileStore.cs`** — Add the URL suffix → `SimHubField` mapping to `UrlFieldMap` (legacy fallback when `SimHubProperty` is cleared)
3. *(optional)* **`DashboardProfile.cs`** — Add the new value to the `SimHubField` enum
4. *(optional)* **`GameDataSnapshot.cs`** — Add a field, populate it in `FromStatusData()`, and add a case to `GetField()`

To add a new `@internal/` sentinel channel: set the sentinel in `Telemetry.json`, then extend the `switch` in `MozaPlugin.ResolveInternalChannel(string path)` to return the computed `double`. No UI changes — internal channels are filtered out of the mapping grid automatically.

### Key Protocol Details

- Message format: `[0x7E] [length] [request_group] [device_id] [command_id...] [payload...] [checksum]`
- Response parsing: toggle bit 7 of request_group, swap nibbles of device_id, then match command_id (with 0xFF wildcards)
- Read messages use a zero-filled payload of the declared byte width (matches boxflat's `prepare_message`). Some wheels silently drop reads with a non-zero payload, even though the protocol nominally ignores payload bytes on reads
- All multi-byte integers are big-endian; floats are byte-reversed
- `MozaSerialConnection` uses two write lanes. **One-shot FIFO** (`_oneShotQueue`) carries session traffic, settings writes, and probes — ordered, paced at 4 ms between consecutive one-shots only (boxflat's proven timing; Moza bases drop rapid settings writes like the 30+ `ApplyProfile` burst). Pacing uses `Stopwatch.GetTimestamp()` (monotonic 64-bit) so the gate stays correct across long uptime; the previous `Environment.TickCount` int math wraps every ~24.8 days. **Stream slots** (`_streamSlots` — one per `StreamKind`: tier dash 0–7, enable, sequence, mode) carry periodic telemetry — latest-wins coalescing, unpaced. `SendStream(kind, bytes)` overwrites any predecessor in the same slot via `Interlocked.Exchange`, so a stale tier-0 frame can never reach the wire if a newer one was already generated. `WriteLoop()` drains one-shots first (paced only when the previous write was also a one-shot), then sweeps stream slots back-to-back. A pooled stuff buffer emits each frame in a single `_port.Write()` call via `MozaProtocol.StuffFrame`. `FlushPendingWrites()` drops both lanes + calls `_port.DiscardOutBuffer()` — used by `TelemetrySender.Pause()` so Test-Stop halts wire traffic within ~1 ms instead of bleeding through the 16 KB OS buffer (~1.4 s at 115200 baud). Pre-2026-04-22 (queue-drift fix): single `ConcurrentQueue`, unconditional 4 ms pacing between every dequeue, per-byte `_port.Write(msg, i, 1)` for each stuffed body byte. Under steady telemetry (3+ frames/tick × 4 ms pacing) drain time hit the 30 ms tick budget, so any OS jitter grew backlog until the displayed dashboard values lagged the game by hundreds of ms over a single lap. **Read path** unchanged: bulk-read thread polls `BytesToRead` at 2 ms, pulls whatever's available via `Read(buf, 0, avail)` into a 64 KB in-memory rx buffer, parses as many complete frames as possible per pass. Per-byte `ReadByte()` was replaced 2026-04-22 because its ~100 µs per-call cost under Wine starved the reader during bursts and lost chunks. Wine's blocking `Read(buf, 0, full_count)` does NOT return early when fewer bytes are available — it hangs — so the BytesToRead guard is load-bearing
- 0x7E byte stuffing: every `0x7E` byte in a frame's body (group, device, payload, checksum) is doubled on the wire. `MozaProtocol.StuffFrame(frame, dest)` writes the start+len header raw then walks body bytes emitting each and appending an extra `0x7E` on match; `StuffedFrameSize(frame)` pre-computes the output size so `WriteLoop`'s pooled stuff buffer can be grown on demand. The single-shot stuffed write replaced per-body-byte `_port.Write(msg, i, 1)` calls in 2026-04-22 — the old path issued 10–32 syscalls per frame, compounding Wine's per-call overhead. `ReadLoop()` reverses stuffing: after reading the start+len header, it reads N+3 decoded bytes, collapsing any `0x7E 0x7E` pair back to a single `0x7E`; an unpaired `0x7E` marks the start of a new frame and the current frame is abandoned. Without this, payload 0x7Es (common in zlib-compressed session data and seq counters) desync the read stream. **Checksum must also account for stuffing** — the wire-level sum includes the duplicated `0x7E` bytes, so `CalculateWireChecksum()` adds `count(0x7E in body) × 0x7E` to the raw sum. Pre-2026-04-22 the plugin's `CalculateChecksum` omitted this term and dropped ~20% of zlib-bearing session chunks (configJson state, dashboard uploads) whose compressed payloads contain 0x7E bytes. See `docs/protocol/wire/checksum.md`
- Session close frame: the type=0x00 end marker carries a 6-byte payload `7C 00 [session] 00 [ack_lo] [ack_hi]` with `len=6`. A 4-byte payload advertised as len=6 causes the wheel/sim to over-read into the next frame and kill the read loop. `SendSessionClose()` zero-fills the ack_seq bytes when simply reclaiming a stale session

### Dependencies

- **NuGet:** `Microsoft.NETFramework.ReferenceAssemblies.net48`, `Newtonsoft.Json`, `log4net`
- **Runtime (Windows only):** `System.Management` — loaded via reflection for WMI port discovery; falls back to probe-based discovery if unavailable. Failures (assembly load, query throw) are caught and logged at Debug as `WMI enumeration failed: <Type>: <Message>`
- **SimHub DLLs** (checked into `libs/SimHub/`, reference-only, not packaged): `SimHub.Plugins.dll`, `GameReaderCommon.dll`, `SimHub.Logging.dll`, `SerialDash.dll`, `BA63Driver.dll`, `HidSharp.dll`. A daily GitHub Actions workflow automatically creates PRs when new SimHub versions are released.

**Important:** The SimHub DLLs in `libs/SimHub/` must match the runtime SimHub version. The PluginSdk ships older DLLs that may be missing interface members added in newer releases, causing `TypeLoadException` at runtime. Always update from the actual SimHub installation directory.
