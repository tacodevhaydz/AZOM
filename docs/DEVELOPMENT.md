# Project Overview

SimHub plugin for MOZA Racing hardware providing two-way telemetry: streams game data (speed, RPM, gear, lap times, fuel, tyre wear, etc.) to the wheel dashboard display, drives wheel/dashboard RPM and flag LEDs, and allows configuring wheelbase settings. Uses a custom binary serial protocol reverse-engineered from the [boxflat](https://github.com/Lawstorant/boxflat) project.

## Building from Source

The project targets .NET Framework 4.8 (x86) and uses the `Microsoft.NETFramework.ReferenceAssemblies.net48` NuGet package so it can cross-compile on Linux without Mono. The built DLL runs on Windows under SimHub.

### Building on Windows

#### Prerequisites

- [VS Code](https://code.visualstudio.com/) with the [C# Dev Kit](https://marketplace.visualstudio.com/items?itemName=ms-dotnettools.csdevkit) extension
- .NET SDK 8.0+ ([download](https://dotnet.microsoft.com/download))

#### Steps

1. **Open the project folder** in VS Code.

2. **Build** from the VS Code terminal:

   ```
   dotnet build -c Release
   ```

3. **Copy the DLL** to your SimHub folder:

   Copy `bin/x86/Release/MozaPlugin.dll` into your SimHub installation directory.

   Or set the `SIMHUB_PATH` environment variable to have it copied automatically on build:

   ```
   set SIMHUB_PATH=C:\Program Files (x86)\SimHub
   dotnet build -c Release
   ```

   PowerShell:
   ```powershell
   $env:SIMHUB_PATH = "C:\Program Files (x86)\SimHub"
   dotnet build -c Release
   ```

4. **Restart SimHub.** The plugin appears under Settings > Plugins as "MOZA Control".

### Cross-Compiling on Linux

You can build the plugin entirely on Linux. The .NET SDK can target .NET Framework 4.8 using the `Microsoft.NETFramework.ReferenceAssemblies.net48` NuGet package (already included in the `.csproj`).

#### Prerequisites

- .NET SDK 8.0+ (`dotnet-sdk` package from your distro or [Microsoft repos](https://dotnet.microsoft.com/download))

#### Steps

1. **Install the .NET SDK** if you don't have it:

   ```bash
   # Arch Linux
   sudo pacman -S dotnet-sdk

   # Ubuntu/Debian
   sudo apt install dotnet-sdk-8.0

   # Fedora
   sudo dnf install dotnet-sdk-8.0
   ```

2. **Build:**

   ```bash
   dotnet build -c Release
   ```

   The output DLL will be in `bin/x86/Release/MozaPlugin.dll`.

3. **Copy the built DLL to your Windows SimHub installation:**

   ```bash
   # Example: copy to a Windows machine via scp
   scp bin/x86/Release/MozaPlugin.dll user@windows-pc:"C:/Program Files (x86)/SimHub/"

   # Or copy to a shared folder, USB drive, etc.
   cp bin/x86/Release/MozaPlugin.dll /mnt/shared/SimHub/
   ```

4. **Restart SimHub** on Windows.

#### Notes

- The `Microsoft.NETFramework.ReferenceAssemblies.net48` NuGet package provides the .NET Framework 4.8 reference assemblies, so you do **not** need Mono or Windows installed.
- The build produces a standard .NET Framework DLL that runs natively on Windows under SimHub.
- SimHub DLLs in `libs/SimHub/` are reference-only (`Private=false`) and not copied to output.
- The build produces a single output DLL with no additional runtime dependencies to deploy.

### CI/CD

- **Build**: Every push to `main` and every PR is built automatically via GitHub Actions.
- **Release**: Pushing a `v*` tag (e.g., `v0.2.0`) builds a Release, generates a changelog, and publishes a GitHub Release with the DLL (device definitions are embedded in the DLL).
- **SimHub dependency updates**: A daily workflow checks for new SimHub releases and creates a PR to update `libs/SimHub/`.

## Architecture

### Code Organization

The plugin's four previously-monolithic codebehinds and senders were split into focused collaborator classes in a 5-PR refactor (2026-05-18). `MozaPlugin.cs` and `Telemetry/TelemetrySender.cs` remain the orchestrators; behavior they used to host inline now lives in dedicated files:

- **`Devices/DeviceDetectionState.cs`** â€” Volatile per-device detection flags (wheel/dash/handbrake/pedals/hub/AB9, plus extended LED group mask) read by poll/UI/serial/telemetry threads.
- **`Devices/DeviceProber.cs`** â€” `DetectDevices` response dispatch + per-device read command arrays (`BaseSettingsReadCommands`, `NewWheelSettingsReadCommands`, â€¦).
- **`Devices/Ab9EngineVibrationWorker.cs`** â€” The 91 Hz host-rendered AB9 engine-vibration loop and its `K = 3.95e11` period formula.
- **`Settings/SettingsMigrator.cs`** â€” Schema v0â†’v8 settings migration + `SeedProfileBaselineFromFlatFields` baseline seeding.
- **`Hardware/HardwareApplier.cs`** â€” Every hardware-side write path: `ApplyProfile`, the seven `Apply*ToHardware` methods, and the detection-gated `WriteIf*` / `WriteColorIf*` / `WriteArrayIf*` family.
- **`Telemetry/DashboardBindingCoordinator.cs`** â€” Profile-driven dashboard switch (kind=4 emit), wheel-initiated switch handling, telemetry-pipeline lifecycle gates, `ApplyTelemetryDashboardFromProfile`.
- **`Telemetry/SimHubPropertyResolver.cs`** â€” `ResolvePropertyAsDouble`/`AsString`, `@internal/` channel resolution, `GetAllSimHubPropertyNames`.
- **`Telemetry/Inbound/TelemetryInboundDispatcher.cs`** â€” Inbound 0xC3/dev 0x71 routing: fc:00 acks, type=0x81 device-init, per-session type=0x01 dispatch.
- **`Telemetry/Watchdog/SessionWatchdogManager.cs`** â€” Four wheel-session watchdogs: sess=0x09 retry, sess=0x02 engagement, configJson gap escalation, configJson stuck-state.
- **`Telemetry/Display/WheelSlotTracker.cs`** â€” `MaybeUpdateWheelReportedSlot` with strict type-04 validation (rejects false-positive slots from collisions with tier-def 0x04 records).
- **`Telemetry/PropertyPushQueue.cs`** â€” Brightness-blanking coalescing via per-(session,kind) seq supersedence.
- **`Telemetry/Frames/TierDefinitionEmitter.cs`** â€” `SendTierDefinition` + blind retransmit for Era2024/25 firmware that doesn't ack sess=0x01.
- **`Telemetry/Lifecycle/SilenceGate.cs`** â€” Two host-side silence timers (Stopâ†’Start ~11 s wait + post-switch UI cooldown). Statics so they survive plugin recycle within a SimHub process; constructor takes a `Func<bool> isHotRenegotiationEnabled` so the cooldown gate can shorten itself when hot mode is on.
- **`Telemetry/Lifecycle/HotSwitchCoordinator.cs`** â€” Hot-renegotiation burst state machine. `ArmBurst` / `ShouldEmitThisTick(catalogLastActivityTickMs, catalogWheelEndMarkerTickMs)` / `MarkEmission` collapse the prior ~100-line first-emission/pacing tree inside `TickGrowSubscriptionIfCatalogStable` into a short decision function with explicit branches.
- **`Telemetry/Lifecycle/CatalogResyncProbe.cs`** â€” Throttle + has-fired memo for the catalog re-sync probe. Decision logic (slot lookup, wheel-on-target shortcut, deferred ThreadPool emit) stays in `TelemetrySender.ScheduleCatalogResyncProbe`; this class only owns the timestamp.
- **UI controllers** (`UI/UiHelpers.cs`, `UI/DiagnosticsTextBuilder.cs`, `UI/DiagnosticsBundleWriter.cs`, `UI/DashboardLibraryResolver.cs`, `Devices/WheelUi/{ChannelMappingRow,WheelUiHelpers,ChannelMappingRowFactory}.cs`) â€” Pure-function text builders, ZIP bundle writer, library-path resolution, and per-wheel UI helpers shared across `SettingsControl.xaml.cs` and `MozaWheelSettingsControl.xaml.cs`.

Cross-collaborator state access uses internal back-references (`_sender._field` via `internal` visibility) rather than partial-class splits â€” accept the field-plumbing cost in exchange for explicit boundaries. The orchestrators construct collaborators in their constructors and delegate via 1-line forwarders.

### Component Layers

**Logging & Diagnostics** (`Diagnostics/`) â€” Shared infrastructure for plugin logging and the Diagnostics-tab export bundle:
- `MozaLog` â€” Static wrapper around `SimHub.Logging.Current` (`Info`/`Debug`/`Warn`/`Error`). Every plugin log call goes through this class instead of `SimHub.Logging.Current` directly so that, in addition to forwarding to SimHub's log4net pipeline, every `[Moza]` line is appended to a 5 000-entry in-process ring buffer with a millisecond local-time stamp and level. `Snapshot()` returns the buffer as a single string for the Diagnostics export â€” sidesteps SimHub's flush cadence and per-version log-path / file-extension differences (`Logs/SimHub.<n>.txt` on current builds). Lives in `namespace MozaPlugin` so all source files reach it without a using statement.
- `SerialTrafficCapture` â€” Singleton ring buffer of timestamped serial frames in both directions, used by the Diagnostics tab's capture toggle. `RecordTx(label, frame)` and `RecordRx(label, frame)` are called from `MozaSerialConnection.WriteFrame` (after a successful port write) and `ReadLoop` (just before `MessageReceived?.Invoke`) respectively. Each `MozaSerialConnection` carries a `CaptureLabel` property (default "wheelbase"; AB9 manager flips its connection's label to "ab9") so a single export can distinguish frames from each pipe. Capacity is 200 000 entries â€” about 30â€“60 minutes of dense telemetry â€” with oldest-drop ring trimming so a long capture never exhausts memory. `Start()` clears the buffer and arms the per-frame fast path; `Stop()` returns the snapshot and disables the gate; `Format(entries)` renders to one-line-per-frame text with local timestamps and uppercase hex bytes.

**Plugin Entry Point** (`MozaPlugin.cs`) â€” Implements SimHub's `IPlugin`, `IDataPlugin`, `IWPFSettingsV2`. Manages lifecycle (Init/DataUpdate/End), connection state, auto-reconnect (5s timer), setting polling (2s timer), `TelemetrySender` for dashboard data streaming, and `DashboardProfileStore` for profile management. `CheckGearshiftEvent` runs every `DataUpdate` and fires `base-gearshift-event` (cmd 0x76 on grp 0x2D, fire-and-forget â€” wheel firmware uses the persisted `base-gearshift-vibration` intensity, payload is always `76 00 01`) when SimHub's reported `Gear` value changes, gated by intensity > 0 plus a user-tunable debounce window so paddle ratcheting coalesces. Transitions *into* neutral ("N" or "0") are suppressed by default â€” an H-pattern shift produces two transitions ("1"â†’"N"â†’"2") and the engagement (Nâ†’2) is the one the user feels; the suppression can be flipped via the "Vibrate on Neutral" plugin setting (`MozaPluginSettings.GearshiftVibrateOnNeutral`/`GearshiftDebounceMs`). **AB9 host-rendered engine vibration (multi-stream worker, 2026-05-15):** `DataUpdate` stashes `data.NewData.Rpms` into `_latestRpmBits` (`Interlocked.Exchange` of a long via `BitConverter.DoubleToInt64Bits` â€” `double` can't be `volatile` in C#) and `data.GameRunning` into `volatile bool _latestGameRunning`. A dedicated `MozaAb9EngineVib` background thread (`Ab9EngineVibrationLoop`) ticks every ~11 ms (target 91 Hz, Stopwatch-deadline scheduled â€” `System.Timers.Timer` would top out at ~64 Hz on Windows's default 15.6 ms clock granularity). Each tick drives the full PitHouse-replicating sub-stream set decoded from the 2026-05-13 capture (`sim/logs/ab9-game-20260513.jsonl`, 942,634 frames; analysis pipeline: `tools/ab9-decode-session`):

  1. **`0x0A 0x05` engine-vibration refresh â€” every tick.** Computes `period_ticks = K / (rpm Ã— freq_hz)` where K â‰ˆ 3.95 Ã— 10Â¹Â¹ and `freq_hz` is the literal user slider value 0..300, clamps to `[0x64, 0xFFFFFF]`, and submits via `MozaAb9DeviceManager.SendEngineVibrationStream`. Slot ID is `0x1996` when `active = gameRunning && intensity > 0 && rpm > 100 && freqHz > 0`, else `0x0000` (silent keepalive).
  2. **`0x0D 0x02 + 0x0D 0x03` keepalive pair â€” every 12 ticks (~9 Hz flat).** Paired triggers PitHouse fires at constant rate regardless of RPM (`SendKeepalivePair`).
  3. **`0x0B 0x02 + 0x0B 0x03` engine-pulse pair â€” RPM-scaled.** Pulse-on / pulse-off pair sharing a 16-bit BE phase counter (`_ab9PulsePhase`) duplicated within each frame at offset 6-7 = 8-9 (the device cross-checks the two copies). Pulse-on amplitude is the intensity slider linearly scaled to PitHouse's observed full-scale `0x2328`; pulse-off carries zero amplitude. Fire interval = `max(2, 62 / (rpm/800))` ticks (~1.5 Hz idle, ~13 Hz redline).
  4. **`0x0D 0x05` RPM-tracking trigger â€” RPM-scaled.** Fire interval = `max(2, 80 / (rpm/800))` ticks.
  5. **`0x08 0x04 + 0x08 0x06` low-rate signed pair â€” every ~260 ticks (~0.35 Hz).** Signed 16-bit magnitude carried as `-mag` in 0x04 + `+mag` in 0x06; engine-cycle phase signal. `_ab9LowRatePhase` accumulator advances 100/emission, saturates at Â±32000 then wraps.
  6. **`0x0D 0x01` sparse trigger â€” every ~920 ticks (~0.10 Hz).** Newly-observed sub-cmd (3-byte frame, value `0x01`); mirrored verbatim for byte-fidelity with PitHouse.

  Sub-streams 2-6 are skipped when `active=false`; only the 0x0A 0x05 silent keepalive ticks. Rate constants live as `*BaseTicks` private consts on `MozaPlugin`. New `MozaSerialConnection.StreamKind` lanes (`Ab9EnginePulse`, `Ab9TriggerA`, `Ab9TriggerRpm`, `Ab9TriggerExtra`, `Ab9LowRate`) coalesce stale frames if a tick falls behind.

  **FFB init handshake** fires on the rising edge of `_ab9Detected` in `OnAb9MessageReceived`: `MozaAb9DeviceManager.SendFfbInitSequence` emits the exact PitHouse session-start sequence (`0e02, 0e01, 0703, 0709Ã—2, 0701, 0704, 0701, 130000`) before any streaming starts. Idempotent per connection; reset on disconnect. Without this handshake the device's FFB stack stays uninitialised and the streaming frames have no audible/tactile effect â€” this was the root cause of the pre-2026-05-15 "no engine rumble" symptom.

  The loop is started in `Init()` after `_ab9Manager` construction, stopped in `End()` / `CleanupPartialInit()` with `_ab9EngineVibStop = true` + `Join(1000)` before the AB9 manager is disposed. Tick gating on `_ab9Detected && _ab9Manager.IsConnected` makes it a cheap no-op when the AB9 isn't present. `ApplyAb9ToHardware` (called from `OnAb9MessageReceived` on first detection and from `ApplyProfile` on game/profile switch) pushes the persisted `GearShiftVibrationIntensity` via `SendGearShiftVibrationIntensity` (one-shot `0x0A 01` config write â€” AB9 firmware fires the actual rumble pattern autonomously on each HID-detected gear engagement, no per-shift host trigger). Uses phased per-device initialization: `TryConnect()` sends only lightweight detection probes, then `DetectDevices()` applies saved settings and reads back device state as each device (base, wheel, dashboard, handbrake, pedals, hub) is confirmed present. This is the orchestrator that wires everything together. **Reload safety:** `Init()` is wrapped in try/catch; on partial-init failure `CleanupPartialInit()` mirrors `End()` so SimHub's reload-without-restart path can't leak timers, serial ports, or HID readers. `Instance = this` is published only after every resource is wired so background callbacks can't observe a half-built plugin. `End()` symmetrically detaches `_connection.MessageReceived -= OnMessageReceived` and `_subscribedProfileStore.CurrentProfileChanged -= OnProfileChanged` (the subscribed store is tracked separately so `ClearSettings()` â€” which builds a fresh `MozaPluginSettings` â€” can detach the orphaned subscription before re-subscribing on the new store). `OnMessageReceived` and `PollStatus` short-circuit on `IsShuttingDown` so in-flight callbacks delivered after `End()` started can't touch disposed state. Property delegates registered with `AttachDelegate` are null-guarded (`_data?.X ?? default`) so SimHub's property polling is immune to partial-init / mid-teardown windows.

**Serial Protocol Layer** (`Protocol/`) â€” Binary protocol over USB serial at 115200 baud (VID `0x346E`):
- `MozaSerialConnection` â€” Device auto-discovery via `MozaPortDiscovery` (Windows registry walk of `HKLM\SYSTEM\CurrentControlSet\Enum\USB\VID_346E&PID_*&MI_00\<instance>\Device Parameters\PortName`, the standard `usbser.sys` layout), background read/write threads, frame assembly, full 0x7E byte stuffing (read + write â€” every 0x7E byte after the start+len header is doubled on the wire; without this, payload 0x7Es in zlib blobs / seq counters desync the read stream after ~15s and trigger the wheel-alive watchdog). The registry path is the always-on primary discovery: it returns full `(PortName, VID, PID, FriendlyName, Category)` for every MOZA composite (the `Category` member of `PortInfo` is derived from `MozaUsbIds.Categorize(pid)`), cross-references against `SerialPort.GetPortNames()` to drop ghost entries left behind by previous USB-port attachments, and routes each port to the right connection class without writing any serial bytes. **PID routing (2026-05-16)** is driven by the inventory in `Protocol/MozaUsbIds.cs` â€” the single source of truth, mirrored in [`docs/protocol/devices/usb-ids.md`](protocol/devices/usb-ids.md). The wheelbase connection accepts `MozaDeviceCategory.Wheelbase` PIDs (R16/R21 `0x0000`, R9 `0x0002`, R5 `0x0004`, R3 `0x0005`, R12/R12v2 `0x0006`) plus any PID not present in the inventory (`!IsKnownMozaPid`) as a future-hardware fallback. The AB9 connection accepts only `MozaDeviceCategory.Ab9` (`0x1000`) plus the same unknown set. Known non-target categories â€” `Pedals` (`0x0001`/`0x0003`), `Shifter` (`0x001E`), `Handbrake` (`0x001F`), `Hub` (`0x0020`) â€” are silently dropped from both connections so the wheelbase pipe stops emitting base/hub probe frames at devices that ignore them. Before this change the wheelbase filter was `pid => !IsAb9Pid(pid)`, which let every non-AB9 Moza composite through and produced cross-device probe noise whenever pedals/handbrake/shifter/hub were attached to the same bus. `MozaPortDiscovery` logs a single Info-level line per process the first time it sees an unknown PID (`[Moza] Unknown Moza PID 0xXXXX on COMn â€” not in usb-ids inventoryâ€¦`) so unmapped hardware surfaces in support bundles; the `SummarizePorts` line accompanying every cache refresh now includes the friendly name (`COM5:0x0006(R12 / R12v2)`). The legacy serial-probe path is preserved as an automatic fallback that runs whenever the registry didn't already hand out a port matching the connection's PID filter. **Registry-aware probe loop (2026-05-16):** `FindMozaPort` builds a `port â†’ PortInfo` lookup from the registry walk and consults it inside both the `BaseAndHub` two-pass loop and the `Ab9` base-disambig + identity loop. Three outcomes per port: (a) unclassified â†’ probe normally; (b) classified with a matching PID â†’ claim the port immediately with no probe writes (uses the registry-reported PID); (c) classified with a mismatching MOZA PID â†’ skip with a debug log and zero serial bytes. Aggregate gate: when every COM port is registry-classified with a non-matching PID (e.g. Windows wheelbase-only user, no AB9 on the bus), the probe is short-circuited entirely so no frames go on the wire â€” this preserves the pre-2026-05-16 "no AB9 probe storm" behaviour. When at least one unclassified COM port exists alongside classified ones, the probe runs only against the unclassified remainder, which unblocks discovery of MOZA hardware on bus configurations where some devices register and others don't. The most concrete leak this closed: the AB9 manager's base-disambig probe was previously sent at every COM port whenever the registry was empty, and could in principle reach a port whose PID the registry already classified as a wheelbase â€” `RegistrySaysSkip` now suppresses that probe before it ever opens the port. The fallback can be hard-disabled via `MozaPluginSettings.DisableSerialProbeFallback` (default off) for users who want to guarantee no Moza writes ever reach a non-Moza serial device. Probe target is scoped via `MozaProbeTarget` (`BaseAndHub` for the wheelbase connection, `Ab9` for the AB9 shifter). `BaseAndHub` runs a two-pass probe: bases first (group 0x2B â†’ expect response group 0xAB), then hubs (group 0x64 dev 0x12 â†’ expect 0xE4) so hub-only setups are discovered. `Ab9` runs the AB9 identity probe (group 0x09 dev 0x12 â†’ expect 0x89) with a base-disambiguation pre-probe â€” needed because `DeviceAb9` and `DeviceHub` share id 0x12. Pre-2026-05-11 the WMI path was loaded via `Assembly.Load("System.Management")` reflection and failed silently in SimHub's plugin AppDomain, dropping every install to the probe path; "all AB9 users had to disable AB9 detection" because the probe writes disturbed non-Moza serial peripherals (Arduino, OBD2, GPS, 3D printers). The registry-based discovery eliminated both problems â€” see `Protocol/MozaPortDiscovery.cs`. `HubProbeSucceeded` (used by `TelemetrySender` to gate the hub-only 5-slot enumeration burst) is set from `MozaPlugin.OnMessageReceived` on the first 0xE4 (hub) reply rather than from the probe path, since registry discovery doesn't probe the hub
- `MozaCommand` â€” Message builder with big-endian int encoding
- `MozaCommandDatabase` â€” 200+ pre-built command definitions, including PitHouse-style extended wheel identity probes (`wheel-presence` group 0x09, `wheel-device-presence` 0x02, `wheel-device-type` 0x04, `wheel-capabilities` 0x05, `wheel-mcu-uid` 0x06 (12-byte STM32 UID), `wheel-hw-sub` 0x08 cmd=02, `wheel-identity-11` 0x11 cmd=04) fired by `MozaDeviceManager.SendPithouseIdentityProbe()` at wheel detect for full 12-frame PitHouse parity, and the wheel-base ambient LED command block (`base-ambient-*` on group 0x20 write / 0x22 read against dev 0x12 / device type `"main"`) covering live telemetry (per-strip color chunks `0x1A` and bitmask `0x1B`), persistent settings (indicator state, standby mode + per-mode interval, brightness, per-LED static colors, sleep timing, startup/shutdown color), and the 36-entry `base-ambient-led-color-strip{0,1}-mode{1,2}-led{0..8}` matrix populated via nested `for` loops
- `MozaResponseParser` â€” Decodes responses using bit-7 toggle, nibble swap, and wildcard matching. `Parse(data, busHint = null)` â€” when `busHint` is supplied (e.g. `"ab9"` from `OnAb9MessageReceived`), the auto-derived device hint is overridden so the parser only considers commands matching that bus. Needed because the AB9 main and wheelbase main both use dev id `0x12`; without the explicit hint, AB9 responses get auto-tagged as `"base"` and every `ab9-*` command is filtered out â€” which broke detection pre-2026-05-15. Unwraps Display sub-device identity responses (0xC3 group + inner 0x8X response byte) into `display-*` command names (`display-model-name`, `display-hw-version`, `display-sw-version`, `display-serial`, `display-mcu-uid`, `display-device-type`, `display-capabilities`, `display-presence`, `display-device-presence`, `display-identity-11`). Silent-drops SerialStream control frames (0xC3 + `7C/FC 00` session-open/data wrappers + 0x7C 0x23 dashboard-activate advertisements) so they don't spam unmatched-log
- `MozaProtocol` â€” Constants (start byte `0x7E`, device IDs, checksum formula: `(13 + sum) % 256`). Two checksum helpers: `CalculateChecksum()` (raw sum, for legacy/test use) and **`CalculateWireChecksum()`** (raw sum + `count(0x7E in body positions 2..len-1) Ã— 0x7E`). Production code uses `CalculateWireChecksum` on both send and verify paths so frames with 0x7E in body validate correctly â€” before this fix (pre-2026-04-22), ~20% of zlib-bearing session chunks (configJson state, dashboard uploads) failed verify and were silently dropped. The static probe builder (`MozaSerialConnection.BuildProbe`) also uses the wire variant; current probe payloads contain no 0x7E so the values coincide, but the wire variant is forward-correct if a future probe template needs a 0x7E byte. Also exposes `WheelEchoPrefixes` + `IsWheelEcho(data)`: the set of `(group, device, payload-prefix)` tuples the wheel echoes back on LED/brightness/page-config writes. `MozaPlugin.OnMessageReceived` short-circuits recognized echoes â€” marks wheel alive + skips unmatched-log noise. Mirrors `sim/wheel_sim.py:_WHEEL_ECHO_PREFIXES`
- `MozaHidReader` â€” Reads physical input positions (steering, throttle, brake, clutch, paddles, handbrake) and button states from Moza HID devices using HidSharp. Enumerates devices by name regex (matching boxflat/foxblat patterns), opens per-device receiver threads, and writes normalized values to `MozaData`. Tracks axis usages (GenericDesktop page) and button usages (Button page 0x0009). Handbrake button events are routed to a dedicated `HandbrakeButtonPressed` field â€” detected either by device name (standalone HBP USB device) or by button index 121 (base-attached handbrake). Also matches the Universal Hub HID device (`gudsen universal hub`). Independent of the serial protocol â€” provides live axis and button data for UI display at 30 Hz

**Device Management** (`MozaDeviceManager.cs`) â€” High-level read/write API. Handles wheel device ID cycling (IDs 23â†’21â†’19) since ES wheels respond on different IDs. `ResetWheelDetection()` must be called on disconnect so detection probes are re-sent on reconnect. `ReadSettingsPaced()` runs a batch of reads on a background task with an extra ~10ms gap between enqueues â€” used for large startup bursts (30+ wheel setting reads) that the wheel would otherwise drop, without throttling steady-state telemetry.

**AB9 Device Management** (`Devices/MozaAb9DeviceManager.cs`) â€” Owns the dedicated `MozaSerialConnection` for the AB9 active shifter (VID `0x346E`, PID `0x1000`, dev id `0x12`, probe target `MozaProbeTarget.Ab9`).

  **Identity probe** (`SendIdentityProbe`) emits PitHouse's full 10-group cascade (`0x09, 0x04, 0x06, 0x02, 0x05, 0x07, 0x0F, 0x11, 0x08, 0x10`) â€” extended from the original 5-group set 2026-05-15 to match what PitHouse actually sends so a real AB9 responds with the same probe coverage.

  **Stored-setting read group is `0x1E`, not `0x1F`** â€” distinct from the write group. Wire shape: `7E 01 1E 12 <cmdId> <chk>` with a single-byte cmd-id payload (writes use `7E 03 1F 12 <cmdId> 00 <val> <chk>` with the cmd+0x00+val triplet). The plugin built `SendAb9Read(byte cmdId)` directly rather than going through `MozaCommand.BuildReadMessage` because the read shape doesn't fit the single-cmd-id-array convention. Read responses arrive on group `0x9E`; `MozaCommandDatabase` registers parallel `ab9-*-read` entries on group `0x1E` with single-byte CommandId and 2-byte BE payload so the parser decodes the response correctly. The pre-2026-05-15 read path sent reads on group 0x1F with the wrong payload shape, so the real AB9 never responded and `_ab9Detected` stayed false (the "Probing AB9â€¦" panel symptom).

  **Mechanical-mode set** (`SendMode` â€” cmd `0xD3 00` on grp 0x1F) and **feel slider writes** (`SendSlider` â€” mech-resistance/spring/damping/friction/max-torque, all single-byte 0..100 on grp 0x1F) route through `MozaCommandDatabase` and `MozaCommand.BuildWriteMessage`. The write entries have `ReadGroup = 0xFF` so `BuildReadMessage` returns null and callers are forced through `SendAb9Read`.

  **FFB session-init handshake** (`SendFfbInitSequence`) â€” fire-and-forget byte-for-byte replica of PitHouse's session-start cascade: `0e02, 0e01, 0703, 0709Ã—2, 0701, 0704, 0701, 130000`. Fires once on the rising edge of `_ab9Detected` (gated by `_ffbInitSent`); resets on disconnect. The 1-byte slot indices returned in `0x07` ACKs are device-side internal effect handles and are intentionally ignored â€” the streaming frames use 16-bit slot IDs that are runtime Windows DirectInput effect handles (the device firmware routes through its own allocation table, not the streaming-frame slot IDs).

  **Engine-vibration streaming** â€” `SendEngineVibrationStream(bool active, uint periodTicks)` builds the 24-byte `0x20/0x0A 05` frame inline (16-bit BE slot ID + 7 zero bytes + 24-bit BE period + `0x04` type tag + 4 zero bytes) and submits via `_connection.SendStream(StreamKind.Ab9EngineVibration, frame)`. Latest-wins coalescing on this lane means a slow 91 Hz tick discards the previous frame instead of piling up.

  **Engine-pulse pair** â€” `SendEnginePulsePair(ushort phase, int intensity0to100)` builds the paired 22-byte `0x0B 0x02` (on) + `0x0B 0x03` (off) frames. Both carry a 16-bit BE phase counter duplicated at payload offsets 6-7 = 8-9 (device cross-check), fixed `0xFFFA` envelope param, `0x04` type tag, and 16-bit BE amplitude (intensity-derived for ON: linearly scaled from 0..100 to `0x0000..0x2328`; zero for OFF). On frame uses `StreamKind.Ab9EnginePulse` (latest-wins); OFF frame uses one-shot FIFO so it always follows the on frame onto the wire.

  **Triggers** â€” `SendTrigger(Ab9Trigger trigger)` builds 3-byte `0x0D <subLo> 0x01` frames. Each lane routed separately: `Ab9TriggerA` for keepalive A/B (`0x02`/`0x03`), `Ab9TriggerRpm` for RPM-tracker (`0x05`), `Ab9TriggerExtra` for sparse (`0x01`). `SendKeepalivePair` fires `0x0D 0x02` via the stream lane + `0x0D 0x03` via the one-shot FIFO so the pair always lands together.

  **Low-rate signed pair** â€” `SendLowRatePair(short magnitude)` builds the paired 11-byte `0x08 0x04` (`-magnitude`) + `0x08 0x06` (`+magnitude`) frames, encoding a bipolar engine-cycle phase signal. Fixed `00 64 04` envelope/tag bytes verbatim from PitHouse. Stream-lane + one-shot FIFO pairing same as the keepalive pair.

  **Gear-shift vibration intensity** â€” `SendGearShiftVibrationIntensity(int 0..100)` builds the 24-byte `0x20/0x0A 01` frame (16-bit BE intensity at payload offset 0-1, scaled linearly from 0..100 to 0..`0x332C` â€” verified empirically: 30% = `0x0F5A`, 100% = `0x332C`) and submits one-shot via `_connection.Send`. AB9 firmware fires the actual rumble pattern autonomously on each HID-detected gear engagement, no per-shift host trigger.

  See `docs/protocol/devices/ab9-shifter.md` for the live-capture decode of every frame shape and the full Phase 1 resolution of previously-open schema questions. Frame layout constants live on the class (`SilentSlotId = 0x0000`, `DefaultEngineVibSlotId = 0x1996`, `MinPeriodTicks = 0x64`, `MaxPeriodTicks = 0xFFFFFF`) so `MozaPlugin.ProcessEngineVibTick` can clamp before calling.

**mBooster Pedals Multi-Device Registry** (`Devices/MozaMBoosterRegistry.cs`, `Devices/MBoosterDeviceController.cs`, `Devices/MBoosterEffectWorker.cs`, `Devices/MBoosterEffectSynthesizer.cs`, `Devices/MBoosterTypes.cs`, `Protocol/MozaMBoosterProtocol.cs`) â€” Multi-device owner for Moza mBooster Pedals (VID `0x346E`, PID `0x0008`). The mBooster is a single-axis analog pedal with a built-in vibration motor â€” the user picks whether the pedal serves as throttle, brake, or clutch in the plugin UI, and the motor is driven by host-rendered effects (ABS, Lockup, Threshold, Engine) per the user-supplied protocol note in [`docs/MozamBooster â€” Protocol Note.md`](MozamBooster%20â€”%20Protocol%20Note.md). The HID-side identity is extracted from `HidDevice.DevicePath`'s parent-USB instance segment via `MozaHidReader.ExtractUsbParentInstance` â€” both stacks expose the same USB parent so the registry can pair them, though this is currently unverified on real hardware (the plan calls out a port-order fallback if identities diverge on a particular unit).

  **Effect synthesis** follows the protocol note Â§ 4 verbatim â€” `MBoosterEffectSynthesizer.cs` is pure functions: `SynthesizeAbs(intensity, phase) = (0.9 + 0.1 * sin(phase)) * intensity` (oscillating pulse 80â€“100%), `SynthesizeLockup(intensity, elapsed) = clamp(elapsed/0.5, 0, 1) * intensity` (linear ramp), `SynthesizeThreshold(intensity, elapsed)` = 5 Hz burst envelope (20 ms full + 120 ms 80% + 60 ms gap), `SynthesizeEngine(intensity, phase) = (0.5 + 0.5 * sin(phase)) * intensity` (biased sine). **Engine intensity is clamped to 10 %** at the apply layer (`EngineIntensityCap = 0.10` in `MBoosterEffectWorker`) â€” the protocol note warns engine runs continuously and would dominate the other effects without the cap.

  **Wire format** â€” `Protocol/MozaMBoosterProtocol.cs` owns the motor-write opcode (`0xb1`) and keepalive shape. Motor frames are 14 bytes pre-stuffing: `7e 09 24 12 b1 [effect:u8] [enable:u8] 00 [param1:u8] [freq:u16 BE] [amp:u16 BE] [chk]` where `param1 = clamp(round(paramK / freq_hz), 1, 255)` and the per-effect ParamK constants are `Abs=2000, Lockup=2640, Threshold=3080, Engine=1000` (verified against the protocol note's hardware captures). Frequency encoding is `u16 = round(hz * 65536 / 200)` saturating, amplitude is `u16 = clamp(round(amp01 * 65535), 0, 0xFFFF)`. Keepalive is the literal `7e 00 00 12 9d` â€” `MozaMBoosterProtocol.BuildKeepalive()` â€” sent every ~500 ms via the one-shot FIFO regardless of effect state; without it the motor drops connection state per protocol note Â§ 3. Disable frames (one per effect ID with `enable=0` and all params zero) fire at every effect-deactivation edge AND for all four effect IDs in `MBoosterDeviceController.Dispose()` so the motor doesn't latch the last-active waveform after the port closes.

  **Stream lane**: all motor frames go through `StreamKind.MBoosterEffect` on each controller's connection â€” a single coalesced lane per device (latest-wins) is sufficient because the worker emits at most one motor frame per ~20 ms tick (50 Hz per protocol note Â§ 4) and the four effects rotate through `ProcessEffect` sequentially within each tick. Keepalive frames go via the one-shot FIFO so they aren't coalesced against motor frames. Frame layout constants live on `MozaMBoosterProtocol` (`GroupMotorWrite = 0x24`, `DeviceMotor = 0x12`, `CmdMotorWrite = 0xb1`, `MotorPayloadLen = 0x09`).

  **Telemetry â†’ effect parameters** is per protocol note Â§ 4 pseudocode, computed inside `MBoosterEffectWorker.Tick`. `MozaPlugin.DataUpdate` builds an `MBoosterTelemetrySnapshot` (game-running flag, RPM, brake 0..1, ABS-active bool, vehicle speed m/s, average wheel speed m/s) per SimHub tick and fans it out to every controller via `MozaMBoosterRegistry.OnDataUpdate`. The worker then maps:
  - **ABS**: `freq = 18 + abs_active * 12` Hz, `intensity = abs * absScale`, fires when `abs > 0.1`. SimHub's `ABSActive` is converted defensively via `Convert.ToInt32` because the backing type varies across SDK versions (some int, some bool).
  - **Lockup**: `freq = 40 + brake * 30` Hz, `intensity = brake * lockupScale`, fires when `brake > 0.8 && vehicle_speed > 5 && avg_wheel_speed < vehicle_speed * 0.3`. Per-wheel speeds aren't compile-time properties on this SimHub SDK's `StatusDataBase`, so `avg_wheel_speed` is fed as 0 and a fallback path triggers on `brake > 0.9 && vehicle_speed > 5` to keep the effect meaningful without per-wheel data.
  - **Threshold**: `freq = 60 + brake * 30` Hz, `intensity = brake * thresholdScale`, hysteresis-latched on rising-edge `brake > 0.6`, released on `brake < 0.3`.
  - **Engine**: `freq = clamp(rpm / 20000 * 200, 10, 200)` Hz (~10â€“100 Hz typical), `intensity = engineScale * 0.1` (constant amplitude, capped at 10 % per the doc).

  Each effect goes through `ProcessEffect` which advances the per-effect phase + elapsed, calls the synthesizer, encodes via `MozaMBoosterProtocol.EncodeFreq` / `EncodeAmp` / `ComputeParam1`, and submits the motor frame via the stream lane. On the deactivation edge (intensity drops to 0), one disable frame goes out via the one-shot FIFO and the worker stops emitting for that effect until the next activation.

  **Settings storage** is on `MozaProfile.MBoosterSettings : Dictionary<string, MBoosterDeviceSettings>` keyed by device identity. `MBoosterDeviceSettings` carries `Role` (enum), four `MBoosterEffectSettings` (ABS / Lockup / Threshold / Engine â€” each with `Enabled` + `IntensityPct`), plus calibration fields (`Direction`, `Min`, `Max`, `CurveY[5]` â€” written via `MBoosterDeviceController.SendIntWrite` / `SendFloatWrite` against `mbooster-*` commands in `MozaCommandDatabase`). ### Adding New Device Settings

When adding a new setting that is written to the device, it must also be saved/restored with the profile system. Pick the storage tier first â€” see the four-tier classification in the Profile System section above (plugin-global / per-wheel-page / per-(profile Ã— wheel-page) / per-game profile baseline) â€” then walk the workflow:

1. **`MozaCommandDatabase.cs`** â€” Add command definition (name, device, read/write groups, command ID, payload size, type)
2. **`MozaDeviceManager.cs`** â€” Add device type mapping in `GetDeviceId()` if it's a new device
3. **`MozaProtocol.cs`** â€” Add device ID and read/write group constants if needed
4. **`MozaData.cs`** â€” Add volatile field(s) and `UpdateFromCommand` case(s)
5. **`Devices/DeviceProber.cs`** â€” Add to `StatusPollCommands` (if it's a detection probe) and the appropriate per-device read array (e.g. `BaseSettingsReadCommands`, `NewWheelSettingsReadCommands`, `HandbrakeSettingsReadCommands`, etc.) so it's read after that device is detected. Add detection logic in `DetectDevices()` if needed. Push the new value through the matching `Apply*ToHardware` method in `Hardware/HardwareApplier.cs` (sentinel-guard the write: `if (value >= 0) _deviceManager.WriteSetting(cmd, value)`). For wheel-overlay fields source from `Eff(overlay?.X ?? -1, profile.X)`; for profile-level fields source from `profile.X` directly; for per-wheel-page fields read the matching `MozaPluginSettings.*ByPageGuid[pageGuid]` entry.
6. **Storage** â€” depends on the chosen tier:
   - **Per-game profile baseline** â†’ add the property to `MozaProfile` directly, copy in `CopyProfilePropertiesFrom()`. Only add a `CaptureFromCurrent()` line if the value flows from device reads via `_data`; UI-edited fields are written to the profile by handlers directly and capture would clobber them.
   - **Per-(profile Ã— wheel-page)** â†’ add the property to `WheelOverride` (nested in `MozaProfile.cs`), copy in `WheelOverride.Clone()`. No `CaptureFromCurrent` capture â€” UI handlers write the overlay directly.
   - **Per-wheel-page (shared across games)** â†’ add a `Dictionary<Guid, â€¦>ByPageGuid` on `MozaPluginSettings`. Add a getter/setter on `MozaPlugin` that resolves the current page GUID (via `GetCurrentWheelPageGuid` â†’ `MozaDeviceConstants.ResolveWheelGuid`) and reads/writes the dict. Add a migration step under `Settings/SettingsMigrator.MigrateToSchemaV2` that (a) drains pre-removal per-overlay values from `WheelOverride.LegacyJsonFields` and (b) seeds every known wheel page GUID from any pre-existing flat field. Bump `SettingsSchemaVersion`.
   - **Plugin-global** â†’ add the property directly to `MozaPluginSettings`.
7. **`SettingsControl.xaml`** (or `Devices/MozaWheel/Dash/BaseSettingsControl.xaml`) â€” Add UI controls
8. **UI handler** â€” Update the matching control's `_ValueChanged`/`_Click` handler. Pattern by tier:
   - Profile-level field: `_plugin.UpdateActiveProfile(p => p.X = val)` then `_plugin.WriteIf<Device>(cmd, val)` then `_plugin.SaveSettings()`.
   - Wheel-overlay field: `_plugin.UpdateActiveWheelOverlay(o => o.X = val)` then `_plugin.WriteIfWheelDetected(cmd, val)` then `_plugin.SaveSettings()`.
   - Per-wheel-page field: `_plugin.ActiveXxx = val;` (the setter auto-routes to the dict via the page-guid resolver) then `_plugin.WriteIf<Device>(cmd, val)` then `_plugin.SaveSettings()`.
   - Plugin-global field: `_plugin.Settings.X = val;` then `_plugin.SaveSettings()`.
   - Color: use the matching `_plugin.WriteColorIf<Device>(cmd, r, g, b)` helper.
   - Array: use `_plugin.WriteArrayIfWheelDetected(cmd, payload)`.
   - `SaveSettings()` is debounced (500 ms). UI handlers update `_data.X` too where applicable so the next 500 ms refresh tick shows the new value immediately without waiting for the device echo.

Every setting that writes to the device on UI change must round-trip through profiles or the per-wheel-page dicts. If it's only in a transient field, it will be lost on game/profile switch.

**Host-rendered settings** (e.g. AB9 engine vibration intensity/frequency) skip steps 1â€“4 entirely â€” there is no `MozaCommandDatabase` entry because the wire frame is built inline by the device manager, no `MozaData` field because the value lives only in the profile, no detection probe because the device never persists the setting. The workflow collapses to step 6 (`MozaProfile` property) + step 7/8 (UI + handler). The UI handler does **not** call any `Send*` device write â€” it just updates the profile and `SaveSettings()`. Whatever periodic worker reads the setting (e.g. `Ab9EngineVibrationLoop`) picks up the new value on its next tick. **One-shot host-side config writes that ARE device-persisted but bypass the command-DB** (e.g. AB9 gear-shift-vib intensity â€” frame shape doesn't fit the single-byte slider convention) follow steps 6â€“8 plus an explicit `Send*` call in the UI handler AND in `ApplyAb9ToHardware` so the value is pushed on device attach / profile load.

**Settings storage and migration.** `MozaPluginSettings` holds plugin-global toggles, the SimHub-managed `ProfileStore`, the three per-wheel-page dicts (`WheelMzdashFolderByPageGuid`, `WheelTelemetryEnabledByPageGuid`, `WheelTelemetryEraByPageGuid`), and one-shot scratch space for legacy stores still readable from disk. `MozaPluginSettings.SettingsSchemaVersion` tracks the migration cutover and gates `Settings/SettingsMigrator.MigrateToSchemaV2`:
- v2: legacy UID-keyed slots (`PerWheelSlots`, `TelemetryByWheelUid`, `WheelMzdashFolderByUid`, `TelemetryChannelMappingsByWheel`) translated into `profile.WheelOverridesByPageGuid` / `profile.TelemetryChannelMappings`.
- v3: flat telemetry fields drained into overlays.
- v4: `TelemetryMzdashFolder` lifted off the overlay into `WheelMzdashFolderByPageGuid`.
- v5: `TelemetryEnabled` lifted off the overlay into `WheelTelemetryEnabledByPageGuid`.
- v6: `TelemetryWheelEra` lifted off the overlay into `WheelTelemetryEraByPageGuid`.

The migration is gated by `SettingsSchemaVersion >= 6`; users at any prior version run forward through every step in one pass. `WheelOverride.LegacyJsonFields` (a `[JsonExtensionData(WriteData = false)]` catch-all) preserves removed JSON keys through deserialization so each migration step can drain them; afterwards the dict is cleared and the `WriteData = false` flag stops them from re-serializing. `ClearLegacyAfterMigration` empties the legacy dicts and remaining flat fields once each successful pass completes. Use `MozaDeviceConstants.ResolveWheelGuid(prefix)` for any new code that needs to map a wheel-model prefix to a SimHub page GUID â€” never hard-code GUIDs.

### String-typed channels (out-of-band on sess=0x01)

Channels declared with `compression: "string"` in `Telemetry.json` (`TrackId`, `CarModel`, `SessionTypeName`, `Flag_Name`, `patch/TrackName`, `patch/DisplayTrackName`, vehicle tags, tyre compound types, etc. â€” 23 in total) **do not bit-pack into the value frame**. They ride a separate transport: `type=0x05` sub-msgs on session 0x01 with `[type=0x05][size_LE u32 = 2+strlen][channel_idx u8][0x80|strlen u8][ASCII strlen bytes]`. Full wire-format reference: [`protocol/sessions/session-0x01-channel-protocol.md`](protocol/sessions/session-0x01-channel-protocol.md); discovery details: [`protocol/findings/2026-05-14-sess01-channel-protocol-and-string-values.md`](protocol/findings/2026-05-14-sess01-channel-protocol-and-string-values.md).

Plumbing inside the plugin:

- **`DashboardProfileStore.BuildMultiStreamProfile` / `BuildProfileFromCatalog` / `BuildPerWidgetProfile`** route any URL whose Telemetry.json compression is `"string"` to `MultiStreamProfile.StringChannels` (separate from the bit-packed `Tiers` list). The wheel does NOT expect string-typed entries in any `type=0x01` tier-def, so a `string` compression must NEVER be assigned a `CompressionTable` entry â€” that would land the channel in a bit-packed slot the firmware refuses to bind.
- **`ChannelCatalogParser.FindIdxByUrl(url)`** maps a URL string to its 1-based catalog idx as announced by the wheel on b2h sess=0x01 `type=0x04` records. The wheel re-indexes URLs per dashboard, so this lookup is the authoritative source; idx values cannot be hardcoded per channel.
- **`StringValueBuilder.Build(channelIdx, value)`** builds the type=0x05 sub-msg body. Output is byte-exact against the captured PitHouse wire (verified against `imola`â†’idx7 and `ks_laguna_seca`â†’idx7 from `bridge-20260514-204307.jsonl`). Max strlen 127 â€” high bit of the flag byte is the string-type signal, low 7 bits are the length.
- **`TelemetrySender.TickEmitStringValues()`** runs immediately after `TickEmitValueFrames` in the steady-state tick loop. Per-URL state dict (`_stringChannelState`) tracks the last value + last-emit `Environment.TickCount` so the wire only carries a re-send when the value changes, with a 2-second keepalive floor when it doesn't. Records are chunked via `TierDefinitionBuilder.ChunkMessage(session: 0x01, seq: ref _session01OutboundSeq)` so the existing chunk CRC + framing path is reused.
- **Test mode** uses the channel's `TestSignal.StringValue` (assigned at dashboard-load by `TestSignalCatalog.Resolve` when `data_type == "string"`, defaulting to `"DEMO"`). The bit-packed test path's `TestSignalGenerator.Compute` returns `0.0` for string channels â€” the string is read directly off `TestSignal.StringValue`, not through the numeric pipeline.
- **Live SimHub property wiring is NOT yet implemented.** When the game is running but TestMode is off, the emitter currently re-emits the static `TestSignal.StringValue` ("DEMO" by default) â€” placeholder until per-channel SimHub property reads land. See the `TODO(2026-05-15)` in `TickEmitStringValues`.

Cadence observed from PitHouse (single-channel `TrackId` sample): bursts of 3â€“6 retransmits on value-change, 10â€“30 s silence at steady state. Plugin re-implements this approximately with the 2-second keepalive floor plus event-driven sends on value-change.

### Dashboard Switch State Machine

`DashboardBindingCoordinator.ApplyTelemetryDashboardFromProfile(MozaProfile)` (proxied via `MozaPlugin`) is the single entry point for binding the wheel's displayed dashboard to whatever the current SimHub game profile saved. It fires from `ApplyProfile` at plugin Init / game-switch and from `PollStatus`'s retry loop (driven by the coordinator's `_pending` snapshot â€” see below). Returns `true` once the apply has resolved (or been permanently dropped), `false` to defer.

**Inputs.** The saved `TelemetryDashboardKey` (`wheel:<id>` / `file:<filename>:<sha1-8>` / `builtin:<name>`) resolves to a *target name*; that name is then looked up in `WheelStateForDiagnostics.ConfigJsonList` (the wheel's alphabetical dashboard library) to find a *target slot* (uint32, the kind=4 frame index). All three key kinds funnel through the same slot lookup â€” previously only `wheel:` keys emitted kind=4, which silently left the wheel pointing at the prior game's dashboard when a `file:`/`builtin:` profile was activated.

**Readiness gate (defer if not met).** Apply defers and returns `false` when:
1. `sender == null` â€” sender hasn't been constructed yet (plugin Init has fired but `StartTelemetryIfReady` hasn't created the sender).
2. `!sender.IsActive` â€” sender exists but is in `Starting`/`Preamble`/`Idle`. A kind=4 emitted before preamble completes is silently dropped by the wheel.
3. `sender.IsInSilenceCooldown` â€” a previous kind=4 went out on the wire within the silence window (`SilenceGate.StopReopenSilenceMs` = 11 s, or `HotSwitchCooldownMs` = 200 ms in hot-renegotiation mode); the wheel's sess=0x09 binding-state timeout is still running and another emit would race the in-flight restart.
4. `state == null || state.ConfigJsonList.Count == 0` â€” `ConfigJsonClient.LastState` hasn't produced its first parse yet (or every chunk dropped). The `_cachedLastState` static fallback fills this in across plugin reloads in the same SimHub process so a cold-start chunk drop doesn't leave the apply blind.

`PollStatus` retries via the coordinator's `_pending` field â€” a `PendingApply` record (`Key` / `DeadlineUtc` / `RetryCount` / `LastRetryWarnUtc`) protected by `_stateLock` so the PollStatus, dispatcher, and profile-change threads can read/write it atomically. The deadline is `PendingProfileKeyTimeout = 5 min`; `_lastApplyDeferReason` (also lock-guarded) throttles the per-defer Debug log to one line per state transition AND surfaces via `PendingDashboardApplyDescription` so the UI status label reads e.g. `Switching dashboardâ€¦ (sender=not-Active cooldown=False) â€” N frames sent` instead of a bare "Switching dashboardâ€¦". The retry path increments `RetryCount` each tick and emits a Warn line every `RetryWarnInterval = 30 s` so a silent multi-minute wait doesn't pass unnoticed; timeout escalates from the prior Info log to a Warn line including the retry count. All `_pending` mutations after `ApplyTelemetryDashboardFromProfile` returns use `ReferenceEquals(_pending, snap)` so a concurrent `SetPendingDashboardKey` / `ClearPendingDashboardKey` from another thread cannot be overwritten.

**Apply path (slot resolved, gate passed).** Two paths based on whether the wheel is already on the target slot:

1. **Slot already bound** (`sender.WheelReportedSlot == slot` OR `sender.LastEmittedKind4Slot == slot`). The wheel itself emits a `04 <slot:u32> ...` record on sess=0x02 b2h after every dashboard switch â€” that's wire-level ground truth, parsed by `MaybeUpdateWheelReportedSlot` and exposed as `WheelReportedSlot`. The static `LastEmittedKind4Slot` survives plugin reload, so game-switching between profiles that share a dashboard hits this path on the first apply. Skip: no kind=4 emit, no pipeline action. Logs `no wire action needed`.
2. **Slot mismatch** â€” emit kind=4 to the target slot via `OnDashboardSwitched((uint)slot)` â†’ `SwitchToProfile` â†’ `SendDashboardSwitch`. With `EnableHotRenegotiation` (default), no Stop+Start cycle; the tick handler emits a paced multi-emission tier-def burst echoing the wheel's END marker (see [`protocol/tier-definition/handshake.md`](protocol/tier-definition/handshake.md) Â§ In-game dashboard switch).

**Wheel-initiated switch handling** (`MaybeUpdateWheelReportedSlot` + `TelemetrySender.WheelInitiatedSwitch` event). ### Key Protocol Details

- Message format: `[0x7E] [length] [request_group] [device_id] [command_id...] [payload...] [checksum]`
- Response parsing: toggle bit 7 of request_group, swap nibbles of device_id, then match command_id (with 0xFF wildcards)
- Read messages use a zero-filled payload of the declared byte width (matches boxflat's `prepare_message`). Some wheels silently drop reads with a non-zero payload, even though the protocol nominally ignores payload bytes on reads
- All multi-byte integers are big-endian; floats are byte-reversed
- `MozaSerialConnection` uses two write lanes. **One-shot FIFO** (`_oneShotQueue`) carries session traffic, settings writes, and probes â€” ordered, paced at 4 ms between consecutive one-shots (boxflat's proven timing; Moza bases drop rapid settings writes like the 30+ `ApplyProfile` burst). Under bandwidth pressure the `WriteBudget` token-bucket extends the 4 ms gate further (still order-preserving FIFO, never drops) so the wheel can drain its receive buffer before we pile on more. Pacing uses `Stopwatch.GetTimestamp()` (monotonic 64-bit) so the gate stays correct across long uptime; the previous `Environment.TickCount` int math wraps every ~24.8 days. **Stream slots** (`_streamSlots` â€” one per `StreamKind`: tier dash 0â€“7, enable, sequence, mode, ab9-engine-vibration, ab9-engine-pulse, ab9-trigger-a, ab9-trigger-rpm, ab9-trigger-extra, ab9-low-rate) carry periodic telemetry â€” latest-wins coalescing, unpaced and **never bandwidth-gated**. `SendStream(kind, bytes)` overwrites any predecessor in the same slot via `Interlocked.Exchange`, so a stale tier-0 frame can never reach the wire if a newer one was already generated. `WriteLoop()` drains one-shots first (paced only when the previous write was also a one-shot), then sweeps stream slots back-to-back. A pooled stuff buffer emits each frame in a single `_port.Write()` call via `MozaProtocol.StuffFrame`. `FlushPendingWrites()` drops both lanes + calls `_port.DiscardOutBuffer()` â€” used by `TelemetrySender.Pause()` so Test-Stop halts wire traffic within ~1 ms instead of bleeding through the 16 KB OS buffer (~1.4 s at 115200 baud). Pre-2026-04-22 (queue-drift fix): single `ConcurrentQueue`, unconditional 4 ms pacing between every dequeue, per-byte `_port.Write(msg, i, 1)` for each stuffed body byte. Under steady telemetry (3+ frames/tick Ã— 4 ms pacing) drain time hit the 30 ms tick budget, so any OS jitter grew backlog until the displayed dashboard values lagged the game by hundreds of ms over a single lap. **Stream-lane gating** (the `WriteBudget.MayDrainStreams()` check that suppressed stream writes when the rolling 1-s window crossed the budget) was removed 2026-05-09 â€” the gate held value frames for ~700 ms-1 s during routine bursts and starved the wheel into dropping configJson chunks, while latest-wins coalescing already provides natural backpressure (fresh `SendStream` overwrites pending slots before the writer drains them) and `SerialPort.Write` blocks when the OS write buffer fills, providing a hard physical-layer gate. **Read path** unchanged: bulk-read thread polls `BytesToRead` at 2 ms, pulls whatever's available via `Read(buf, 0, avail)` into a 64 KB in-memory rx buffer, parses as many complete frames as possible per pass. Per-byte `ReadByte()` was replaced 2026-04-22 because its ~100 Âµs per-call cost under Wine starved the reader during bursts and lost chunks. Wine's blocking `Read(buf, 0, full_count)` does NOT return early when fewer bytes are available â€” it hangs â€” so the BytesToRead guard is load-bearing
- 0x7E byte stuffing: every `0x7E` byte in a frame's body (group, device, payload, checksum) is doubled on the wire. `MozaProtocol.StuffFrame(frame, dest)` writes the start+len header raw then walks body bytes emitting each and appending an extra `0x7E` on match; `StuffedFrameSize(frame)` pre-computes the output size so `WriteLoop`'s pooled stuff buffer can be grown on demand. The single-shot stuffed write replaced per-body-byte `_port.Write(msg, i, 1)` calls in 2026-04-22 â€” the old path issued 10â€“32 syscalls per frame, compounding Wine's per-call overhead. `ReadLoop()` reverses stuffing: after reading the start+len header, it reads N+3 decoded bytes, collapsing any `0x7E 0x7E` pair back to a single `0x7E`; an unpaired `0x7E` marks the start of a new frame and the current frame is abandoned. Without this, payload 0x7Es (common in zlib-compressed session data and seq counters) desync the read stream. **Checksum must also account for stuffing** â€” the wire-level sum includes the duplicated `0x7E` bytes, so `CalculateWireChecksum()` adds `count(0x7E in body) Ã— 0x7E` to the raw sum. Pre-2026-04-22 the plugin's `CalculateChecksum` omitted this term and dropped ~20% of zlib-bearing session chunks (configJson state, dashboard uploads) whose compressed payloads contain 0x7E bytes. See `docs/protocol/wire/checksum.md`
- Session close frame: the type=0x00 end marker carries a 6-byte payload `7C 00 [session] 00 [ack_lo] [ack_hi]` with `len=6`. A 4-byte payload advertised as len=6 causes the wheel/sim to over-read into the next frame and kill the read loop. `SendSessionClose()` zero-fills the ack_seq bytes when simply reclaiming a stale session
