# Project Overview

SimHub plugin for MOZA Racing hardware providing two-way telemetry: streams game data (speed, RPM, gear, lap times, fuel, tyre wear, etc.) to the wheel dashboard display, drives wheel/dashboard RPM and flag LEDs, and allows configuring wheelbase settings. Also supports standalone USB dashboards (CM2 Racing Dash, PID `0x0025`) that connect without a wheelbase — the same dashboard pipeline retargets to the CM2's bridge/main `dev_id=0x12` (see [Standalone dashboard pipeline](#standalone-dashboard-pipeline-cm2) below). Uses a custom binary serial protocol reverse-engineered from the [boxflat](https://github.com/Lawstorant/boxflat) project.

## Building from Source

The project targets .NET Framework 4.8 (x86) and uses the `Microsoft.NETFramework.ReferenceAssemblies.net48` NuGet package so it can cross-compile on Linux without Mono. The built DLL runs on Windows under SimHub.

### Building on Windows

#### Prerequisites

- [VS Code](https://code.visualstudio.com/) with the [C# Dev Kit](https://marketplace.visualstudio.com/items?itemName=ms-dotnettools.csdevkit) extension
- .NET SDK 8.0+ ([download](https://dotnet.microsoft.com/download))

#### Steps

1. **Open the project folder** in VS Code.

2. **Build** from the VS Code terminal:

   ```
   dotnet build -c Release
   ```

3. **Copy the DLL** to your SimHub folder:

   Copy `bin/x86/Release/MozaPlugin.dll` into your SimHub installation directory.

   Or set the `SIMHUB_PATH` environment variable to have it copied automatically on build:

   ```
   set SIMHUB_PATH=C:\Program Files (x86)\SimHub
   dotnet build -c Release
   ```

   PowerShell:
   ```powershell
   $env:SIMHUB_PATH = "C:\Program Files (x86)\SimHub"
   dotnet build -c Release
   ```

4. **Restart SimHub.** The plugin appears under Settings > Plugins as "MOZA Control".

### Cross-Compiling on Linux

You can build the plugin entirely on Linux. The .NET SDK can target .NET Framework 4.8 using the `Microsoft.NETFramework.ReferenceAssemblies.net48` NuGet package (already included in the `.csproj`).

#### Prerequisites

- .NET SDK 8.0+ (`dotnet-sdk` package from your distro or [Microsoft repos](https://dotnet.microsoft.com/download))

#### Steps

1. **Install the .NET SDK** if you don't have it:

   ```bash
   # Arch Linux
   sudo pacman -S dotnet-sdk

   # Ubuntu/Debian
   sudo apt install dotnet-sdk-8.0

   # Fedora
   sudo dnf install dotnet-sdk-8.0
   ```

2. **Build:**

   ```bash
   dotnet build -c Release
   ```

   The output DLL will be in `bin/x86/Release/MozaPlugin.dll`.

3. **Copy the built DLL to your Windows SimHub installation:**

   ```bash
   # Example: copy to a Windows machine via scp
   scp bin/x86/Release/MozaPlugin.dll user@windows-pc:"C:/Program Files (x86)/SimHub/"

   # Or copy to a shared folder, USB drive, etc.
   cp bin/x86/Release/MozaPlugin.dll /mnt/shared/SimHub/
   ```

4. **Restart SimHub** on Windows.

#### Notes

- The `Microsoft.NETFramework.ReferenceAssemblies.net48` NuGet package provides the .NET Framework 4.8 reference assemblies, so you do **not** need Mono or Windows installed.
- The build produces a standard .NET Framework DLL that runs natively on Windows under SimHub.
- SimHub DLLs in `libs/SimHub/` are reference-only (`Private=false`) and not copied to output.
- The build produces a single output DLL with no additional runtime dependencies to deploy.

### CI/CD

- **Build**: Every push to `main` and every PR is built automatically via GitHub Actions.
- **Dev pre-release**: Every push to `dev` builds a Release and publishes a GitHub pre-release (a per-commit `dev-<sha>` tag plus a rolling `dev-latest` tag) — this is where active-development builds come from.
- **Release**: Pushing a `v*` tag (e.g., `v0.2.0`) builds a Release, generates a changelog, and publishes a GitHub Release with the DLL (device definitions are embedded in the DLL).
- **SimHub dependency updates**: A daily workflow checks for new SimHub releases and creates a PR to update `libs/SimHub/`.

## Architecture

### Code Organization

The plugin's four previously-monolithic codebehinds and senders were split into focused collaborator classes in a 5-PR refactor (2026-05-18). `MozaPlugin.cs` and `Telemetry/TelemetrySender.cs` remain the orchestrators; behavior they used to host inline now lives in dedicated files:

- **`Devices/DeviceDetectionState.cs`** — Volatile per-device detection flags (wheel/dash/handbrake/pedals/hub/AB9, plus extended LED group mask) read by poll/UI/serial/telemetry threads.
- **`Devices/DeviceProber.cs`** — `DetectDevices` response dispatch + per-device read command arrays. **Wheel reads are split into two phases**: `NewWheelCoreReadCommands` (LED-layout-independent — telemetry mode, sleep config, paddles / clutch / stick) fires at the initial new-protocol-wheel detect site, before `WheelModelInfo` is known; `BuildNewWheelLedReadCommands(WheelModelInfo info)` fires from the `wheel-model-name` handler once the model resolves, returning only the LED-related reads the wheel actually has (per-RPM color reads capped at `info.RpmLedCount`, button reads capped at `info.ButtonLedCount`, knob mode / signal-mode reads gated on `info.KnobCount > 0`, flag-color reads only when `info.HasFlagLeds`, the `wheel-single-brightness` / `wheel-ambient-brightness` group-probes only for unknown wheels). The split fixes the older-CS-style "wheel keeps timing out reads" pattern — the old flat 51-command list blasted at every wheel hammered the wheel with per-LED reads for hardware it didn't have; each unanswered read burned its 3-attempt retry budget in `PendingResponseTracker`. Also exposes three idempotent first-sight detection helpers (`MarkDashDetected` / `MarkHandbrakeDetected` / `MarkPedalsDetected`) callable from both the data-response parser case branches AND the new empty-presence-probe ACK path in `MozaPlugin.OnPresenceProbeAck`, so the dispatch order in/out doesn't matter — whichever response form arrives first flips the flag.
- **`Devices/Ab9EngineVibrationWorker.cs`** — The 91 Hz host-rendered AB9 engine-vibration loop and its `K = 3.95e11` period formula.
- **`Settings/SettingsMigrator.cs`** — Schema v0→v8 settings migration + `SeedProfileBaselineFromFlatFields` baseline seeding.
- **`Hardware/HardwareApplier.cs`** — Every hardware-side write path: `ApplyProfile`, the seven `Apply*ToHardware` methods, and the detection-gated `WriteIf*` / `WriteColorIf*` / `WriteArrayIf*` family.
- **`Telemetry/DashboardBindingCoordinator.cs`** — Profile-driven dashboard switch (kind=4 emit), wheel-initiated switch handling, telemetry-pipeline lifecycle gates, `ApplyTelemetryDashboardFromProfile`.
- **`Telemetry/SimHubPropertyResolver.cs`** — `ResolvePropertyAsDouble`/`AsString`, `@internal/` channel resolution, `GetAllSimHubPropertyNames`.
- **`Telemetry/Inbound/TelemetryInboundDispatcher.cs`** — Inbound 0xC3 routing keyed on `_sender.TargetDeviceIdSwapped` (default 0x71 for wheel; 0x21 for CM2 bridge/main, with a wider 0x21/0x41/0x71 fan-in when `IsStandaloneDashboardTarget`): fc:00 acks, type=0x81 device-init, per-session type=0x01 dispatch.
- **`Telemetry/Watchdog/SessionWatchdogManager.cs`** — Four wheel-session watchdogs: sess=0x09 retry, sess=0x02 engagement, configJson gap escalation, configJson stuck-state.
- **`Telemetry/Display/WheelSlotTracker.cs`** — `MaybeUpdateWheelReportedSlot` with strict type-04 validation (rejects false-positive slots from collisions with tier-def 0x04 records).
- **`Telemetry/PropertyPushQueue.cs`** — Brightness-blanking coalescing via per-(session,kind) seq supersedence.
- **`Telemetry/Frames/TierDefinitionEmitter.cs`** — `SendTierDefinition` + blind retransmit for Era2024/25 firmware that doesn't ack sess=0x01.
- **`Telemetry/Lifecycle/SilenceGate.cs`** — Two host-side silence timers (Stop→Start ~11 s wait + post-switch UI cooldown). Statics so they survive plugin recycle within a SimHub process; constructor takes a `Func<bool> isHotRenegotiationEnabled` so the cooldown gate can shorten itself when hot mode is on.
- **`Telemetry/Lifecycle/HotSwitchCoordinator.cs`** — Hot-renegotiation burst state machine. `ArmBurst` / `ShouldEmitThisTick(catalogLastActivityTickMs, catalogWheelEndMarkerTickMs)` / `MarkEmission` collapse the prior ~100-line first-emission/pacing tree inside `TickGrowSubscriptionIfCatalogStable` into a short decision function with explicit branches.
- **`Telemetry/Lifecycle/CatalogResyncProbe.cs`** — Throttle + has-fired memo for the catalog re-sync probe. Decision logic (slot lookup, wheel-on-target shortcut, deferred ThreadPool emit) stays in `TelemetrySender.ScheduleCatalogResyncProbe`; this class only owns the timestamp.
- **UI controllers** (`UI/UiHelpers.cs`, `UI/DiagnosticsTextBuilder.cs`, `UI/DiagnosticsBundleWriter.cs`, `UI/DashboardLibraryResolver.cs`, `Devices/WheelUi/{ChannelMappingRow,WheelUiHelpers,ChannelMappingRowFactory}.cs`) — Pure-function text builders, ZIP bundle writer, library-path resolution, and per-wheel UI helpers shared across `SettingsControl.xaml.cs` and `MozaWheelSettingsControl.xaml.cs`.
- **`ControlMapper/MozaVariantProvider.cs` + `ControlMapper/ControlMapperBridge.cs`** — SimHub Control Mapper `IVariantProvider` integration so the Add Source Controller dropdown shows each MOZA wheel under its friendly name (CS Pro / KS Pro / KS / FSR V2 / …) and `SharpHelper.AquireController`'s per-mapping variant gate dispatches input only to the saved mapping whose stored Variant matches the currently-attached wheel. Full IL reverse-engineering of the SimHub side lives in [`simhub.md` § Control Mapper Variant Providers](simhub.md#control-mapper-variant-providers); this entry summarizes our adapter. `MozaVariantProvider` implements `IVariantProvider.GetVariant(int vid, int pid)` by reading `MozaData.WheelModelName`, extracting the prefix via `WheelModelInfo.ExtractPrefix`, and returning `WheelModelInfo.GetFriendlyName(prefix)` when `vid == MozaProtocol.VendorId` and the PID resolves to `MozaUsbIds.IsWheelbasePid` or `IsHubPid`; null otherwise. Exposes a `public event EventHandler VariantChanged` matching the Fanatec/Simucube convention; fires it from `Poll()` when the resolved variant string transitions. `ControlMapperBridge` handles the reflection chain (`ControlMapperPlugin → remapperWorker → variantHelper → VariantProviders`) defensively — every step that fails calls `LogGiveUp("…")` once and disables the bridge for the session rather than throwing. Lazy-materializes the provider list via `VariantHelper.Start()` when the user hasn't yet enabled Control Mapper's "Recognize Individual Wheels" toggle (without which `VariantHelper.GetVariant` returns null for every query, killing every mapping). After registering, calls `RemapperWorker.UpdateControllerList()` once so any wheel already plugged in at SimHub launch gets re-keyed with the MOZA variant on the first pass instead of waiting for the wheel-attach `VariantChanged` to fire it later. Runs three workarounds keyed off the `ControllerMappings.CollectionChanged` event subscribed during registration: (a) **Detach shared Description on `Add`** — `set_ControllerDescription` stores by reference and `UpdateOrAdd` into `UnmappedControllers` uses a ControllerID-only predicate whose `CopyFrom` updater mutates the shared description on every `UpdateControllerList` tick, so saved mappings inherit Variant rewrites whenever the wheel changes. The fix clones the new mapping's `ControllerDescription` (`Activator.CreateInstance` + `CopyFrom`) and stamps `Variant` with the currently-detected wheel before SimHub's next tick sees it. (b) **Auto-create per-variant mapping** in `Poll()` — when current variant has no matching `ControllerSourceMapping` AND at least one MOZA mapping already exists (signals the user has engaged with the feature so we don't auto-add on cold start), build a fresh CSM with a cloned description and add to `ControllerMappings`. This bypasses SimHub's Add Source Controller UI, which dedupes the wheelbase by ControllerID and won't offer it a second time once any saved mapping exists. A `HashSet<string> _autoCreatedVariants` (per-session, not persisted) gates re-creation: deleted auto-adds stay deleted until the next plugin Init. (c) **Dispatcher marshal** — `ControllerMappings` is a WPF-bound `ObservableCollection<ControllerSourceMapping>` whose `CollectionView` throws `"This type of CollectionView does not support changes to its SourceCollection from a thread different from the Dispatcher thread"` on background-thread `Add`. Background mutation half-succeeds (`Count` rises, view stays stale), so the auto-add routes through `Application.Current.Dispatcher.BeginInvoke` when `dispatcher.CheckAccess()` is false. Wired into `MozaPlugin` via a `ControlMapper.ControlMapperBridge?` field (constructed in `Init` only when `MozaPluginSettings.EnableControlMapperVariants` is true — hidden setting, default true, flippable via JSON file edit if a future SimHub assembly change breaks the reflection chain); registration tries immediately and retries up to ~50 ticks (~0.8 s at 60 Hz) in `DataUpdate` for slow `ControlMapperPlugin` load order; `Poll()` runs every `DataUpdate` tick; `Unregister()` removes the provider in `End` so plugin reload without SimHub restart doesn't leave a dead entry in `VariantHelper.VariantProviders`. Logging keeps Info-level reserved for significant lifecycle/action events (`registered MozaVariantProvider`, `auto-create: added new mapping`, `removed`) and routes the verbose per-tick diagnostics (`CM diag` snapshots, `variant changed`, `detached shared Description`) to Debug.

- **Custom WPF control library** (`UI/Controls/*.cs` + `Themes/{MozaTheme,MozaIcons,Generic}.xaml`) — Themed primitives used by the plugin pane + device controls. `SectionCard` (`HeaderedContentControl` with accent stripe + icon + subtitle), `SegmentedControl` (`ListBox` subclass with radio-group styling; used wherever a small fixed choice replaces a `ComboBox`), `OffOnToggle` (`ToggleButton` showing OFF / ON pills next to an optional inline label — drop-in replacement for `<CheckBox Content="…">`), `PaletteStrip` (16 curated swatches + a CUSTOM hue-picker chip that defers to the legacy `ColorPickerDialog` via the static `PaletteStrip.CustomPickerFactory`), `KnobRingViz` (12-LED ring + centre swatch SVG-equivalent, `Canvas`-based), `BandwidthSparkline` (rolling-window line chart with optional fixed-ceiling `MaxValue` for absolute-saturation rendering), `ConnectionPill` / `MozaCurveEditor` / `SteeringArc` / `TemperatureGraph` (visual primitives for telemetry / FFB / pedals; the FFB equalizer is `MozaCurveEditor` styled via the `MozaEqualizerLineStyle` key, not a separate control). Each control declares its default style key via `DefaultStyleKeyProperty.OverrideMetadata`; the styles live in `Themes/Generic.xaml` (referenced via `[assembly: ThemeInfo(ResourceDictionaryLocation.None, ResourceDictionaryLocation.SourceAssembly)]` in `Properties/AssemblyInfo.cs`). Theme tokens (color brushes, font keys, drop-shadow effects) are in `Themes/MozaTheme.xaml`; SVG icon path data in `Themes/MozaIcons.xaml`.

Cross-collaborator state access uses internal back-references (`_sender._field` via `internal` visibility) rather than partial-class splits — accept the field-plumbing cost in exchange for explicit boundaries. The orchestrators construct collaborators in their constructors and delegate via 1-line forwarders.

### Component Layers

**Logging & Diagnostics** (`Diagnostics/`) — Shared infrastructure for plugin logging and the Diagnostics-tab export bundle:
- `MozaLog` — Static wrapper around `SimHub.Logging.Current` (`Info`/`Debug`/`Warn`/`Error`). Every plugin log call goes through this class instead of `SimHub.Logging.Current` directly so that, in addition to forwarding to SimHub's log4net pipeline, every `[Moza]` line is appended to a 5 000-entry in-process ring buffer with a millisecond local-time stamp and level. `Snapshot()` returns the buffer as a single string for the Diagnostics export — sidesteps SimHub's flush cadence and per-version log-path / file-extension differences (`Logs/SimHub.<n>.txt` on current builds). Lives in `namespace MozaPlugin` so all source files reach it without a using statement.
- `SerialTrafficCapture` — Singleton ring buffer of timestamped serial frames in both directions, used by the Diagnostics tab's capture toggle. `RecordTx(label, frame)` and `RecordRx(label, frame)` are called from `MozaSerialConnection.WriteFrame` (after a successful port write) and `ReadLoop` (just before `MessageReceived?.Invoke`) respectively. Each `MozaSerialConnection` carries a `CaptureLabel` property (default "wheelbase"; AB9 manager flips its connection's label to "ab9") so a single export can distinguish frames from each pipe. Capacity is 200 000 entries — about 30–60 minutes of dense telemetry — with oldest-drop ring trimming so a long capture never exhausts memory. `Start()` clears the buffer and arms the per-frame fast path; `Stop()` returns the snapshot and disables the gate; `Format(entries)` renders to one-line-per-frame text with local timestamps and uppercase hex bytes. **Wire-trace file sink** — `StartFileSink(path)` writes a bridge-format JSONL (`{t, dir, hex, len}` per frame, compatible with `tools/moza_trace.py` consumers) to `SimHub/Logs/moza-wire-<timestamp>.jsonl` for the entire plugin session. Toggled via `MozaPluginSettings.EnableWireTraceFileSink` — code-only `[JsonIgnore]` default (currently `true` while catalog-only mode is being verified; flip back to `false` once that work is signed off). One fresh file per plugin Init.

**Plugin Entry Point** (`MozaPlugin.cs`) — Implements SimHub's `IPlugin`, `IDataPlugin`, `IWPFSettingsV2`. Manages lifecycle (Init/DataUpdate/End), connection state, auto-reconnect (5s timer), setting polling (2s timer), `TelemetrySender` for dashboard data streaming, and `DashboardProfileStore` for profile management. `CheckGearshiftEvent` runs every `DataUpdate` and fires `base-gearshift-event` (cmd 0x76 on grp 0x2D, fire-and-forget — wheel firmware uses the persisted `base-gearshift-vibration` intensity, payload is always `76 00 01`) when SimHub's reported `Gear` value changes, gated by intensity > 0 plus a user-tunable debounce window so paddle ratcheting coalesces. Transitions *into* neutral ("N" or "0") are suppressed by default — an H-pattern shift produces two transitions ("1"→"N"→"2") and the engagement (N→2) is the one the user feels; the suppression can be flipped via the "Vibrate on Neutral" plugin setting (`MozaPluginSettings.GearshiftVibrateOnNeutral`/`GearshiftDebounceMs`). **AB9 host-rendered engine vibration (multi-stream worker, 2026-05-15):** `DataUpdate` stashes `data.NewData.Rpms` into `_latestRpmBits` (`Interlocked.Exchange` of a long via `BitConverter.DoubleToInt64Bits` — `double` can't be `volatile` in C#) and `data.GameRunning` into `volatile bool _latestGameRunning`. A dedicated `MozaAb9EngineVib` background thread (`Ab9EngineVibrationLoop`) ticks every ~11 ms (target 91 Hz, Stopwatch-deadline scheduled — `System.Timers.Timer` would top out at ~64 Hz on Windows's default 15.6 ms clock granularity). Each tick drives the full PitHouse-replicating sub-stream set decoded from the 2026-05-13 capture (`sim/logs/ab9-game-20260513.jsonl`, 942,634 frames; analysis pipeline: `tools/ab9-decode-session`):

  1. **`0x0A 0x05` engine-vibration refresh — every tick.** Computes `period_ticks = K / (rpm × freq_hz)` where K ≈ 3.95 × 10¹¹ and `freq_hz` is the literal user slider value 0..300, clamps to `[0x64, 0xFFFFFF]`, and submits via `MozaAb9DeviceManager.SendEngineVibrationStream`. Slot ID is `0x1996` when `active = gameRunning && intensity > 0 && rpm > 100 && freqHz > 0`, else `0x0000` (silent keepalive).
  2. **`0x0D 0x02 + 0x0D 0x03` keepalive pair — every 12 ticks (~9 Hz flat).** Paired triggers PitHouse fires at constant rate regardless of RPM (`SendKeepalivePair`).
  3. **`0x0B 0x02 + 0x0B 0x03` engine-pulse pair — RPM-scaled.** Pulse-on / pulse-off pair sharing a 16-bit BE phase counter (`_ab9PulsePhase`) duplicated within each frame at offset 6-7 = 8-9 (the device cross-checks the two copies). Pulse-on amplitude is the intensity slider linearly scaled to PitHouse's observed full-scale `0x2328`; pulse-off carries zero amplitude. Fire interval = `max(2, 62 / (rpm/800))` ticks (~1.5 Hz idle, ~13 Hz redline).
  4. **`0x0D 0x05` RPM-tracking trigger — RPM-scaled.** Fire interval = `max(2, 80 / (rpm/800))` ticks.
  5. **`0x08 0x04 + 0x08 0x06` low-rate signed pair — every ~260 ticks (~0.35 Hz).** Signed 16-bit magnitude carried as `-mag` in 0x04 + `+mag` in 0x06; engine-cycle phase signal. `_ab9LowRatePhase` accumulator advances 100/emission, saturates at ±32000 then wraps.
  6. **`0x0D 0x01` sparse trigger — every ~920 ticks (~0.10 Hz).** Newly-observed sub-cmd (3-byte frame, value `0x01`); mirrored verbatim for byte-fidelity with PitHouse.

  Sub-streams 2-6 are skipped when `active=false`; only the 0x0A 0x05 silent keepalive ticks. Rate constants live as `*BaseTicks` private consts on `MozaPlugin`. New `MozaSerialConnection.StreamKind` lanes (`Ab9EnginePulse`, `Ab9TriggerA`, `Ab9TriggerRpm`, `Ab9TriggerExtra`, `Ab9LowRate`) coalesce stale frames if a tick falls behind.

  **FFB init handshake** fires on the rising edge of `_ab9Detected` in `OnAb9MessageReceived`: `MozaAb9DeviceManager.SendFfbInitSequence` emits the exact PitHouse session-start sequence (`0e02, 0e01, 0703, 0709×2, 0701, 0704, 0701, 130000`) before any streaming starts. Idempotent per connection; reset on disconnect. Without this handshake the device's FFB stack stays uninitialised and the streaming frames have no audible/tactile effect — this was the root cause of the pre-2026-05-15 "no engine rumble" symptom.

  The loop is started in `Init()` after `_ab9Manager` construction, stopped in `End()` / `CleanupPartialInit()` with `_ab9EngineVibStop = true` + `Join(1000)` before the AB9 manager is disposed. Tick gating on `_ab9Detected && _ab9Manager.IsConnected` makes it a cheap no-op when the AB9 isn't present. `ApplyAb9ToHardware` (called from `OnAb9MessageReceived` on first detection and from `ApplyProfile` on game/profile switch) pushes the persisted `GearShiftVibrationIntensity` via `SendGearShiftVibrationIntensity` (one-shot `0x0A 01` config write — AB9 firmware fires the actual rumble pattern autonomously on each HID-detected gear engagement, no per-shift host trigger). Uses phased per-device initialization: `TryConnect()` sends only lightweight detection probes, then `DetectDevices()` applies saved settings and reads back device state as each device (base, wheel, dashboard, handbrake, pedals, hub) is confirmed present. This is the orchestrator that wires everything together. **Reload safety:** `Init()` is wrapped in try/catch; on partial-init failure `CleanupPartialInit()` mirrors `End()` so SimHub's reload-without-restart path can't leak timers, serial ports, or HID readers. `Instance = this` is published only after every resource is wired so background callbacks can't observe a half-built plugin. `End()` symmetrically detaches `_connection.MessageReceived -= OnMessageReceived` and `_subscribedProfileStore.CurrentProfileChanged -= OnProfileChanged` (the subscribed store is tracked separately so `ClearSettings()` — which builds a fresh `MozaPluginSettings` — can detach the orphaned subscription before re-subscribing on the new store). `OnMessageReceived` and `PollStatus` short-circuit on `IsShuttingDown` so in-flight callbacks delivered after `End()` started can't touch disposed state. Property delegates registered with `AttachDelegate` are null-guarded (`_data?.X ?? default`) so SimHub's property polling is immune to partial-init / mid-teardown windows.

**Serial Protocol Layer** (`Protocol/`) — Binary protocol over USB serial at 115200 baud (VID `0x346E`):
- `MozaSerialConnection` — Device auto-discovery via `MozaPortDiscovery` (Windows registry walk of `HKLM\SYSTEM\CurrentControlSet\Enum\USB\VID_346E&PID_*&MI_00\<instance>\Device Parameters\PortName`, the standard `usbser.sys` layout), background read/write threads, frame assembly, full 0x7E byte stuffing (read + write — every 0x7E byte after the start+len header is doubled on the wire; without this, payload 0x7Es in zlib blobs / seq counters desync the read stream after ~15s and trigger the wheel-alive watchdog). The registry path is the always-on primary discovery: it returns full `(PortName, VID, PID, FriendlyName, Category)` for every MOZA composite (the `Category` member of `PortInfo` is derived from `MozaUsbIds.Categorize(pid)`), cross-references against `SerialPort.GetPortNames()` to drop ghost entries left behind by previous USB-port attachments, and routes each port to the right connection class without writing any serial bytes. **PID routing (2026-05-16)** is driven by the inventory in `Protocol/MozaUsbIds.cs` — the single source of truth, mirrored in [`docs/protocol/devices/usb-ids.md`](protocol/devices/usb-ids.md). The wheelbase connection accepts `MozaDeviceCategory.Wheelbase` PIDs (R16/R21 `0x0000`, R9 `0x0002`, R5 `0x0004`, R3 `0x0005`, R12/R12v2 `0x0006`), `MozaDeviceCategory.Hub` (`0x0020`), `MozaDeviceCategory.Dashboard` (CM2 Racing Dash `0x0025`, so the standalone-USB dashboard pipeline reuses the same connection without a wheelbase present), plus any PID not present in the inventory (`!IsKnownMozaPid`) as a future-hardware fallback. The AB9 connection accepts only `MozaDeviceCategory.Ab9` (`0x1000`) plus the same unknown set. Known non-target categories — `Pedals` (`0x0001`/`0x0003`), `Shifter` (`0x001E`), `Handbrake` (`0x001F`) — are silently dropped from both connections so the wheelbase pipe stops emitting base/hub probe frames at devices that ignore them. (`Hub` `0x0020` is NOT dropped — the wheelbase connection admits it for the hub-only case, and the dedicated `MozaHubDeviceManager` admits only it.) Before this change the wheelbase filter was `pid => !IsAb9Pid(pid)`, which let every non-AB9 Moza composite through and produced cross-device probe noise whenever pedals/handbrake/shifter/hub were attached to the same bus. `MozaPortDiscovery` logs a single Info-level line per process the first time it sees an unknown PID (`[Moza] Unknown Moza PID 0xXXXX on COMn — not in usb-ids inventory…`) so unmapped hardware surfaces in support bundles; the `SummarizePorts` line accompanying every cache refresh now includes the friendly name (`COM5:0x0006(R12 / R12v2)`). The legacy serial-probe path is preserved as an automatic fallback that runs whenever the registry didn't already hand out a port matching the connection's PID filter. **Registry-aware probe loop (2026-05-16):** `FindMozaPort` builds a `port → PortInfo` lookup from the registry walk and consults it inside both the `BaseAndHub` two-pass loop and the `Ab9` base-disambig + identity loop. Three outcomes per port: (a) unclassified → probe normally; (b) classified with a matching PID → claim the port immediately with no probe writes (uses the registry-reported PID); (c) classified with a mismatching MOZA PID → skip with a debug log and zero serial bytes. Aggregate gate: when every COM port is registry-classified with a non-matching PID (e.g. Windows wheelbase-only user, no AB9 on the bus), the probe is short-circuited entirely so no frames go on the wire — this preserves the pre-2026-05-16 "no AB9 probe storm" behaviour. When at least one unclassified COM port exists alongside classified ones, the probe runs only against the unclassified remainder, which unblocks discovery of MOZA hardware on bus configurations where some devices register and others don't. The most concrete leak this closed: the AB9 manager's base-disambig probe was previously sent at every COM port whenever the registry was empty, and could in principle reach a port whose PID the registry already classified as a wheelbase — `RegistrySaysSkip` now suppresses that probe before it ever opens the port. The fallback can be hard-disabled via `MozaPluginSettings.DisableSerialProbeFallback` (default off) for users who want to guarantee no Moza writes ever reach a non-Moza serial device. Probe target is scoped via `MozaProbeTarget` (`BaseAndHub` for the wheelbase connection, `Ab9` for the AB9 shifter, `HubOnly` for the dedicated Universal Hub connection — see **Universal Hub Device Management** below). `BaseAndHub` runs a two-pass probe: bases first (group 0x2B → expect response group 0xAB), then hubs (group 0x64 dev 0x12 → expect 0xE4) so hub-only setups are discovered; its registry path prefers a `Wheelbase`-category port so a base stays the telemetry-driving primary when a hub is also present. `Ab9` runs the AB9 identity probe (group 0x09 dev 0x12 → expect 0x89) with a base-disambiguation pre-probe — needed because `DeviceAb9` and `DeviceHub` share id 0x12. `HubOnly` is a single hub-probe pass (no base probe, so it can never claim a wheelbase port) used by the dedicated `MozaHubDeviceManager` when a base + hub coexist; under registry discovery (the norm) it's dormant since the hub PID is always classified. `FindMozaPort`'s registry path also skips ports already in `_activePorts`, so a sibling connection can't re-open the port the primary already holds. Pre-2026-05-11 the WMI path was loaded via `Assembly.Load("System.Management")` reflection and failed silently in SimHub's plugin AppDomain, dropping every install to the probe path; "all AB9 users had to disable AB9 detection" because the probe writes disturbed non-Moza serial peripherals (Arduino, OBD2, GPS, 3D printers). The registry-based discovery eliminated both problems — see `Protocol/MozaPortDiscovery.cs`. `HubProbeSucceeded` (used by `TelemetrySender` to gate the hub-only 5-slot enumeration burst) is set from `MozaPlugin.OnMessageReceived` on the first 0xE4 (hub) reply rather than from the probe path, since registry discovery doesn't probe the hub. **Classified open-failure surface** (`Protocol/ConnectionFailure.cs`): `TryOpen`'s catch block routes `UnauthorizedAccessException`, `IOException` with access-denied substrings (`"access is denied"`, `"already in use"`, `"sharing violation"`, `"resource busy"`) or HResult `0x80070005`, `FileNotFoundException`, and `IOException` with port-vanished substrings (`"does not exist"`, `"no such"`) into a `ConnectionFailureKind` enum (`None` / `AccessDenied` / `PortVanished` / `OpenFailedOther` / `IoFailureAfterOpen`). The classification is stored on the connection (`LastFailure`, `ConsecutiveOpenFailures`, `LastSuccessfulOpenUtc` — read under `_failureLock` so the UI sees a consistent snapshot) and consumed by the UI hint-builder to distinguish "PitHouse holds the port" from generic disconnect. `ResetFailureState()` clears the surface on successful Open and from `MozaPlugin.SetConnectionEnabled(false)` so deliberate disable doesn't leave a stale "port in use" banner. The existing `MozaLog.Error` line ("Failed to connect to {portName}: {ex.Message}") is preserved verbatim across all catch arms
- `MozaCommand` — Message builder with big-endian int encoding
- `MozaCommandDatabase` — 200+ pre-built command definitions, including PitHouse-style extended wheel identity probes (`wheel-presence` group 0x09, `wheel-device-presence` 0x02, `wheel-device-type` 0x04, `wheel-capabilities` 0x05, `wheel-mcu-uid` 0x06 (12-byte STM32 UID), `wheel-hw-sub` 0x08 cmd=02, `wheel-identity-11` 0x11 cmd=04) fired by `MozaDeviceManager.SendPithouseIdentityProbe()` at wheel detect for full 12-frame PitHouse parity, and the wheel-base ambient LED command block (`base-ambient-*` on group 0x20 write / 0x22 read against dev 0x12 / device type `"main"`) covering live telemetry (per-strip color chunks `0x1A` and bitmask `0x1B`), persistent settings (indicator state, standby mode + per-mode interval, brightness, per-LED static colors, sleep timing, startup/shutdown color), and the 36-entry `base-ambient-led-color-strip{0,1}-mode{1,2}-led{0..8}` matrix populated via nested `for` loops
- `MozaResponseParser` — Decodes responses using bit-7 toggle, nibble swap, and wildcard matching. `Parse(data, busHint = null)` — when `busHint` is supplied (e.g. `"ab9"` from `OnAb9MessageReceived`), the auto-derived device hint is overridden so the parser only considers commands matching that bus. Needed because the AB9 main and wheelbase main both use dev id `0x12`; without the explicit hint, AB9 responses get auto-tagged as `"base"` and every `ab9-*` command is filtered out — which broke detection pre-2026-05-15. Unwraps Display sub-device identity responses (0xC3 group + inner 0x8X response byte) into `display-*` command names (`display-model-name`, `display-hw-version`, `display-sw-version`, `display-serial`, `display-mcu-uid`, `display-device-type`, `display-capabilities`, `display-presence`, `display-device-presence`, `display-identity-11`). Silent-drops SerialStream control frames (0xC3 + `7C/FC 00` session-open/data wrappers + 0x7C 0x23 dashboard-activate advertisements) so they don't spam unmatched-log
- `MozaProtocol` — Constants (start byte `0x7E`, device IDs, checksum formula: `(13 + sum) % 256`). Two checksum helpers: `CalculateChecksum()` (raw sum, for legacy/test use) and **`CalculateWireChecksum()`** (raw sum + `count(0x7E in body positions 2..len-1) × 0x7E`). Production code uses `CalculateWireChecksum` on both send and verify paths so frames with 0x7E in body validate correctly — before this fix (pre-2026-04-22), ~20% of zlib-bearing session chunks (configJson state, dashboard uploads) failed verify and were silently dropped. The static probe builder (`MozaSerialConnection.BuildProbe`) also uses the wire variant; current probe payloads contain no 0x7E so the values coincide, but the wire variant is forward-correct if a future probe template needs a 0x7E byte. Also exposes `WheelEchoPrefixes` + `IsWheelEcho(data)`: the set of `(group, device, payload-prefix)` tuples the wheel echoes back on LED/brightness/page-config writes. `MozaPlugin.OnMessageReceived` short-circuits recognized echoes — marks wheel alive + skips unmatched-log noise. Mirrors `sim/wheel_sim.py:_WHEEL_ECHO_PREFIXES`
- `MozaHidReader` — Reads physical input positions (steering, throttle, brake, clutch, paddles, handbrake) and button states from Moza HID devices using HidSharp. Enumerates devices by **VID 0x346E** (`MozaPortDiscovery.MozaVid`) and classifies each by PID through `MozaUsbIds.Categorize` — `Wheelbase` / `Pedals` / `Handbrake` / `Hub` route through the standard usage→field mapping; `MBooster` routes through the multi-device registry (see below); `Ab9` / `Shifter` are skipped (their input surface is serial-only); unknown Moza PIDs are admitted optimistically so future hardware works without a code change. Opens per-device receiver threads and writes normalized values to `MozaData`. Tracks axis usages (GenericDesktop page) and button usages (Button page 0x0009). Handbrake button events are routed to a dedicated `HandbrakeButtonPressed` field — detected either by PID (`MozaUsbIds.IsHandbrakePid(device.ProductID)` for the standalone HBP USB device) or by button index 121 (base-attached handbrake). Independent of the serial protocol — provides live axis and button data for UI display at 30 Hz, **and for the `Moza.*` input properties** (`HidConnected`, `SteeringAngle`, `SteeringPosition`, `Throttle`, `Brake`, `Clutch`, `Handbrake`, `LeftPaddle`, `RightPaddle`, `CombinedPaddle`) exposed via `AttachDelegate` in `MozaPlugin.RegisterProperties` — so dashboards can read physical input with no game running (issue #59). `GetCurrentAngleDegrees(maxAngleDeg)` yields the signed angle used by both `Moza.SteeringAngle` and the `@internal/SteeringWheelAngle` telemetry channel; `GetSteeringPositionPercent()` yields the max-angle-independent 0–100 position behind `Moza.SteeringPosition` (returns -1 when the HID range is unknown). **Pre-2026-05-23**, this enumeration used a `GetFriendlyName()` regex (`"gudsen (moza )?r[0-9]{1,2} ..."`, `"hbp handbrake"`, `"gudsen universal hub"`, …); the names match the kernel HID name on Linux but the SetupAPI device description on Windows is the generic `"HID-compliant game controller"`, so no pattern matched and the steering / pedal / handbrake bars stayed blank on Windows. VID-based filtering (already in use for mBooster's PID 0x0008) replaces the regex entirely

**Device Management** (`MozaDeviceManager.cs`) — High-level read/write API. Handles wheel device ID cycling (IDs 23→21→19) since ES wheels respond on different IDs. `ResetWheelDetection()` must be called on disconnect so detection probes are re-sent on reconnect. `ReadSettingsPaced()` runs a batch of reads on a background task with an extra ~10ms gap between enqueues — used for large startup bursts (30+ wheel setting reads) that the wheel would otherwise drop, without throttling steady-state telemetry. **`SendPresenceProbe(byte deviceId)`** emits a PitHouse-style empty probe (`7e 00 00 <deviceId> <chk>`, 5 bytes) — `MozaPlugin.PollStatus` uses this for dash / handbrake / pedals presence detection instead of the prior cmd-specific reads (`dash-rpm-indicator-mode` / `handbrake-direction` / `pedals-throttle-dir`). The empty probe is NOT tracked by `PendingResponseTracker`, so an absent device doesn't burn the 3-attempt retry budget (200/400/800 ms backoff) every 5 s tick — that retry amplification was the dominant idle wire noise on single-base setups (3 × 4 absent-device probes = 12 wasted frames per tick). Hub stays on the cmd-specific `hub-port1-power` path because hub shares dev id 0x12 with the wheelbase main controller — an empty probe to 0x12 always ACKs from the base. Empty-probe ACKs come back as 2-byte frames `{0x80, swap(dev_id)}` and are intercepted by `MozaPlugin.OnMessageReceived` ahead of the cmd-DB parser; they route through `OnPresenceProbeAck` → `DeviceProber.MarkDashDetected / MarkHandbrakeDetected / MarkPedalsDetected` (the same idempotent helpers the data-response parser calls when one of the cmd-specific reads happens to fire from another code path).

**AB9 Device Management** (`Devices/MozaAb9DeviceManager.cs`) — Owns the dedicated `MozaSerialConnection` for the AB9 active shifter (VID `0x346E`, PID `0x1000`, dev id `0x12`, probe target `MozaProbeTarget.Ab9`).

  **Identity probe** (`SendIdentityProbe`) emits PitHouse's full 10-group cascade (`0x09, 0x04, 0x06, 0x02, 0x05, 0x07, 0x0F, 0x11, 0x08, 0x10`) — extended from the original 5-group set 2026-05-15 to match what PitHouse actually sends so a real AB9 responds with the same probe coverage.

  **Stored-setting read group is `0x1E`, not `0x1F`** — distinct from the write group. Wire shape: `7E 01 1E 12 <cmdId> <chk>` with a single-byte cmd-id payload (writes use `7E 03 1F 12 <cmdId> 00 <val> <chk>` with the cmd+0x00+val triplet). The plugin built `SendAb9Read(byte cmdId)` directly rather than going through `MozaCommand.BuildReadMessage` because the read shape doesn't fit the single-cmd-id-array convention. Read responses arrive on group `0x9E`; `MozaCommandDatabase` registers parallel `ab9-*-read` entries on group `0x1E` with single-byte CommandId and 2-byte BE payload so the parser decodes the response correctly. The pre-2026-05-15 read path sent reads on group 0x1F with the wrong payload shape, so the real AB9 never responded and `_ab9Detected` stayed false (the "Probing AB9…" panel symptom).

  **Mechanical-mode set** (`SendMode` — cmd `0xD3 00` on grp 0x1F) and **feel slider writes** (`SendSlider` — mech-resistance/spring/damping/friction/max-torque, all single-byte 0..100 on grp 0x1F) route through `MozaCommandDatabase` and `MozaCommand.BuildWriteMessage`. The write entries have `ReadGroup = 0xFF` so `BuildReadMessage` returns null and callers are forced through `SendAb9Read`.

  **FFB session-init handshake** (`SendFfbInitSequence`) — fire-and-forget byte-for-byte replica of PitHouse's session-start cascade: `0e02, 0e01, 0703, 0709×2, 0701, 0704, 0701, 130000`. Fires once on the rising edge of `_ab9Detected` (gated by `_ffbInitSent`); resets on disconnect. The 1-byte slot indices returned in `0x07` ACKs are device-side internal effect handles and are intentionally ignored — the streaming frames use 16-bit slot IDs that are runtime Windows DirectInput effect handles (the device firmware routes through its own allocation table, not the streaming-frame slot IDs).

  **Engine-vibration streaming** — `SendEngineVibrationStream(bool active, uint periodTicks)` builds the 24-byte `0x20/0x0A 05` frame inline (16-bit BE slot ID + 7 zero bytes + 24-bit BE period + `0x04` type tag + 4 zero bytes) and submits via `_connection.SendStream(StreamKind.Ab9EngineVibration, frame)`. Latest-wins coalescing on this lane means a slow 91 Hz tick discards the previous frame instead of piling up.

  **Engine-pulse pair** — `SendEnginePulsePair(ushort phase, int intensity0to100)` builds the paired 22-byte `0x0B 0x02` (on) + `0x0B 0x03` (off) frames. Both carry a 16-bit BE phase counter duplicated at payload offsets 6-7 = 8-9 (device cross-check), fixed `0xFFFA` envelope param, `0x04` type tag, and 16-bit BE amplitude (intensity-derived for ON: linearly scaled from 0..100 to `0x0000..0x2328`; zero for OFF). On frame uses `StreamKind.Ab9EnginePulse` (latest-wins); OFF frame uses one-shot FIFO so it always follows the on frame onto the wire.

  **Triggers** — `SendTrigger(Ab9Trigger trigger)` builds 3-byte `0x0D <subLo> 0x01` frames. Each lane routed separately: `Ab9TriggerA` for keepalive A/B (`0x02`/`0x03`), `Ab9TriggerRpm` for RPM-tracker (`0x05`), `Ab9TriggerExtra` for sparse (`0x01`). `SendKeepalivePair` fires `0x0D 0x02` via the stream lane + `0x0D 0x03` via the one-shot FIFO so the pair always lands together.

  **Low-rate signed pair** — `SendLowRatePair(short magnitude)` builds the paired 11-byte `0x08 0x04` (`-magnitude`) + `0x08 0x06` (`+magnitude`) frames, encoding a bipolar engine-cycle phase signal. Fixed `00 64 04` envelope/tag bytes verbatim from PitHouse. Stream-lane + one-shot FIFO pairing same as the keepalive pair.

  **Gear-shift vibration intensity** — `SendGearShiftVibrationIntensity(int 0..100)` builds the 24-byte `0x20/0x0A 01` frame (16-bit BE intensity at payload offset 0-1, scaled linearly from 0..100 to 0..`0x332C` — verified empirically: 30% = `0x0F5A`, 100% = `0x332C`) and submits one-shot via `_connection.Send`. AB9 firmware fires the actual rumble pattern autonomously on each HID-detected gear engagement, no per-shift host trigger.

  See `docs/protocol/devices/ab9-shifter.md` for the live-capture decode of every frame shape and the full Phase 1 resolution of previously-open schema questions. Frame layout constants live on the class (`SilentSlotId = 0x0000`, `DefaultEngineVibSlotId = 0x1996`, `MinPeriodTicks = 0x64`, `MaxPeriodTicks = 0xFFFFFF`) so `MozaPlugin.ProcessEngineVibTick` can clamp before calling.

**Universal Hub Device Management** (`Devices/MozaHubDeviceManager.cs`, added 2026-05-29) — Owns a **dedicated** `MozaSerialConnection` (PID `0x0020`, `CaptureLabel = "hub"`, probe target `MozaProbeTarget.HubOnly`, probe fallback force-disabled = registry-only) for a Universal Hub on its **own COM port**, used when a wheelbase is **also** present. Motivating case: a base model with no pedal port + a Universal Hub providing the pedal connection — the hub is a separate USB CDC composite, so its peripherals (pedals `0x19`, handbrake `0x1B`, hub port-power `0x64`) never enumerated when discovery latched onto the single first matching port.

  **Wheel-location rule.** When a wheelbase is present it stays the telemetry-driving primary connection: `FindMozaPort`'s registry walk prefers a `Wheelbase`-category port for the `BaseAndHub` connection (falling back to the hub only when no wheelbase exists — the hub-only case, where the primary binds to the hub and runs the full wheel/session/telemetry/dashboard pipeline exactly as before). The dedicated hub connection then claims the leftover hub port and enumerates its peripherals **in parallel**. The `_activePorts` guard — now also consulted in `FindMozaPort`'s registry path, not just the probe loops — stops the dedicated connection from re-opening a hub the primary already holds. A wheel attached to the hub **while a base is present** is not driven (telemetry stays on the base); multiple simultaneous wheels are out of scope.

  **Per-device ownership routing.** Pedals/handbrake share identical device IDs + protocol commands regardless of which CDC pipe carries them (`docs/protocol/transport/usb-topology.md`), so "routing" is purely *which `MozaDeviceManager` to send through*. A **second `DeviceProber`** (`_hubDeviceProber`, `drivesTelemetry: false` so it never touches the primary `TelemetrySender.DetectedDeviceMask`) is bound to the hub connection + hub device manager, sharing the same `MozaData` + `DeviceDetectionState` as the primary prober. `DeviceProber.MarkPedalsDetected`/`MarkHandbrakeDetected` stamp `DeviceDetectionState.PedalsOwner`/`HandbrakeOwner` (the prober's own manager) **before** flipping the shared `*Detected` flag — first responder across the base + hub pipes wins and becomes owner. `HardwareApplier` resolves `PedalsOwner ?? _deviceManager` (and handbrake) for both the `Apply*ToHardware` batch and the live `WriteIf*Detected` calibration path, so curve/calibration writes land on the pipe that actually owns the device. Null owner → falls back to the primary manager → zero behavior change for base-attached peripherals.

  **Per-pipe retransmit.** `MozaDeviceManager` takes an injectable `PendingResponseTracker` (defaults to the global `MozaPlugin.PendingResponses` singleton for the primary). The hub manager owns its **own** tracker, ticked by the 250 ms retry timer against `_hubManager.Connection.Send` — without this, hub reads tracked in the global singleton would be retransmitted on the *base* port. `MozaDeviceManager.PendingResponses` exposes the effective tracker so the inbound handler calls `NoteResponse` on the correct one.

  **Inbound + poll.** `MozaPlugin.OnHubMessageReceived` parses the hub pipe but forwards **only** `pedals-*` / `handbrake-*` / `hub-*` command responses (+ presence-probe `0x80` ACKs for `DevicePedals`/`DeviceHandbrake`) into `_hubDeviceProber`, dropping wheel/base/dash/session frames so the wheel/telemetry pipeline stays exclusively on the primary. `PollHubPeripherals` (called from `PollStatus` *before* its primary-connection guard, and once on connect) sends pedals/handbrake presence probes + `hub-port1-power` on the hub pipe — gated on the shared `*Detected` flags so probes fire on **both** pipes until detected. `TryConnectHub` runs from the 5 s reconnect timer (same Wine registry guard as the AB9/dashboard managers) and persists `MozaPluginSettings.LastHubPort`. `OnHubDisconnected` clears `HubDetected` and re-routes any peripherals owned by the hub pipe back to "no opinion". Once the dedicated connection sets the shared `HubDetected`, the primary's existing `hub-port1-power` probe (which logs a benign "Unexpected cmd: 100" on a base main that has no hub) stops.

**mBooster Pedals Multi-Device Registry** (`Devices/MozaMBoosterRegistry.cs`, `Devices/MBoosterDeviceController.cs`, `Devices/MBoosterEffectWorker.cs`, `Devices/MBoosterEffectSynthesizer.cs`, `Devices/MBoosterTypes.cs`, `Protocol/MozaMBoosterProtocol.cs`) — Multi-device owner for Moza mBooster Pedals (VID `0x346E`, PID `0x0008`). The mBooster is a single-axis analog pedal with a built-in vibration motor — the user picks whether the pedal serves as throttle, brake, or clutch in the plugin UI, and the motor is driven by host-rendered effects (ABS, Lockup, Threshold, Engine) per the user-supplied protocol note in [`docs/MozamBooster — Protocol Note.md`](MozamBooster%20—%20Protocol%20Note.md). Users may attach **more than one mBooster simultaneously** (one each for throttle / brake / clutch is the canonical layout); each unit gets its own COM port + connection + 50 Hz effect worker.

  **Discovery** is registry-driven by `MozaMBoosterRegistry.Refresh()`, called from the 5 s reconnect timer alongside `TryConnectAb9()`. `MozaPortDiscovery.EnumerateMatching(MozaUsbIds.IsMBoosterPid)` returns the set of currently-attached mBooster ports; the registry spawns one `MBoosterDeviceController` per discovered port and disposes controllers whose ports disappear. `MozaProbeTarget.MBooster` is intentionally a no-op — mBooster device id `0x12` collides with wheelbase Main + AB9 Main, so a serial probe at every COM port would be high-risk for non-mBooster peripherals. Linux users without registry entries get no mBooster discovery by design; this only matters on Wine/Proton and is acceptable since users without mBooster hardware see no effect.

  **Stable per-device identity** uses the USB device instance ID surfaced by `MozaPortDiscovery.PortInfo.InstanceId` (e.g. `a&399b951f&0&0000`), keyed into `MozaProfile.MBoosterSettings : Dictionary<string, MBoosterDeviceSettings>` so per-device settings (role + per-effect enables + calibration) survive replug. The HID-side identity is extracted from `HidDevice.DevicePath`'s parent-USB instance segment via `MozaHidReader.ExtractUsbParentInstance` — both stacks expose the same USB parent so the registry can pair them, though this is currently unverified on real hardware (the plan calls out a port-order fallback if identities diverge on a particular unit).

  **Position routing**: `MozaHidReader` matches mBooster by VID 0x346E + PID 0x0008 (`MozaHidClass.MBooster` path in `FindMozaDevices`), captures every GenericDesktop axis the device exposes (usages `0x30..0x37`), and on each report change raises `MBoosterAxisChanged(identity, pos01)`. `MozaMBoosterRegistry.OnHidAxisUpdate` looks up the controller, stores the position, and runs the merge step: walks `_order` and writes each device's position into `MozaData.ThrottlePosition` / `BrakePosition` / `ClutchPosition` based on its `Role` setting (`MBoosterRole.{Throttle,Brake,Clutch}`). First-wins on role collision — if two devices both pick Brake, the first in enumeration order writes the value and a warning logs once per `(role,identity)` combo per session. `MBoosterRole.Disabled` skips routing entirely. Wheelbase-attached pedals (the standard `MozaHidReader` path — `MozaHidClass.Standard`, VID 0x346E + wheelbase PID) are unaffected; an mBooster configured as brake just overrides what the wheelbase-pedal would have set for that slot.

  **Effect synthesis** follows the protocol note § 4 verbatim — `MBoosterEffectSynthesizer.cs` is pure functions: `SynthesizeAbs(intensity, phase) = (0.9 + 0.1 * sin(phase)) * intensity` (oscillating pulse 80–100%), `SynthesizeLockup(intensity, elapsed) = clamp(elapsed/0.5, 0, 1) * intensity` (linear ramp), `SynthesizeThreshold(intensity, elapsed)` = 5 Hz burst envelope (20 ms full + 120 ms 80% + 60 ms gap), `SynthesizeEngine(intensity, phase) = (0.5 + 0.5 * sin(phase)) * intensity` (biased sine). **Engine intensity is clamped to 10 %** at the apply layer (`EngineIntensityCap = 0.10` in `MBoosterEffectWorker`) — the protocol note warns engine runs continuously and would dominate the other effects without the cap.

  **Wire format** — `Protocol/MozaMBoosterProtocol.cs` owns the motor-write opcode (`0xb1`) and keepalive shape. Motor frames are 14 bytes pre-stuffing: `7e 09 24 12 b1 [effect:u8] [enable:u8] 00 [param1:u8] [freq:u16 BE] [amp:u16 BE] [chk]` where `param1 = clamp(round(paramK / freq_hz), 1, 255)` and the per-effect ParamK constants are `Abs=2000, Lockup=2640, Threshold=3080, Engine=1000` (verified against the protocol note's hardware captures). Frequency encoding is `u16 = round(hz * 65536 / 200)` saturating, amplitude is `u16 = clamp(round(amp01 * 65535), 0, 0xFFFF)`. Keepalive is the literal `7e 00 00 12 9d` — `MozaMBoosterProtocol.BuildKeepalive()` — sent every ~500 ms via the one-shot FIFO regardless of effect state; without it the motor drops connection state per protocol note § 3. Disable frames (one per effect ID with `enable=0` and all params zero) fire at every effect-deactivation edge AND for all four effect IDs in `MBoosterDeviceController.Dispose()` so the motor doesn't latch the last-active waveform after the port closes.

  **Stream lane**: all motor frames go through `StreamKind.MBoosterEffect` on each controller's connection — a single coalesced lane per device (latest-wins) is sufficient because the worker emits at most one motor frame per ~20 ms tick (50 Hz per protocol note § 4) and the four effects rotate through `ProcessEffect` sequentially within each tick. Keepalive frames go via the one-shot FIFO so they aren't coalesced against motor frames. Frame layout constants live on `MozaMBoosterProtocol` (`GroupMotorWrite = 0x24`, `DeviceMotor = 0x12`, `CmdMotorWrite = 0xb1`, `MotorPayloadLen = 0x09`).

  **Telemetry → effect parameters** is per protocol note § 4 pseudocode, computed inside `MBoosterEffectWorker.Tick`. `MozaPlugin.DataUpdate` builds an `MBoosterTelemetrySnapshot` (game-running flag, RPM, brake 0..1, ABS-active bool, vehicle speed m/s, average wheel speed m/s) per SimHub tick and fans it out to every controller via `MozaMBoosterRegistry.OnDataUpdate`. The worker then maps:
  - **ABS**: `freq = 18 + abs_active * 12` Hz, `intensity = abs * absScale`, fires when `abs > 0.1`. SimHub's `ABSActive` is converted defensively via `Convert.ToInt32` because the backing type varies across SDK versions (some int, some bool).
  - **Lockup**: `freq = 40 + brake * 30` Hz, `intensity = brake * lockupScale`, fires when `brake > 0.8 && vehicle_speed > 5 && avg_wheel_speed < vehicle_speed * 0.3`. Per-wheel speeds aren't compile-time properties on this SimHub SDK's `StatusDataBase`, so `avg_wheel_speed` is fed as 0 and a fallback path triggers on `brake > 0.9 && vehicle_speed > 5` to keep the effect meaningful without per-wheel data.
  - **Threshold**: `freq = 60 + brake * 30` Hz, `intensity = brake * thresholdScale`, hysteresis-latched on rising-edge `brake > 0.6`, released on `brake < 0.3`.
  - **Engine**: `freq = clamp(rpm / 20000 * 200, 10, 200)` Hz (~10–100 Hz typical), `intensity = engineScale * 0.1` (constant amplitude, capped at 10 % per the doc).

  Each effect goes through `ProcessEffect` which advances the per-effect phase + elapsed, calls the synthesizer, encodes via `MozaMBoosterProtocol.EncodeFreq` / `EncodeAmp` / `ComputeParam1`, and submits the motor frame via the stream lane. On the deactivation edge (intensity drops to 0), one disable frame goes out via the one-shot FIFO and the worker stops emitting for that effect until the next activation.

  **Settings storage** is on `MozaProfile.MBoosterSettings : Dictionary<string, MBoosterDeviceSettings>` keyed by device identity. `MBoosterDeviceSettings` carries `Role` (enum), four `MBoosterEffectSettings` (ABS / Lockup / Threshold / Engine — each with `Enabled` + `IntensityPct`), plus calibration fields (`Direction`, `Min`, `Max`, `CurveY[5]` — written via `MBoosterDeviceController.SendIntWrite` / `SendFloatWrite` against `mbooster-*` commands in `MozaCommandDatabase`). The calibration surface (group 35 read / group 36 write, mirroring the wheelbase-pedal `pedals-*` namespace) is **experimental** — the protocol note marks it "likely but unverified" on mBooster firmware, and the UI's Calibration expander surfaces this with a yellow warning.

  **Response parsing**: `MBoosterDeviceController.OnConnectionMessage` parses inbound frames with `MozaResponseParser.Parse(data, busHint: "mbooster")` so dev 0x12 responses don't cross-match against `base-*` (wheelbase main) or `ab9-*` (AB9 main). Detection latches on the first valid `mbooster-*` response and fires `DetectedRisingEdge` so `MozaPlugin.OnMBoosterDeviceDetected` can apply saved calibration via `ApplyMBoosterToHardware` and kick off a `RequestCalibrationReads` burst.

  **UI tab** (`MozaPlugin.UI` panel) is hidden until `MozaMBoosterRegistry.AnyDetected == true`. **Dashboard Telemetry Streaming** (`Telemetry/`) — Streams game data to the wheel dashboard display via a multi-tier binary protocol:
- `TelemetrySender` — Timer-based sender with a multi-phase startup matching PitHouse's observed sequence. Also pushes an empty-state tile-server blob on session 0x03 after tier definition via `SendTileServerState()` (matches PitHouse — captures show wheel never pushes back on 0x03, direction is host→wheel only). Exposes `TileServerState : TileServerState?` for UI diagnostics panel binding. **All inbound `7c:00 type=01` chunks are acked with the literal received seq** (sess=0x01 mgmt, sess=0x02 telemetry FlagByte, sess=0x04..0x08 FT-candidate range, sess=0x09/0x0a configJson). Pre-2026-05-09 the mgmt/telemetry paths acked the running max `_sessionAckSeq` / `_mgmtAckSeq` instead — the wheel re-pushed older seqs forever because its retransmit-suppression keys on the literal acked seq, not a cumulative high-water mark. The per-chunk `SendSessionAck(session, (ushort)seq)` change closed this. `SendStatusPush()` (the prior ~1 Hz "harmless re-ack of running max") is now a no-op since per-chunk acks cover what's needed and the periodic re-ack was the only remaining stale-ack emission site. Inbound chunk feeds for `ChannelCatalogParser` (sess=0x01 / FlagByte) and `TileServerStateParser` (sess=0x03 / 0x0b) carry a 4-byte CRC32 LE trailer that's now **validated** before the body is fed to the parser; mismatches drop silently and increment `CatalogCrcRejects` / `TileServerCrcRejects` for diagnostics. Same feeds also dedup by per-session highest seen seq so wheel retransmits don't double/triple-append duplicate bytes (root cause of the deterministic tire-URL corruption tracked in `findings/2026-05-09-acks-dedup-and-catalog-persistence.md`). Sequence: (0) **host session opens** — first sends type=0x00 end markers across either `0x01..0x03` (game-switch reload via persistent wire — keeps wheel-managed `0x04..0x0a` intact so the configJson handshake stays bound) or the wider `0x01..0x0a` range (cold start in a fresh SimHub process — covers host-managed slots PLUS wheel-managed ones the prior process may have left half-engaged). The cold-start signal is `isFirstStartInProcess` (= `_silenceGate.LastStopUtcTicks == 0`, the same flag that drives the silence-gate skip a few lines below) and `ProbeAndOpenSessions(bool isColdStart)` carries it down to the close loop. Background: CS-Pro / KS-Pro have been observed to silently swallow fresh session-open frames when stale wheel-side session state from a prior SimHub process is still live; manually toggling the plugin "Connection enabled" off and on recovered it (because that path closes the OS serial port and forces the wheel to drop its sessions), but a SimHub restart alone did not. The cold-start wide close emulates the off/on recovery proactively. Waiting is best-effort up to 500ms for each wheel fc:00 close-ack via `TryCloseSession`; a close-ack timeout is non-fatal and falls back to today's blind-blast behavior. `CloseHostSessions()` (called from `Stop()` to clean-close 0x01..0x03 on shutdown / reload) is gated on `MozaPlugin.Instance?.ShouldDriveDashboard() != false` so screenless wheels never emit `7c 00` close frames for sessions they never opened. Then directly opens 0x01 (mgmt) and 0x02 (telem = FlagByte) with type=0x81, waiting up to 500ms for each fc:00 open-ack. Proceeds with PitHouse defaults if open-acks don't arrive — the sess=0x02 engagement watchdog (below) catches the case where the wheel later wakes up but never engages sess=0x02 on its own. The previous 48-port probe was removed — the write-queue burst it caused saturated the line long enough for the wheel-alive watchdog to reset the handshake mid-flight. Session 0x03 (aux) is opened fire-and-forget later. (1) **device-initiated session intake** — real wheel opens sessions 0x04/0x06/0x08/0x09/0x0A on its own side ~40–400 ms after host opens 0x01/0x02 on direct wheelbase, ~2.1 s when behind Universal Hub; sender acks each incoming type=0x81 with `openSeq = data[6]|data[7]<<8` echoed in an fc:00 ACK, routes subsequent 7c:00 data to per-session handlers. (2) **dashboard upload (session 0x04)** — waits up to 3 s for device open, then if the active mzdash's MD5 doesn't match any `WheelState.EnabledDashboards[].hash` (skip-if-unchanged), sends sub-msg 1 (path registration) + sub-msg 2 (file content) + type=0x00 end marker via `FileTransferBuilder`/`DashboardUploader`, with waits for device echoes between sub-messages. After the END marker the sender sleeps 500 ms so the wheel's post-upload dir-listing refresh chunks on session 0x04 arrive while the preamble handler is still subscribed. Hub-attached wheels are detected at port-probe time via `MozaSerialConnection.HubProbeSucceeded` (set true when the hub probe — group 0x64, dev 0x12, cmd 0x03 — returned a reply on the chosen COM port). When set, `SendHubSlotEnumeration()` fires PitHouse's 5-slot Form B burst (`7E 03 64 12 01 NN 00 [chk]` for NN=1..5) right after sessions open, ~300 ms before tier-def. Hub responds with `7E 03 E4 21 01 NN VV [chk]` per slot (VV = device-type code on that port; 0x00 = empty). Burst is gated to hub-only setups so direct-wheelbase doesn't waste 5 frames. (3) **configJson RPC (session 0x09)** — `ConfigJsonClient` parses the device's state blob (TitleId, enableManager dashboards, imageRefMap, etc.) as it arrives; on first complete blob the sender replies once with a `configJson()` canonical library list so the wheel's `configJsonList` picks up our profile set. (4) **channel-catalog quiet wait + tier definition + Display probe** — `WaitForChannelCatalogQuiet(200, 2000)` blocks until the wheel's pre-tier-def tag-0x04 channel burst on session 0x02 has been idle for 200 ms (max 2 s); `SendTierDefinition` intersects the active `MultiStreamProfile` with `_catalogParser.Catalog` via `FilterProfileToCatalog` (drops channels the wheel didn't advertise, drops empty tiers, falls back to unfiltered if everything is dropped), then sends the tier def. Format depends on `EraPolicy.Encoding` (resolved by `ResolveAutoPolicy` from the per-wheel-page `WheelEra` or a wheel-model guess): V2 compact = sub-msg 1 preamble + compact numeric tier defs via `TierDefinitionBuilder`; V0Url = URL subscription via `BuildV0UrlSubscription`. Then probes Display sub-device via 0x43 identity. (5) **preamble** (~1s) — acks incoming 7c:00 channel data with fc:00, buffers for channel catalog parsing, detects Display from 0x87 response; for v0 re-sends URL subscription (PitHouse double-sends). At preamble end the catalog is parsed into `_catalogParser.Catalog` (full URLs) but the `OnMessageDuringPreamble` handler **stays subscribed** so session 0x04 dir-refresh, 0x09 configJson updates, and 0x0a RPC replies keep flowing during the active phase. (6) **active** — `0x40` channel config burst (channel enables, dashboard state queries 28:00/28:01, mode 28:02), `0x41` enable signal (~30 Hz), `7d:23` telemetry per tier (30ms/500ms/2000ms), `0x2D` sequence counter, mode frames, dash/wheel keepalives (0x43 to 0x14/0x15/0x17), display config (7C:27) + dashboard activate (7C:23) interleaved per page, session acks. Era selection: `WheelEra` (Auto / Era2024 / Era2025 / Era2026, enum `MozaWheelEra` in `Telemetry/Era/MozaWheelEra.cs`) is resolved per-wheel-page from `WheelTelemetryEraByPageGuid` and applied through `EraPolicy` (`Telemetry/Era/EraPolicy.cs`), which sets the tier-def session (`TierDefSessionPolicy`: FlagByte vs MgmtPort), encoding (`TierDefEncoding`: V0Url / V2Compact / V2Type02), preamble policy, blind-retransmit, and `FileTransferWireFormat` for uploads. `Auto` resolves via `EraPolicy.GuessFromWheelModel`. (The earlier `MozaFirmwareEra` / `ProtocolVersion` / `FlagByteMode` settings were removed in favour of this era policy.) Telemetry deferred until wheel is detected (`StartTelemetryIfReady`), which dispatches `Start()` to a background thread so the serial read thread stays free to deliver fc:00 ack responses during port probing. `Start()` is serialized via a `SemaphoreSlim(1, 1)` + per-run `CancellationTokenSource`: a second `Start()` arriving while the first is mid-`StartInner` cancels the prior run's CTS (the in-progress `StartInner` bails at its next gate point) and then queues behind it on the semaphore with a 10 s wait. `StartInner(CancellationToken cancel)` checks `cancel.IsCancellationRequested || _state == TelemetryState.Idle` after each session-open phase so cancellation AND an external `Stop()` both abort cleanly. `Dispose()` also cancels the live CTS so any in-flight `Start` exits before disposed waiter slims are touched. This replaced the pre-2026-05 `Interlocked.CompareExchange` + 10 s `SpinWait.SpinUntil` + outside-in `TransitionTo(Idle)` pattern, which burned a ThreadPool worker for the wait and mixed supersession with the Stop signal. (See `TelemetryState` enum at the top of `Telemetry/TelemetrySender.cs` for the lifecycle states; replaced the prior 3-boolean soup `_enabled` / `_preambleComplete` / `_dashSwitchMuted`.) `OnTimerElapsed` is guarded by `_tickInProgress` (also `Interlocked.CompareExchange`): `System.Timers.Timer` fires on ThreadPool and queues concurrent invocations if a handler overruns its interval, which races `_tickCounter`/`_slowCounter`/`_modeCounter` and fires non-coalesced one-shot frames (heartbeat, display_cfg) 2–3× the intended rate. Dropping overlapping ticks is safe because stream-lane traffic is latest-wins coalesced and re-covered by the next tick. `Pause()` / `Resume()` toggle the timer without tearing down session state — used by the UI Test Stop button when telemetry remains enabled so the wheel goes quiet within ~1 ms instead of continuing at tier-0 rate from the still-running timer. `FlushPendingOutput()` exposes the connection's `FlushPendingWrites()` so the Stop button also drops any in-flight stream slots + OS write buffer. `Stop()` resets `FramesSent` + session state AND unsubscribes the session handler so disable→re-enable performs a full fresh startup. **`Stop()` also calls `SilenceGate.MarkStopped` at the end** — `StartInner` reads `_silenceGate.LastStopUtcTicks` on entry (BEFORE the internal `Stop()` overwrites it) and `Thread.Sleep`s `_silenceGate.RemainingStopReopenWaitMs(preStopTicks)` of host silence between the last Stop and any new prime/open-request emission (cap = `SilenceGate.StopReopenSilenceMs` = 11 s). The wheel has a ~10–14 s internal timeout on sess=0x09 dashboard-binding state during which it ignores every host emission; without this gate, rapid Stop+Start cycles (SimHub game-switch plugin reload, manual disable+re-enable, dashboard switches via `RestartForSwitch`) would land inside the timeout window and the wheel never re-engages configJson. The timestamp is static on `SilenceGate` so it survives plugin instance recycle in the same process. Cold-start skips the gate (`LastStopUtcTicks == 0` initially). See [`docs/protocol/findings/2026-05-08-wheel-sess09-timeout.md`](protocol/findings/2026-05-08-wheel-sess09-timeout.md). Dashboard switches with `EnableHotRenegotiation` (default true) do NOT go through Stop+Start — they emit kind=4 + a paced multi-emission tier-def burst on the still-open sess=0x01/0x02 (see [`protocol/tier-definition/handshake.md`](protocol/tier-definition/handshake.md) § In-game dashboard switch). UI cooldown shrinks to `HotSwitchCooldownMs` (200 ms). The legacy `RestartForSwitch()` path (`PreStopDrainMs` 300ms drain → Stop → Start with the silence gate enforcing the 11s wait) is reached only from (a) `EnableHotRenegotiation = false`, (b) game-switch / wheel hot-swap, and (c) sess=0x02 engagement-watchdog exhaustion — never a normal user dash switch. As secondary defense against dropped chunks under Wine SerialPort R/W contention, `TickRetryS09IfNotEstablished` runs once per Active tick re-emitting `SendSessionPrime(0x09) + SendConfigJsonOpenRequest(0x09)` at 1s intervals (10-round budget) until `_sessions.GetOrCreate(0x09).DeviceInitiated == true`. Parallel watchdog for sess=0x02: `TickSession02EngagementWatchdog` fires when no inbound chunk has been seen on `FlagByte` for 3s after entering Active. Re-arm sequence is `TryCloseSession(0x02) → TryOpenSession(0x02) → SendSessionInitHandshake() → ApplySubscription(force:true)` on a 3/5/7/10/15s backoff (5 rounds total), escalating to `RestartForSwitch` on exhaustion. The engagement gate (`_session02FirstInboundUtcTicks`) is sticky — once any inbound on `sess=FlagByte` arrives the watchdog stays dormant for the rest of the session. Diagnoses the "wheel never engaged sess=0x02 → dashboard layout renders but every channel sits at zero" failure mode observed in bundle `CS-Pro-1stLaunchAfterDll-20260518` (W17 wheel, 17s of post-connect silence during which the 500ms-per-open timeout gave up and there was no path back to retry). `Stop()` also drains every outstanding `SendRpcCall` waiter (sets the `ManualResetEventSlim` so blocked callers return immediately with a null reply) instead of leaving them parked on `Wait()` until their own per-call timeout fires. `Dispose()` is idempotent (`Interlocked.Exchange` gate on `_disposed`) and disposes the two TelemetrySender-owned waiter slims (`_ackReceived`, `_mgmtResponseEvent`) plus the helper coordinators (`_uploader`, `_rpc`, `_dashboardDownloader`) which each own and dispose their own wait events; subsequent `Start`/`SendRpcCall` calls bail with a warning instead of touching disposed state. Exposes `DisplayDetected`/`DisplayModelName` for UI gating, `WheelChannelCatalog`, `WheelState` (most-recent session 0x09 dashboard state), `Session04DirListingRefreshed`, and `SendRpcCall(method, arg, timeoutMs)` for session 0x0a JSON RPC round-trips (e.g. `completelyRemove`). `CanonicalDashboardList` is populated by the plugin from the built-in profile set. **Catalog-only mode** (no local mzdash folder configured): the plugin can drive dashboards entirely from the wheel-advertised channel catalog when `WheelMzdashFolderByPageGuid[pageGuid]` is empty. Flow:
- `DashboardBindingCoordinator.ApplyTelemetrySettings` finds no resolvable profile in the cache, sets `sender.Profile = null` and logs the "catalog-only mode" branch.
- `TelemetrySender.MaybeSwapProfileForCatalog` (called from `StartInner`, `ApplySubscription`, AND every steady-state tick) checks: `_profile` is null or a previously-synthesised one (`_profile.Name == CatalogProfileName` = `"WheelCatalog"`); `_catalogParser.LiveCatalog` is non-empty; `LastWheelEndMarker != 0` (wheel has committed at least one tier-def generation). If all gates pass, it calls `DashProfileStore.BuildProfileFromCatalog(LiveCatalog, "WheelCatalog")`, applies user channel overrides via `DashboardProfileStore.ApplyUserMappings` (the binding coordinator skips that branch when profile is null at apply time, so the synthesis path owns it), and assigns through the `Profile` setter (so multi-broadcast expansion + `_tiers` allocation match the mzdash path exactly). **After the assign it raises `DashboardSelectionChanged`** so the channel-mapping grid in the wheel device tab re-populates deterministically against the freshly committed profile. The original event was raised only at the moment the dashboard switch was *initiated* (from `DashboardBindingCoordinator.OnWheelInitiatedSwitch` / the UI dropdown handler) — at that point `sender.Profile` is either null or the prior dash's profile, so the UI handler populated with stale data and waited on the 500 ms `RefreshTelemetryStatus` polling signature to notice the new synth. Hash collisions in `ComputeMappingDataSignature` (24-bit hash + count masks) OR a second switch arriving inside the 500 ms window left the grid stuck on the prior dash's channels. The MaybeSwapProfileForCatalog early-return below means this event only fires when the wheel actually advanced state — not on every 30 ms timer tick.
- Re-synthesis triggers: catalog count changed since last synthesis OR `LastWheelEndMarker` advanced (the wheel committed a new tier-def generation — typically a dashboard switch). Detection state is `_catalogCountAtSynthesis` + `_catalogEndMarkerAtSynthesis`. Re-synthesis is gated to be a no-op when neither has moved, so the tick-path call is two integer compares in the steady state.
- Wheel-initiated switches in catalog-only mode: `DashboardBindingCoordinator.OnWheelInitiatedSwitch` cannot resolve a profile by name (no mzdash). It clears the synthesised profile (`sender.Profile = null` — but only when the current profile is one we synthesised, never an mzdash one), then raises `DashboardSelectionChanged` so the UI dropdown updates. The next tick rebuilds the synthesised profile from the fresh post-switch catalog.
- mzdash users are unaffected — `MaybeSwapProfileForCatalog` early-returns when `_profile != null && _profile.Name != CatalogProfileName`, so a user-loaded mzdash profile is never replaced. The catalog-fallback is strictly additive.

Tiered gap recovery in `TelemetrySender.HandleConfigJsonGap`: cached-state present → no recovery action (the wheel won't re-burst on a re-issued OpenRequest once it considers the session initialised); cached-state absent + first gap → prime + open-request; cached-state absent + ≥4 gaps within 8 s cooldown → full `RestartForSwitch`. The 30 s no-progress watchdog (`TickConfigJsonStuckWatchdog`) was over-aggressive pre-2026-05-09 and triggered Stop+Start whenever `LastState` was null even with healthy catalog + active subscription; it now skips when catalog is populated AND `_activeSubscription != null` (dashboard rendering doesn't need configJson library state).
- `WheelStateParser` / `WheelDashboardState` — Schema-tolerant parser that accepts BOTH firmware shapes (2025-11 `enableManager`/`dashboards`/`configJsonList`/`displayVersion`/`rootPath`, and legacy 2026-04 `enabledManager`/`updateDashboards`/`imagePath`). `WheelDashboardState` captures all 11 2025-11 top-level fields (TitleId, DisplayVersion, ResetVersion, SortTag, RootDirPath, ConfigJsonList, EnabledDashboards, DisabledDashboards, ImageRefMap, ImagePath, FontRefMap, RootPath) + full per-dashboard metadata (Title, DirName, Hash, Id, CreateTime, LastModified, PreviewImageFilePaths, ResourceImageFilePaths, IdealDeviceInfos). `Parse(bytes, out missingFields)` overload reports top-level fields absent from the parsed state for drift diagnostics. See `docs/protocol/dashboard-upload/config-rpc-session-09.md` for the full schema table.
- `TileServerStateBuilder` — Builds session 0x03 host→wheel tile-server state blobs. `BuildEmptyStateJson()` produces the 775-byte empty-ATS/ETS2 JSON PitHouse sends on connect. `Compress()` wraps with zlib. `BuildEnvelope(compSize, uncompSize)` emits the 12-byte `FF 01 00 [comp+4 u32 LE] FF 00 [uncomp u24 BE]` wrapper observed in captures. `BuildFullBlob(json)` combines the three into a ready-to-chunk payload for `TierDefinitionBuilder.ChunkMessage(payload, session=0x03, ref seq)`.
- `TileServerStateParser` — Inbound decoder for session 0x03 if the wheel ever pushes blobs back (not observed in any of 5 captures — session 0x03 is host→wheel only in observed protocol). `OnChunk()` accumulates chunks, recognises the 12-byte envelope, decompresses zlib, parses JSON into `TileServerState { Root, Version, Games[ats/ets2]: { Populated, Name, MapVersion, TileSize, ... } }`. The accumulator is capped at 1 MiB and dropped wholesale on overflow so a malformed envelope (sentinel mismatch never trims the buffer) cannot grow unbounded. Exposed via `TelemetrySender.TileServerState`. Dormant in normal operation; kept for forward compatibility.
- `SessionDataReassembler` — Reassembles 7c:00 type=0x01 chunks. CRC-aware trailer strip (tries full-chunk CRC first, then 1-byte flag-prefix variant), `TryDecompress()` pops the 9-byte `[flag][comp_size][uncomp_size][zlib]` envelope. Falls back to `TryDecompressByMagic` (scan for `78 9c` / `78 da` zlib magic + trial decompress each hit) when the offset-based header fails — survives envelope corruption from embedded 0x7E bytes in mzdash JSON payloads, mirroring `sim/wheel_sim.py`'s approach. Seq-aware overload `AddChunk(int seq, byte[], string tag)` handles three non-monotonic cases: (a) `seq == _lastSeq` → silent retransmit dedup (returns `true`, no buffer change); (b) `seq < _lastSeq` → silent re-burst (clear buffer, accept this chunk as new burst start — common on file-transfer sessions where each dir-listing/RPC-reply burst is independent); (c) `seq > _lastSeq + 1` → forward gap (e.g. when a chunk drops on Wine SerialPort R/W contention) → clears the buffer and returns `false` so the caller can re-handshake. The retransmit dedup at (a) is what keeps configJson / RPC / upload reassembly clean despite the wheel's natural retransmit cadence; without it the seq-unaware `AddChunk(byte[])` overload appends duplicates and corrupts the zlib stream.
- `RpcCallChannel` — Owns session 0x0a JSON RPC machinery (host→wheel `{method(): arg, id}` calls, replies routed by integer `id`). Extracted from TelemetrySender 2026-05-08; `TelemetrySender.SendRpcCall` is now a thin pass-through to `RpcCallChannel.Call`.
- `ChannelCatalogParser` — Owns the inbound 7c:00 byte buffer and the parsed wheel channel-URL list. Replaces the prior `_incomingSessionBuffer`/`_wheelChannelCatalog`/`_lastCatalogParseLen` fields scattered across TelemetrySender. Exposes `AppendChunk` (legacy seq-unaware), `AppendChunkIfNew(session, seq, …)` (preferred — dedupes wheel retransmits via per-session highest seen seq), `TryParse`, `BufferLength`, `Catalog`, **`LiveCatalog`** (see below), `LastActivityMs`, `HasCompleteCatalog`, `ClearBuffer` (drops in-progress buffer + dedup map but **preserves resolved `_catalog`** so post-switch back-references can resolve), `Reset` (full clear including `_catalog` + live-set state; for hot-swap only). **URL encoding forms accepted:** full text (must start with `v1/`), `0x01` prefix-compressed (expands to `"v1/gameData/" + rest`), `0x5C 0x31` (`\1`) abbreviation-compressed (expands to `"v1/gameData/" + rest` with `\t`/`\P`/`{FL}`/`{FR}`/`{RL}`/`{RR}` placeholder substitution), and `0x5C 0x70` (`\p`) abbreviation for `v1/gameData/patch/` channels — added 2026-05-22 after `patch/TrackPositionPercent` (track completion %) was silently rejected by the plausibility check and dropped from the live catalog every dashboard switch (the synth profile then sub'd back a tier-def without it, the wheel rendered the dash slot blank). Zero-length back-references (`size=1`, idx-only — "URL at this idx unchanged from before") are also accepted. Each accepted full URL is validated as printable ASCII (0x20..0x7E) before merge — catches the case where a stray `0x04` byte inside another record's data passes the lenient first-3-byte plausibility check and starts a fake URL parse. **Live-set tracking** (`LiveCatalog`, `_pendingIdxs`, `_lastCommittedIdxs`): the parser walks records in byte order; full-URL records add their idx to `_pendingIdxs`; back-refs do NOT (ambiguous with keepalive — crediting them polluted post-switch live sets with stale historical idxs in observed traces); each tag-0x06 END marker triggers an inline commit. The commit reads the END marker's actual `markerValue` (bytes 5..8 at the parse loop's current position) — NOT `_lastWheelEndMarker`, which holds the LATEST value from the pre-parse wire scan and was wrong for every non-final END in a multi-END buffer (a buffer like `[urls A] END=109 [urls B] END=138` produced two commits that both saw `_lastWheelEndMarker = 138` and misgrouped them into one generation). **First commit per `markerValue` wins**: subsequent commits at the same `markerValue` are protocol re-affirmation bursts (the wheel maintains its full historical catalog mapping across dashboards for back-ref resolution) and would otherwise extend the current dashboard's idx set with stale prior-dashboard idxs whose URLs the new dashboard doesn't use. Dropping them keeps the live catalog narrow and correct; the same drop semantically preserves the FIRST commit's idxs against later subset bursts (the dropped-Gear case where the wheel emits `{1..9}` then `{1..3}` keepalives at the same generation). Cross-generation commits replace; pending is also cleared at the start of every parse pass so carry-over from uncommitted generations can't pollute the FIRST commit of the next pass. Consumers needing "channels in the current dashboard" (catalog-only profile synthesis, channel-mapping UI) read `LiveCatalog`; back-ref resolution still uses `Catalog` (the full history). Full protocol details in [`protocol/tier-definition/session-02-channel-catalog.md`](protocol/tier-definition/session-02-channel-catalog.md) § Live-set tracking. **Pre-2026-05-09 bugs:** (1) `AppendChunk` blindly appended every received chunk, so wheel retransmits (2-3× per chunk before our ack lands) duplicated bytes in the buffer and misaligned the size-prefixed TLV walk for any record straddling a duplicated chunk boundary — surfaced as `v1/gameDa???????t???`-style garbled URLs deterministically every cold-start; (2) `StartInner` called `Reset()` which wiped `_catalog`, so post-switch back-reference records (commonly used to compress unchanged URLs across dashboard switches) all resolved to `backRefFail` and the post-switch catalog had gaps. See `findings/2026-05-09-acks-dedup-and-catalog-persistence.md`.
- `WheelUploadCoordinator` — Owns the wheel-side mzdash upload session lifecycle: FT-eligible session tracking (`NoteDeviceInit`), sub-msg state machine, dir-listing reassembly (`NoteInboundChunk`), end-marker observation (`NoteEndMarker`), `RunBackgroundUpload` entry point, the 2025-11 ↔ 2026-04 wire-format auto-fallback, and skip-if-already-loaded MD5 check. Hosts `MzdashContent` / `MzdashName` / `UploadDashboard` / `UploadSessionOverride` properties (TelemetrySender exposes thin pass-throughs).
- `SessionRegistry` — Tracks session byte → role assignment for 2025-11 firmware (0x01 mgmt, 0x02 telemetry, 0x03 aux config, 0x04 file transfer, 0x06/0x08/0x0A keepalive, 0x09 configJson). Records whether each session was opened by host or device (type=0x81 source).

**Profile System** (`UI/MozaProfile.cs`, `UI/MozaProfileStore.cs`, `UI/MozaPluginSettings.cs`) — Per-game configuration snapshots using SimHub's `ProfileBase`. After the 2026-05-14 refactor (see `docs/refactor-profile-system-2026-05-14.md` + the companion audit) telemetry/wheel state is split across four storage tiers based on what's actually game-scoped vs wheel-scoped:

- **Plugin-global** on `MozaPluginSettings` (one value, regardless of game/wheel): `ConnectionEnabled`, `LastWheelbasePort/LastAb9Port`, `DisableSerialProbeFallback`, `AutoApplyProfileOnLaunch`, `LimitWheelUpdates`, `WheelKeepalive`, telemetry send-rate/byte-budget toggles, etc.
- **Per-wheel-page** on `MozaPluginSettings` dicts keyed by SimHub `DescriptorUniqueId` GUID (shared across every game using that wheel):
  - `WheelMzdashFolderByPageGuid : Dictionary<Guid, string>` — mzdash library folder per wheel-page (one folder per physical wheel).
  - `WheelTelemetryEnabledByPageGuid : Dictionary<Guid, bool>` — "is dashboard telemetry on for this wheel".
  - `WheelTelemetryEraByPageGuid : Dictionary<Guid, int>` — firmware-era pick (cast from `MozaWheelEra`); Auto/missing means the auto-resolver picks from the wheel's handshake.
- **Per-(profile × wheel-page)** on `MozaProfile.WheelOverridesByPageGuid[pageGuid]` (one value per game-and-wheel combo). `WheelOverride` carries wheel LED / mode / brightness / sleep / colors / idle effects / input modes / knob colors, plus the per-game dashboard pick (`TelemetryProfileName`, `TelemetryMzdashPath`). Every scalar uses -1 (or null for arrays/strings) for "not overridden — fall through to baseline". `ApplyWheelToHardware` resolves the overlay via `_data.WheelModelName` → `WheelModelInfo.ExtractPrefix` → `MozaDeviceConstants.ResolveWheelGuid` and applies `Eff(overlay.X, profile.X)`. `WheelOverride` also has `[JsonExtensionData(WriteData = false)] LegacyJsonFields` — a deserialization catch-all that preserves removed properties (pre-v5 `TelemetryEnabled`, pre-v4 `TelemetryMzdashFolder`, pre-v6 `TelemetryWheelEra`) just long enough for the migration to drain them into the per-wheel-page dicts. The flag is set so post-migration empty dicts don't re-serialize.
- **Per-game profile baseline** on `MozaProfile` directly (one value per game, applies to whichever wheel is active): motor / FFB / handbrake / pedals, dash brightness, base-ambient (`BaseAmbient*`), gearshift event tuning (`GearshiftVibrateOnNeutral`, `GearshiftDebounceMs`), `Ab9Settings? Ab9` (mechanical mode + 7 sliders + `ushort EngineVibrationFrequency` 0..300 Hz), and `TelemetryDashboardKey` — the captured dashboard identity that gets re-applied on game-switch in the format `wheel:<configJsonId>` / `file:<filename>:<sha1-8>` / `builtin:<name>` per `MozaPlugin.GetActiveDashboardKeyCandidates`. `Ab9Settings.Clone()` must enumerate every property explicitly; the profile loader relies on it. `MozaProfile.TelemetryChannelMappings : Dictionary<Guid, Dictionary<string, Dictionary<string, string>>>` keys profile × wheel-page × dashboard-key × channel-URL → SimHub property path; wheel-page-scoped because dashboards available on a VGS aren't available on a CS V2.1.

`MozaProfile.CaptureFromCurrent` is now narrow: it captures only the device-read-sourced state from `_data` (motor/FFB/handbrake/pedals/FFB-curve/FFB-equalizer). Wheel-LED, dash, base-ambient, gearshift, AB9, and the wheel overlays are written directly by UI handlers (or by extension `ApplyTo`) — `CaptureFromCurrent` does not touch them, so capturing after a partial device read can't corrupt user-edited values.

The hardware-apply side lives on `Hardware/HardwareApplier.cs` as seven entry points: `ApplyWheelToHardware` (sources profile + wheel overlay, detection-gated on `DetectionState.NewWheelDetected || OldWheelDetected`), `ApplyDashToHardware` (gated on `DetectionState.DashDetected`), `ApplyBaseToHardware` (gated on `_data.IsBaseConnected`), `ApplyBaseAmbientToHardware` (gated on `DetectionState.BaseAmbientLedSupported`), `ApplyHandbrakeToHardware`, `ApplyPedalsToHardware`, `ApplyAb9ToHardware`. `MozaPlugin` owns the `HardwareApplier` instance and exposes `ApplyProfile` and `WriteIf*` forwarders. Every individual `_deviceManager.WriteSetting`/`WriteColor`/`WriteFloat` is sentinel-guarded (`if (value >= 0)`) inside these methods so a cold-start profile with all sentinels produces zero hardware writes — fixes the pre-refactor brightness write storm that pushed `100/100/15` for RPM/buttons/ES brightness unconditionally on every connect (the unconditional writes lived in the now-deleted `ApplySavedWheelSettings`). **`ApplyWheelToHardware` additionally gates LED writes by `WheelModelInfo` capabilities**: `wheel-rpm-brightness` / `wheel-telemetry-idle-effect` only when `RpmLedCount > 0`; `wheel-buttons-brightness` / `wheel-buttons-led-mode` / `wheel-buttons-idle-effect` / `wheel-button-color*` only when `ButtonLedCount > 0`; `wheel-knob-led-mode` / `wheel-knob-idle-effect` / knob signal modes only when `KnobCount > 0`; `wheel-flag-color*` only when `HasFlagLeds`. `WriteColorArray` calls pass the model's actual count instead of a hardcoded 18/14 so a 10-RPM wheel like the original CS doesn't receive 8 wasted RPM color writes. Same gating applies to the post-detect read batch — see `DeviceProber.BuildNewWheelLedReadCommands(WheelModelInfo)` below. Each method always mirrors profile/overlay values into `_data` first (so the UI reflects the active state even before the device is connected), then writes to hardware only when the matching detection flag is set. Invoked from `DetectDevices` (each detection-flag flip) and from `ApplyProfile` (game/profile switch). At the end of `ApplyProfile`, the telemetry pipeline is synced to the new active state: `ApplyTelemetrySettings()` + `StartTelemetryIfReady()` when `ActiveTelemetryEnabled` is true, `sender.Stop()` when false — so a game-switch flips the pipeline live.

The active-state read API on `MozaPlugin` is the five `ActiveTelemetry*` accessors: `ActiveTelemetryEnabled`, `ActiveTelemetryProfileName`, `ActiveTelemetryMzdashPath`, `ActiveTelemetryMzdashFolder`, `ActiveTelemetryWheelEra`. Each resolves the current wheel's page GUID and reads/writes the appropriate tier — `Enabled`/`Folder`/`Era` from the `*ByPageGuid` dicts (per-wheel-page), `ProfileName`/`MzdashPath` from the active profile's overlay. Returns sentinels ("", `false`, `MozaWheelEra.Auto`) when no wheel has identified yet.

UI handlers go through `WriteIf*` helpers (`WriteIfWheelDetected`, `WriteIfDashDetected`, `WriteIfBaseConnected`, `WriteIfHandbrakeDetected`, `WriteIfPedalsDetected`, `WriteIfBaseAmbientSupported`, `WriteColorIf*`, `WriteFloatIf*`, `WriteArrayIfWheelDetected`) so a slider drag while the device is disconnected updates the profile/overlay + persists but does not queue a wire write against nothing. **Device Extension System** (`Devices/`) — Registers MOZA devices as SimHub devices so they appear in SimHub's Devices section with native LED effects support. Each device type (wheel, dashboard) has its own extension, LED manager, settings class, and settings control:

*Shared:*
- `MozaDeviceExtensionFilter` — `IDeviceExtensionFilter` that routes devices by `DescriptorUniqueId` GUID to the correct extension. Uses `MozaDeviceConstants.GetWheelModelPrefix()` to match all known wheel GUIDs → `MozaWheelDeviceExtension`, `IsDashDevice()` to match the dashboard GUID → `MozaDashDeviceExtension`, and `IsBaseAmbientDevice()` to match the wheel-base ambient LED GUID → `MozaBaseDeviceExtension`
- `MozaDeviceConstants` — Per-model `DescriptorUniqueId` GUIDs, GUID-to-model-prefix mapping, and LED count constants. `GetWheelModelPrefix()` resolves a SimHub `DeviceTypeID` to the firmware model prefix the device expects. `BaseAmbientGuid` is a fixed (non-per-model) GUID used by the wheel-base ambient LED device, with `IsBaseAmbientDevice()` for filter routing
- `WheelModelInfo` — Per-model LED layout descriptor (RPM LED count, button count, flag presence, button index remapping, knob count + ring-LED layout, `bool? HasDisplay`). Resolved from the firmware model name string after wheel detection. Known models: GS V2P (10 RPM, 10 buttons, no flags, no display), CS V2.1 (10 RPM, 6 non-contiguous buttons, no display), original CS (10 RPM only, no buttons / flags / knobs / display — firmware reports the bare prefix `"CS"` with no version suffix; entry must follow `"CS V2.1"` in the `KnownModels` list so the longer prefix wins for newer firmware reports), W17 → friendly name "CS Pro" (16 RPM, 8 buttons, no flags, has display — firmware reports "W17"; verified by physical hardware inspection), W18 → friendly name "KS Pro" (18 RPM, 14 buttons, no flags, has display — firmware reports "W18"), W13 → friendly name "FSR V2" (16 RPM, 10 buttons, no flags, has display — firmware reports "W13"). Unknown models default to 10 RPM, 14 buttons, no flags, `HasDisplay = null` (treated as "probe to confirm"). Wheels with flag LEDs use a 3/N/3 physical layout (3 flag LEDs on each side of the RPM strip) exposed to SimHub as a single contiguous (RpmLedCount + 6)-LED telemetry sequence. **`HasDisplay` gates dashboard-related traffic** via `MozaPlugin.ShouldDriveDashboard()` (returns `false` when `HasDisplay == false`, `true` when `HasDisplay == true`, falls through to `IsDisplayDetected` for the unknown case): the dashboard pipeline (sess=0x01/0x02 tier-def, sess=0x02 init handshake, sess=0x03 tile-server state push), `CloseHostSessions()`, the wheel-knob/button LED writes that don't apply, and the `SendDisplayProbe()` 11-frame group-0x43 burst all skip when the gate returns false. Without this guard the plugin used to drive the full dashboard pipeline at a screenless wheel like CS V2.1 — the wheel's command parser would eventually park into a half-engaged state where settings reads timed out, producing the "wheel keeps dropping after detection" bundles. `DeviceProber` defers `SendDisplayProbe` from the wheel-detect site to the `wheel-model-name` handler so the `info.HasDisplay != false` check has real input to gate on; `MozaPlugin.PollStatus`'s re-probe path applies the same gate, so screenless wheels never see the burst. **Display-detected gate on telemetry start (2026-05-25):** `DashboardBindingCoordinator.StartTelemetryIfReady` additionally defers `TelemetrySender.Start()` until `MozaPlugin.IsDisplayDetected == true` for any wheel where the display gate above lets the pipeline run (i.e., `HasDisplay != false`). The W17 (CS Pro) — and likely other wheels with similar two-stage boot — answers wheel-MCU settings reads (`wheel-telemetry-mode` etc.) within ~50 ms of being clipped onto the base, but its display sub-device takes ~20 s to boot. The wheel ignores `7c:00 type=0x81` session opens on sess=0x01/0x02 during that window — they're addressed to the dashboard channel which physically isn't listening yet. Without the gate, hot-attach hits the failure mode where the host's session-open frames go unacked, the wheel never engages sess=0x01/0x02 from its own side, no channel catalog is pushed, and the dashboard layout renders locally with every channel stuck at zero until SimHub restarts. The gate is self-recovering: `DeviceProber`'s `display-model-name` case re-calls `StartTelemetryIfReady` once the identity response decodes, and `PollStatus` re-issues `SendDisplayProbe()` every 5 s while the wheel is detected but the display isn't — so a probe lost to USB jitter doesn't permanently wedge it. **Hot-swap reset:** `ResetWheelDetection` calls `_telemetrySender.ResetDisplayDetection()` to clear the sender's own `_displayDetected` / `_displayModelName` latch (separate from `_data.Display*` which `ClearWheelIdentity` already blanks); without that second clear the next wheel's `IsDisplayDetected` reads stale `true` from the prior wheel's latch and the gate bypasses, re-creating the hot-attach failure mode after every wheel-to-wheel swap. `Stop()` does NOT clear the display latch — game-switch and dashboard-switch cycles reuse the same wheel and must not re-pay the ~20 s display-probe wait every time. **Wedge watchdog:** `PollStatus` bounds the gate's wait via `WheelDetectedUtcTicks` (stamped by `DeviceProber`'s wheel-rising-edge sites via `NoteWheelDetected`); after 60 s of `WheelModelInfo.HasDisplay != false && !IsDisplayDetected`, the watchdog logs a wedge warning and calls `_connection.Disconnect()` — the 5 s reconnect timer reopens the port, USB re-enumerates, the display gets a fresh boot. One-shot per attach (`DisplayWedgeRecoveryFired` latch, cleared on the next successful display-model-name decode or by manual `SetConnectionEnabled(true)`) so a permanently-wedged display can't loop the connection. A future successful display detection re-arms the watchdog for the next swap. The wheel-ready latch on `TelemetrySender._wheelReadyObserved` (set by `TelemetryInboundDispatcher.HandleDeviceInit` when the wheel pushes its first sess=0x09 type=0x81) is retained as belt-and-suspenders for the `ProbeAndOpenSessions` extended-wait branch: if the display probe somehow gets through but the session opens are still silent at the 20 s extended-wait cap, the sess=0x09 push from the wheel is recognised as the alternate "alive" signal. Cleared via `SessionWatchdogManager.Reset()` (called from `Stop()`), so the latch re-arms on every Start/Stop cycle. See [`docs/protocol/identity/display-sub-device.md`](protocol/identity/display-sub-device.md), [`docs/protocol/startup-timeline.md`](protocol/startup-timeline.md), and [`docs/protocol/sessions/lifecycle.md`](protocol/sessions/lifecycle.md) for the wire timing.

*Wheel:*
- `MozaWheelDeviceExtension` — `DeviceExtension` subclass providing a settings tab and per-game device profiles via `GetSettings()`/`SetSettings()`. Resolves the expected model prefix from `DeviceTypeID` in `Init()` and passes it to the LED driver for model-aware connection binding
- `MozaLedDeviceManager` — Virtual `ILedDeviceManager` injected via reflection into the device's LED module. `IsConnected()` is model-aware: old-protocol devices match only `IsOldWheelDetected`, per-model devices match by firmware model name prefix, generic devices match any new-protocol wheel (unless a model-specific extension is active — tracked via copy-on-write `_activeModelPrefixes` set for thread safety). Fires `OnConnect`/`OnDisconnect` events from `UpdateConnectionState()`. Forwards computed `Color[]` to MOZA hardware as per-frame color chunks + bitmask in `Display()`. Button LED count and index mapping are model-aware via `WheelModelInfo` — non-contiguous layouts (e.g. CS V2.1) are remapped from SimHub's contiguous indices to the correct protocol positions. Before the bitmask is built, the "default during telemetry" pass (driven by `MozaData.WheelButtonDefaultDuringTelemetry`) clones the incoming SimHub button frame and replaces 'off' (0,0,0) with `WheelButtonColors[i]` for each button whose flag is set — runs whenever SimHub is feeding button colors, and writes via the live-telemetry pipeline (`wheel-telemetry-button-colors` + bitmask) so the override tracks frame-by-frame rather than persisting to EEPROM. For wheels with `HasFlagLeds=true`, SimHub delivers a single (RpmLedCount + 6)-LED telemetry sequence; the driver splits it `[0..2] → flag 1-3`, `[3..N+2] → RPM 1..N`, `[N+3..N+5] → flag 4-6`. RPM colors go to `wheel-telemetry-rpm-colors`. **RPM active-LED bitmask is sized per wheel** via `wheel-send-rpm-telemetry` — `BuildRpmBitmaskBytes(bitmask, ledCount)` picks 2 bytes for wheels with ≤16 RPM LEDs (legacy wire format) and 4 bytes for wheels with 17+ LEDs (KS Pro — 18 LEDs need bits 16–17). Extended form verified on KS Pro 2026-04-22. Buttons bitmask stays 2 bytes (max 16 buttons across known models). **Flag colors route to the Meter sub-device (device 0x14, write group 50) via `dash-flag-color1..6` per RS21 parameter DB (`MeterSetCfg_SetFlagGroupColor*`) — all flag writes are gated on `IsDashDetected`. The legacy `wheel-flag-color*` / `wheel-flags-brightness` entries on device 0x17 (id `[21, 2, i]` / `[27, 2, 0xFF]`) are left commented out in `MozaCommandDatabase.cs` for rollback reference — newer wheels silently drop them. Flag brightness routes to `dash-flags-brightness` and is slaved to `wheel-rpm-brightness` so the whole strip shares a master brightness knob**
- `MozaWheelExtensionSettings` — Wheel-specific settings serialized to SimHub device profiles. Tags its JSON with `WheelModelName` (informational only post-refactor — the SimHub page GUID derived from `_expectedModelPrefix` is the authoritative key). `ApplyTo(settings, data, profile, pageModelPrefix)` writes wheel LED state into `profile.WheelOverridesByPageGuid[pageGuid]` (the source of truth); `CaptureFromCurrent` reads from the same overlay. The wheel device extension always passes the prefix so `GetSettings` reflects live overlay state. Persists the per-button `WheelButtonDefaultDuringTelemetry[14]` flag array alongside `WheelButtonColors`. Telemetry-related fields (enable / dashboard / folder / era) are NOT in this DTO — telemetry-enable / folder / era live in the per-wheel-page dicts on `MozaPluginSettings` (`WheelTelemetryEnabledByPageGuid`, `WheelMzdashFolderByPageGuid`, `WheelTelemetryEraByPageGuid`); dashboard selection (`TelemetryProfileName`, `TelemetryMzdashPath`) lives on `WheelOverride` so it can differ per-game.
- `MozaWheelSettingsControl` — Per-wheel device page split across four partial-class files in `Devices/`: `MozaWheelSettingsControl.xaml.cs` (main refresh tick + telemetry + dashboard combo + RPM/Buttons/Flag swatches), `.Inputs.cs` (live paddles + buttons display + paddle / joystick / knob-signal handlers), `.Knobs.cs` (the per-knob `KnobRingViz` grid + signal-mode chip cards + PaletteStrip editor), `.Files.cs` (combined dashboard upload + wheel-side file inventory, moved from the plugin pane). XAML hosts capability-driven tabs (`InputsTab`, `DashboardTab`, `RpmTab`, `ButtonsTab`, `KnobsTab`, `FilesTab`, `SleepTab`); `FilesTab.Visibility` mirrors `DashboardTab`'s `showTelemetry` gate (wheel reported a dashboard → upload + file-list pipeline available). Tab styling shared with the plugin pane via `Style="{StaticResource MozaTabControl}"` (cyan underline on active). Connection status is driven by the linked LED driver's `IsConnected()`, not global plugin state. Flag LED color swatches and individual button swatches are shown/hidden based on `WheelModelInfo`. Each button swatch has a "default during telemetry" checkbox stacked below it (bound to `MozaData.WheelButtonDefaultDuringTelemetry[i]`); when checked, off-values sent through the live button-color telemetry path are replaced by that button's configured static color. **Tab gate (`IsWheelPageReady`)** — `newWheel = wheelConnected && !isOldProtoDevice && IsNewWheelDetected && IsWheelPageReady`. The page-ready flag (`MozaPlugin.IsWheelPageReady` → `GetCurrentWheelPageGuid().HasValue`) requires `_data.WheelModelName` to be populated, which only happens after the wheel responds to the identity probe. Detection runs in two beats — `wheel-telemetry-mode` flips `NewWheelDetected`, then `wheel-model-name` arrives a few hundred ms later. Gating tab visibility on the second beat means UI handlers (`GetOrCreateActiveWheelSleep`, `UpdateActiveWheelOverlay`, etc.) can't run before their page-guid lookups resolve — without it, a fast user click would silently drop into a null bundle/overlay and the change would never reach disk. **UI render reads from saved state, not `_data`.** `RefreshWheel` populates the Sleep dropdowns from `_plugin.ActiveWheelSleep` (the persisted bundle) and the RPM/Flag/Button swatches from the per-game overlay (`MergeOverlayIntoData` helper — per-slot, falls back to `_data` only when the overlay slot is null). `_data` mirrors the wheel's transient response state and can drift from saved values (e.g. wheel returns `wheel-idle-timeout=0` when sleep mode disables the active timer, or returns a stale colour for slot 0 before our write lands); reading from `_data` for display caused the "sleep value got wiped" and "first RPM swatch reverts" symptoms even though the bundle/overlay on disk was correct. **Inline colour editor on RPM/Flag/Button pages**: clicking a swatch sets a per-section selection (cyan border + soft-glow), pre-seeds the `PaletteStrip` with the swatch's current colour, and routes colour picks through `CommitColorToSwatch` which writes to the wheel + `_data.Wheel{Rpm,Flag,Button}Colors[i]` + packs the full array into `WheelOverridesByPageGuid[guid].Wheel{Rpm,Flag,Button}Colors` via `UpdateActiveWheelOverlay`. The overlay update is non-optional — `MozaProfile.CaptureFromCurrent` deliberately skips wheel-LED fields ("UI handlers must write the overlay directly"), so a swatch commit that only mutated `_data` would be reverted on the next `ApplyWheelToHardware` tick. **Dashboard telemetry section**: enable checkbox + dashboard profile selector + `.mzdash` loader + test-pattern buttons. A sibling "CHANNEL MAPPINGS" card (no longer behind an `Expander`) shows every channel in the currently-loaded dashboard (Name + editable `SimHubProperty` ComboBox + right-justified current value). `PopulateChannelMappingGrid` re-runs whenever a cheap signature (`profile-ref-hash | tier-count | total-channel-count | string-channel-count | catalog-count`) changes from one `RefreshTelemetryStatus` tick to the next — picks up wheel-streamed catalog growth automatically without polling on every tick. Also re-runs from `OnPluginDashboardSelectionChanged` (wheel-initiated switches go through `DashboardBindingCoordinator.RaiseDashboardSelectionChangedInternal` → fires the event → grid repopulates) and from `InitTelemetryUI` (one-shot seed once `_plugin` is resolved). Channels whose default `SimHubProperty` starts with `@internal/` are hidden from the grid (plugin-locked). **Extended LED groups (experimental)**: at the bottom of the new-wheel panel, a diagnostic section exposes per-slot controls (fill color, Range min/max, Fill all / All off / Send one index, brightness, mode) for slot 0=Shift/RPM, 1=Button, 2=Single, 3=Rotary, 4=Ambient, 5=Meter flags. Slots 0/1/5 always visible; 2/3/4 unhidden on positive brightness-read probe response (**probe is unreliable — firmware acknowledges reads for parameters with no hardware**, so confirmed support must come from per-wheel testing). Groups 0 + 1 write via the live telemetry path (`SendColorChunks` + bitmask — overwritten each frame by SimHub effects). Groups 2/3/4/flags write via static per-LED commands. Per-slot min/max persists via `MozaPluginSettings.ExtLedDiagMin/Max[]`. A read-only summary TextBox at the top of the section lists the configured LED count per group — users paste this into bug reports so `WheelModelInfo` can be populated per model

*Dashboard:*
- `MozaDashDeviceExtension` — Dashboard device extension, same pattern as wheel
- `MozaDashLedDeviceManager` — Virtual `ILedDeviceManager` for the dashboard. Reports connected when the dash is detected. Combines RPM colors (from telemetry LEDs, bits 0-9) and flag colors (from button LEDs, bits 10-15) into a 16-bit bitmask sent via `dash-send-telemetry`. Unlike the wheel, no per-frame colors are sent — the dash firmware uses stored colors and only receives the on/off bitmask. Separate from the telemetry data streaming (handled by `TelemetrySender`), which sends full game data for dashboard display widgets
- `MozaDashExtensionSettings` — Dashboard-specific settings (brightness, indicator modes, colors). `ApplyTo(settings, data, profile)` writes into the active profile baseline (`DashRpmBrightness` / `DashFlagsBrightness` / `DashDisplayBrightness` / `DashDisplayStandbyMin` / `DashRpmColors` / `DashRpmBlinkColors` / `DashFlagColors`). `CaptureFromCurrent` reads from the profile. Indicator/display modes still capture from `_data` since they're polled from the device, not stored on the profile.
- `MozaDashSettingsControl` — Status panel with indicator modes, brightness, and color swatches (RPM, blink, flags). Connection status driven by linked LED driver's `IsConnected()`

*Wheel Base Ambient (R21/R25/R27 family — 18-LED telemetry strip on the wheelbase body):*
- `MozaBaseDeviceExtension` — Same pattern as the dashboard extension; predicate uses `IsBaseAmbientLedSupported`. **Capability gating:** the device definition is deployed lazily, only after the runtime probe in `MozaPlugin` (a `base-ambient-brightness` read scheduled once per base detect, gated by `_baseAmbientProbed`) gets a `0xA2` reply. R9/R12 bases drop the read silently — the flag stays false and `DeviceDefinitionDeployer.DeployBaseAmbient` never runs, so SimHub never sees a "MOZA Wheel Base" device on bases that don't ship the strip. Probe + the diagnostic base identity read (`wheel-model-name` retargeted at dev `0x12`) fire from the existing `case "base-mcu-temp":` block. Identity responses are intercepted ahead of `MozaData.UpdateFromCommand` (route by `r.DeviceId == MozaProtocol.DeviceMain`) and routed to `_data.BaseModelName` so the wheel identity field never gets polluted with the base name
- `MozaBaseLedDeviceManager` — Virtual `ILedDeviceManager` that splits SimHub's 18-LED contiguous strip into the two physical 9-LED strips (LEDs 0–8 → strip 0, 9–17 → strip 1). Sends per-LED color chunks (cmd `0x1A` on group `0x20`, 4-byte `[idx, R, G, B]`, two chunks per strip — LEDs 0–4 = 20 bytes, LEDs 5–8 = 16 bytes + an `0xFF` index-padding byte so the firmware doesn't read the trailing zero-pad as "set LED 0 black"); chunks are re-sent only when the FNV-1a palette hash changes per strip (matches PitHouse capture cadence). Bitmask (cmd `0x1B`, 4-byte LE u32 with 9 active bits per strip) goes out every Display() call when the value changes — or every frame if `AlwaysResendBitmask` is set, mirroring the dash manager. Per-frame `rpmBrightness` from SimHub's pipeline scales each color channel before encoding. Idle handoff: when SimHub stops feeding telemetry colors (game exits / scene transitions deliver a zero-length array), one final bitmask=0 fires to both strips on the active→idle edge and the manager goes quiet, letting the firmware's standby animation (rainbow / breath / cycle / flow) take back over
- `MozaBaseExtensionSettings` — Persisted ambient LED settings (brightness, standby mode, indicator state, sleep mode/timeout, packed startup/shutdown color). Same `-1`-sentinel-for-not-included convention as `MozaDashExtensionSettings`. Per-base hardware setting — does NOT serialize a `WheelModelName` like `MozaWheelExtensionSettings`. The values are profile-scoped (`MozaProfile.BaseAmbient*`) so each game can pick its own ambient palette; `ApplyTo(settings, data, profile)` writes into the profile and `CaptureFromCurrent` reads from it.
- `MozaBaseSettingsControl` — Brightness slider (0..255), standby-mode dropdown (Constant / Breath / Cycle / Rainbow / Flow — device modes 0/2/3/4/5; the unconfirmed mode 1 is hidden via the `MapStandbyDeviceToUi` / `MapStandbyUiToDevice` helpers), indicator toggle, sleep mode + timeout, startup/shutdown color swatches via `ColorPickerDialog`. Connection status driven by linked LED driver's `IsConnected()`. The base model name (`R25 Black # MOT-1` etc., parsed from group `0x07/01` against dev `0x12`) shows in the header for diagnostics

**Device Definitions** (`DeviceTemplates/`) — Device definitions using SimHub 9.11+'s Device Builder format. Each directory contains a `device.json` with the device's LED layout baked in, embedded as resources in the DLL. Deployed lazily to SimHub's `DevicesDefinitions/User/` directory when the matching device is first detected over serial — not at startup. The plugin settings panel surfaces a "restart SimHub" banner via the `StatusHintBuilder` pipeline (see the UI section) when new definitions are deployed, and — once the user has restarted — a follow-up "add the device under SimHub > Devices" banner per detected device type whose corresponding `*DeviceExtensionActive` flag is still false 5 s after the wheel handshake. Per-model wheel definitions are generated dynamically at runtime from `WheelModelInfo` (RPM count, button count, flag presence). `DeployGeneratedWheelDefinition` reads any existing `device.json` and compares `LogicalTelemetryLeds.LedCount` and button-section item count against the model's expected values — a mismatch triggers a rewrite, so plugin updates that change a model's layout (e.g. adding flag LED support) propagate to existing users on next wheel connect. The user must restart SimHub for the refreshed definition to take effect. The wheel-model-prefix → GUID registry is persisted via `MozaDeviceConstants.SaveRegistryFile()` using a write-temp-then-Move sequence so a crash mid-write can't truncate the existing registry to garbage and lose the GUID-to-prefix map on next load.
- `MozaWheelGeneric/` — Generic fallback (14 buttons) for unknown new-protocol wheels
- `MozaWheelOldProto/` — Old-protocol (ES) wheels — RPM LEDs only
- `MozaDashShdp/` — Dashboard definition (10 RPM LEDs + 6 flag LEDs)
- `MozaWheelBase/` — Wheel-base ambient LED strip (R21/R25/R27 family — 18 LEDs as a single contiguous logical telemetry strip; the manager splits to two physical 9-LED strips on the wire). Deployed only when the runtime base-ambient probe succeeds; bases without the strip never see this device

**Telemetry Data** (`Data/`) — Embedded resources for the telemetry streaming system:
- `Telemetry.json` — Channel definitions (400+ entries) describing the telemetry channels supported by Moza dashboards, each with URL, compression type, and package_level. Optional per-sector `simhub_property` (full SimHub property path, e.g. `DataCorePlugin.GameData.Rpms`) seeds a default SimHub binding at load time; optional `simhub_scale` (number, default 1.0) applies a multiplier before encoding — used where SimHub's unit convention differs from the compression's expected range (e.g. Throttle/Brake 0–100 → `float_001` expects 0–1, so `simhub_scale: 0.01`; Heading/Pitch/Roll radians → `tyre_temp_1` expects degrees, so `simhub_scale: 57.2957795`). The sentinel `simhub_property: "@internal/..."` locks the channel to a plugin-computed value (hidden from the mapping UI). Loaded by `DashboardProfileStore` at runtime

### Standalone dashboard pipeline (CM2)

The MOZA CM2 Racing Dash (USB PID `0x0025`, `MozaDeviceCategory.Dashboard`) is a standalone USB dashboard with no wheelbase — the plugin's full dashboard pipeline (screen telemetry, dashboard library + kind=4 switch, stored RPM colors, meter-mode + threshold writes) runs against it. CM2 has 16 physical RPM LEDs (no buttons, no separate flag strip per user-confirmed layout); physical order is logical 1–3 left side bottom-to-top, 4–13 top row left-to-right, 14–16 right side top-to-bottom (`usb-capture/CM2.md` 2026-05-21 lab notes).

**Detection.** `MozaPortDiscovery` admits Dashboard-category PIDs through the wheelbase connection's PID filter alongside `Wheelbase` and `Hub`. `MozaPlugin.IsStandaloneDashboardUsbConnection` returns true whenever the open serial port has a Dashboard PID; `IsDashDetected` ORs that with the wheelbase-relay `DetectionState.DashDetected` so all dash-detected gates flip true. `MarkStandaloneDashboardDetectedFromUsb(reason)` (idempotent) sets `DetectionState.DashDetected` + `_data.IsDashboardConnected`, deploys the CM2 device.json, reads CM2 settings, applies the current dash profile, asks the binding coordinator to apply telemetry settings, and starts the sender. It's wired at the end of `Init()` (covers persistent-connection reuse + reload-without-restart) and at the end of `TryConnect()` after a successful open. `MozaData.IsDashboardConnected` is a third truth source for `MozaData.IsConnected`, so the "can I send commands?" guard flips true with only a CM2 on the bus.

**Telemetry retargeting.** `TelemetrySender` owns `byte _targetDeviceId` (default `MozaProtocol.DeviceWheel = 0x17`) plus `TargetDeviceIdSwapped`, `IsStandaloneDashboardTarget`, `StandaloneDashboardMode`, and `TargetDescription` properties. `DashboardBindingCoordinator.ApplyTelemetrySettings` reads `plugin.ShouldUseStandaloneDashboardTarget()` and sets `sender.TargetDeviceId = plugin.PreferredStandaloneDashboardTargetDeviceId` (pinned to `MozaProtocol.DeviceMain = 0x12` for CM2). The setter invalidates `_cachedDisplayConfigFrames` and rebuilds per-tier `TelemetryFrameBuilder` instances so value frames address the new dev_id from the next tick. `_targetDeviceId` is threaded through `SendSessionOpen`/`SendSessionAck`/`SendSessionPrime`/`SendSessionEnd`, `BuildDisplayFrame`/`BuildDisplayFrameWithData`, `MaybeSendConfigJsonReply`, `SendTileServerState`, every `TierDefinitionBuilder.ChunkMessage` call site (including `TierDefinitionEmitter`, `PropertyPushQueue` for FF kind=4 dashboard switches + brightness pushes), and a new `_dashKeepaliveFrameMain` for the 0x12 ping. `TelemetryFrameBuilder` and `TierDefinitionBuilder.ChunkMessage` take an optional `byte deviceId = MozaProtocol.DeviceWheel` parameter so wheel call sites stay untouched. `TelemetryInboundDispatcher` accepts `0xC3` from `_sender.TargetDeviceIdSwapped` (replaces the previous hardcoded `0x71` filter); when in standalone-dashboard mode it also accepts the swapped forms of `DeviceMain` (0x21), `DeviceDash` (0x41), and `DeviceWheel` (0x71) as a wide fan-in — narrow once CM2 wire-traces nail down which command families answer on which dev_id. `ResolveAutoPolicy` pins `Era2026` when the standalone target is active, so the policy resolver doesn't wait for a wheel-catalog push that never arrives. `DashboardBindingCoordinator.StartTelemetryIfReady` allows start without a wheel detected when the standalone target is active and skips the displayless-wheel capability gate.

**Meter-config + live LED command surface.** `Protocol/MozaCommandDatabase.cs` registers a `cm2-*` command block under write group `0x32` (dev=`0x12` via the new `"cm2-main"` device-type → `MozaDeviceManager.GetDeviceId`): `cm2-indicator-brightness` (`17 00 FF`, lab-verified), `cm2-normal-mode` (`18 00`), `cm2-standby-mode` (`19 00`), `cm2-rpm-group-mode` (`11 00`), `cm2-flag-group-mode` (`11 02`), `cm2-rpm-regulation-mode` (`0D`), `cm2-rpm-percent-thresholds` (`05` + 10 bytes), `cm2-rpm-absolute-threshold{1..10}` (`0E <i>` + u32), `cm2-stored-color{1..16}` (`1B 00 FF <i>` + RGB, lab-verified persistent). All write-only (read_group `0xFF`) to avoid colliding with legacy `dash-*` read paths at group 51. `MozaDeviceManager` exposes `WriteSettingForDevice` / `WriteArrayForDevice` / `WriteColorForDevice` helpers that override a command's default dev_id at call time — used to retarget the wheel RPM-bar live commands (`wheel-telemetry-rpm-colors` cmd `25 00`, `wheel-send-rpm-telemetry` cmd `26 00`) from dev=`0x17` to dev=`0x12` for the per-frame CM2 LED path. The legacy `0x41 FD DE` dashboard bitmask did NOT visibly drive CM2 LEDs in lab tests, so it's skipped for CM2.

**Apply flow.** `HardwareApplier.ApplyDashToHardware` branches on `plugin.ShouldUseStandaloneDashboardTarget()` and calls `ApplyCm2DashboardConfig` which writes `cm2-normal-mode = 1`, `cm2-rpm-group-mode = 1`, `cm2-flag-group-mode = 1`, `cm2-rpm-regulation-mode = 0` (percent), a 10-byte percent ramp, 10 absolute RPM thresholds derived from a static MaxRpm = 8000 (TODO: plumb from active game), `cm2-indicator-brightness` from `DashRpmBrightness`, and 16 stored colors mapped as `DashRpmColors[0..9]` → `cm2-stored-color{1..10}` + `DashFlagColors[0..5]` → `cm2-stored-color{11..16}`. Legacy `dash-flags-indicator-mode = 1` and per-LED `dash-rpm-color*` / `dash-flag-color*` writes are skipped for CM2 (no visible effect at dev=0x14); brightness writes via `dash-rpm-brightness` / `dash-flags-brightness` stay as belt-and-suspenders. `MozaDashLedDeviceManager.Display()` branches on the same flag — for CM2 it emits the wheel-style RPM color chunks (resent only when at least one of 16 slots changes) + per-frame bitmask via `WriteArrayForDevice` at dev=`0x12`, instead of the legacy `dash-send-telemetry` single-int bitmask at dev=`0x14`.

**Device template + GUID.** `MozaDeviceConstants.DashCm2Guid = "6a2d9b0f-8b1e-4d32-9c18-1f5ec8a81025"` (separate from legacy `DashGuid`). `IsDashDevice` matches both GUIDs so the existing `MozaDashDeviceExtension` + filter both attach to CM2 without a new extension class. `DeviceTemplates/MozaDashCm2/MOZA CM2 Racing Dash/device.json` declares 16 RPM telemetry LEDs (`SourceRole=1, RepeatCount=16`) with no buttons section. `DeviceDefinitionDeployer.DeployDashboard(pid)` routes by PID: CM2 PID → CM2 template (with the CM2 GUID), all other dashboard PIDs → legacy SHDP template. `DeployFromResource` now compares both `DescriptorUniqueId` AND `Pid` on existing files (load-bearing fix — previously only PID mismatch triggered rewrite, so a pre-existing SHDP `device.json` with the CM2 PID would never be replaced even with a different GUID); the CM2 path also rewrites when `ProductName != "CM2 Racing Dash"` so hand-renamed JSON gets corrected. `DeviceDetectionState.ResetWheel()` clears `DashDetected` unconditionally; `MozaPlugin.ResetWheelDetection` re-asserts it when the connection is a standalone-USB dashboard so wheel hot-swap events don't blank CM2 detection.

**Unverified pieces (TODO markers in code).** Live per-frame RGB push via the wheel RPM-bar commands at dev=`0x12` is a working hypothesis — never captured against a real CM2. If LEDs stay flat in-game even though screen widgets animate, fall back to firmware-driven LEDs (`cm2-normal-mode = 1` + threshold writes alone drive the LED ramp from the screen-telemetry stream) and file a wire-trace capture request. Also unverified: `cm2-normal-mode 1` vs `2` (visually similar in lab), `cm2-rpm-regulation-mode` encoding (percent vs absolute), the inbound dispatcher's broad 0x21/0x41/0x71 fan-in for standalone targets. `.mzdash` upload to CM2 device storage is explicitly out of scope; the existing `Telemetry/Dashboard/` upload scaffolding remains untouched per the preserve-upload-scaffolding rule.

### Internationalization (i18n)

User-visible strings live in `Resources/Strings.resx` (English neutral / master) plus per-culture variants for de, el, es, fr, it, ru, vi, and zh-Hans (`Strings.<culture>.resx`). XAML references them via `{x:Static res:Strings.<Key>}` after declaring `xmlns:res="clr-namespace:MozaPlugin.Resources"` on the root element; C# code uses `MozaPlugin.Resources.Strings.<Key>`. The strongly-typed accessor lives in `Resources/Strings.Designer.cs` — hand-edited (one line per key), not generated by the SDK.

**Single-DLL deployment.** Every locale is embedded *directly inside `MozaPlugin.dll`* — there are no per-culture satellite assemblies and no `<culture>/MozaPlugin.resources.dll` subfolders to ship. The csproj sets `<WithCulture>false</WithCulture>` on each non-neutral `Strings.<culture>.resx` so MSBuild bundles them as plain embedded resources under explicit `ManifestResourceName` keys (e.g. `MozaPlugin.Resources.Strings.es.resources`) rather than routing them to satellites. `Strings.Designer.cs` builds a `Dictionary<string, ResourceManager>` keyed on BCP-47 tag, one entry per embedded blob, and `Get(key)` walks `Thread.CurrentUICulture`'s parent chain through that dictionary (passing `CultureInfo.InvariantCulture` to each `GetString` call so the RM reads its own constructed resource instead of doing satellite culture lookup). Final fallback is the neutral resource (English).

`Resources/LanguageResolver.cs` resolves the active culture each time `MozaPlugin.Init` and `GetWPFSettingsControl` fire. Precedence: explicit picker preference (`MozaPluginSettings.PreferredLanguage`) → SimHub's own `Culture` field read from `<install>/PluginsData/GlobalSimhubSettings.json` → OS UI culture → English. Each candidate walks its parent chain, so `es-MX` resolves to `es`. The UI thread's `Thread.CurrentUICulture` is reassigned on the WPF UI thread inside `GetWPFSettingsControl` before constructing `SettingsControl` — SimHub doesn't propagate its language onto plugin threads, and `x:Static` bindings evaluate at XAML parse time.

**Adding a new key:** edit `Strings.resx` to add the master entry, add a matching `<data name="...">` line to every other `Strings.*.resx`, and append a one-line `public static string X => Get("X");` property to `Strings.Designer.cs`. Skipping any of these three breaks something — missing resx entry returns the key string at runtime; missing Designer property fails XAML compile.

**Adding a new language:** four edits.
1. Copy `Strings.resx` to `Strings.<culture>.resx` and translate the values.
2. Add an `<EmbeddedResource Update="Resources\Strings.<culture>.resx"><WithCulture>false</WithCulture><ManifestResourceName>MozaPlugin.Resources.Strings.<culture></ManifestResourceName></EmbeddedResource>` entry to `MozaPlugin.csproj` (matching the existing es/fr/ru entries).
3. Add the culture code to `SupportedCultures` and a display name to `DisplayNames` in `Resources/LanguageResolver.cs`.
4. Add a row to the `_byCulture` dictionary in `Resources/Strings.Designer.cs` pointing at the same `ManifestResourceName`.

Rebuild and the new locale is bundled into `MozaPlugin.dll`. The Options-tab picker enumerates `SupportedCultures` so the new language appears without any UI changes.

### Adding New Device Settings

When adding a new setting that is written to the device, it must also be saved/restored with the profile system. Pick the storage tier first — see the four-tier classification in the Profile System section above (plugin-global / per-wheel-page / per-(profile × wheel-page) / per-game profile baseline) — then walk the workflow:

1. **`Protocol/MozaCommandDatabase.cs`** — Add command definition (name, device, read/write groups, command ID, payload size, type)
2. **`MozaDeviceManager.cs`** — Add device type mapping in `GetDeviceId()` if it's a new device
3. **`Protocol/MozaProtocol.cs`** — Add device ID and read/write group constants if needed
4. **`MozaData.cs`** — Add volatile field(s) and `UpdateFromCommand` case(s)
5. **`Devices/DeviceProber.cs`** — Add to `StatusPollCommands` (if it's a detection probe) and the appropriate per-device read array (e.g. `BaseSettingsReadCommands`, `NewWheelSettingsReadCommands`, `HandbrakeSettingsReadCommands`, etc.) so it's read after that device is detected. Add detection logic in `DetectDevices()` if needed. Push the new value through the matching `Apply*ToHardware` method in `Hardware/HardwareApplier.cs` (sentinel-guard the write: `if (value >= 0) _deviceManager.WriteSetting(cmd, value)`). For wheel-overlay fields source from `Eff(overlay?.X ?? -1, profile.X)`; for profile-level fields source from `profile.X` directly; for per-wheel-page fields read the matching `MozaPluginSettings.*ByPageGuid[pageGuid]` entry.
6. **Storage** — depends on the chosen tier:
   - **Per-game profile baseline** → add the property to `MozaProfile` directly, copy in `CopyProfilePropertiesFrom()`. Only add a `CaptureFromCurrent()` line if the value flows from device reads via `_data`; UI-edited fields are written to the profile by handlers directly and capture would clobber them.
   - **Per-(profile × wheel-page)** → add the property to `WheelOverride` (nested in `MozaProfile.cs`), copy in `WheelOverride.Clone()`. No `CaptureFromCurrent` capture — UI handlers write the overlay directly.
   - **Per-wheel-page (shared across games)** → add a `Dictionary<Guid, …>ByPageGuid` on `MozaPluginSettings`. Add a getter/setter on `MozaPlugin` that resolves the current page GUID (via `GetCurrentWheelPageGuid` → `MozaDeviceConstants.ResolveWheelGuid`) and reads/writes the dict. Add a migration step under `Settings/SettingsMigrator.MigrateToSchemaV2` that (a) drains pre-removal per-overlay values from `WheelOverride.LegacyJsonFields` and (b) seeds every known wheel page GUID from any pre-existing flat field. Bump `SettingsSchemaVersion`.
   - **Plugin-global** → add the property directly to `MozaPluginSettings`.
7. **`SettingsControl.xaml`** (or `Devices/Moza{Wheel,Dash,Base}SettingsControl.xaml`) — Add UI controls
8. **UI handler** — Update the matching control's `_ValueChanged`/`_Click` handler. Pattern by tier:
   - Profile-level field: `_plugin.UpdateActiveProfile(p => p.X = val)` then `_plugin.WriteIf<Device>(cmd, val)` then `_plugin.SaveSettings()`.
   - Wheel-overlay field: `_plugin.UpdateActiveWheelOverlay(o => o.X = val)` then `_plugin.WriteIfWheelDetected(cmd, val)` then `_plugin.SaveSettings()`.
   - Per-wheel-page field: `_plugin.ActiveXxx = val;` (the setter auto-routes to the dict via the page-guid resolver) then `_plugin.WriteIf<Device>(cmd, val)` then `_plugin.SaveSettings()`.
   - Plugin-global field: `_plugin.Settings.X = val;` then `_plugin.SaveSettings()`.
   - Color: use the matching `_plugin.WriteColorIf<Device>(cmd, r, g, b)` helper.
   - Array: use `_plugin.WriteArrayIfWheelDetected(cmd, payload)`.
   - `SaveSettings()` is debounced (500 ms). UI handlers update `_data.X` too where applicable so the next 500 ms refresh tick shows the new value immediately without waiting for the device echo.

Every setting that writes to the device on UI change must round-trip through profiles or the per-wheel-page dicts. If it's only in a transient field, it will be lost on game/profile switch.

**Host-rendered settings** (e.g. AB9 engine vibration intensity/frequency) skip steps 1–4 entirely — there is no `MozaCommandDatabase` entry because the wire frame is built inline by the device manager, no `MozaData` field because the value lives only in the profile, no detection probe because the device never persists the setting. The workflow collapses to step 6 (`MozaProfile` property) + step 7/8 (UI + handler). The UI handler does **not** call any `Send*` device write — it just updates the profile and `SaveSettings()`. Whatever periodic worker reads the setting (e.g. `Ab9EngineVibrationLoop`) picks up the new value on its next tick. **One-shot host-side config writes that ARE device-persisted but bypass the command-DB** (e.g. AB9 gear-shift-vib intensity — frame shape doesn't fit the single-byte slider convention) follow steps 6–8 plus an explicit `Send*` call in the UI handler AND in `ApplyAb9ToHardware` so the value is pushed on device attach / profile load.

**Settings storage and migration.** `MozaPluginSettings` holds plugin-global toggles, the SimHub-managed `ProfileStore`, the per-wheel-page dicts (`WheelMzdashFolderByPageGuid`, `WheelTelemetryEnabledByPageGuid`, `WheelTelemetryEraByPageGuid`, `WheelSleepByPageGuid`, `WheelIdleByPageGuid`), the `WheelExtensionDrained` one-shot flag (see `MozaWheelExtensionSettings.ApplyTo` for what it gates), and scratch space for legacy stores still readable from disk. `MozaPluginSettings.SettingsSchemaVersion` tracks the migration cutover and gates `Settings/SettingsMigrator.MigrateToSchemaV2`:
- v2: legacy UID-keyed slots (`PerWheelSlots`, `TelemetryByWheelUid`, `WheelMzdashFolderByUid`, `TelemetryChannelMappingsByWheel`) translated into `profile.WheelOverridesByPageGuid` / `profile.TelemetryChannelMappings`.
- v3: flat telemetry fields drained into overlays.
- v4: `TelemetryMzdashFolder` lifted off the overlay into `WheelMzdashFolderByPageGuid`.
- v5: `TelemetryEnabled` lifted off the overlay into `WheelTelemetryEnabledByPageGuid`.
- v6: `TelemetryWheelEra` lifted off the overlay into `WheelTelemetryEraByPageGuid`.
- v7: per-profile baseline reseed from flat fields (repair pass for users stuck at the broken v6 short-circuit).
- v8: `WheelSleep*` fields (mode/timeout/speed/color) lifted off the overlay into `WheelSleepByPageGuid` — sleep is a per-wheel firmware preference, not per-game. Migration drains the legacy DTO fields + overlay's `LegacyJsonFields` + flat `WheelSleepTimeoutMin`/etc. via `MigrateWheelSleepToPerPage`.
- v9: `WheelIdleEffect`/`WheelButtonsIdleEffect`/`WheelKnobIdleEffect`/`Wheel*IdleSpeedMs` lifted off the overlay into `WheelIdleByPageGuid` — same per-wheel-page rationale.

The migration is gated by `SettingsSchemaVersion >= 9`; users at any prior version run forward through every step in one pass. `WheelOverride.LegacyJsonFields` (a `[JsonExtensionData(WriteData = false)]` catch-all) preserves removed JSON keys through deserialization so each migration step can drain them; afterwards the dict is cleared and the `WriteData = false` flag stops them from re-serializing. `ClearLegacyAfterMigration` empties the legacy dicts and remaining flat fields once each successful pass completes. Use `MozaDeviceConstants.ResolveWheelGuid(prefix)` for any new code that needs to map a wheel-model prefix to a SimHub page GUID — never hard-code GUIDs.

**Device JSON ↔ plugin settings drain** is one-shot, gated by `MozaPluginSettings.WheelExtensionDrained` (bool, persisted). The first time `MozaWheelExtensionSettings.ApplyTo` runs after install (or after the field was added), it merges the DTO's non-sentinel values into the per-page bundles + overlay using **fill-only** semantics (only writes a bundle/overlay field that's still at its sentinel — `-1` for scalars, `null` for arrays), then stamps `settings.WheelExtensionDrained = true`. Every subsequent `SetSettings` call (on any wheel device-extension instance — the user can accumulate several over time) early-returns before touching anything. This protects against stale device JSON clobbering the plugin-settings source of truth: SimHub only re-serializes the device JSON when it calls `GetSettings`, which doesn't reliably happen before shutdown, so an out-of-date DTO can outlive the plugin settings file on disk. Gating on the plugin-side flag (saved via the plugin's own debounce timer + `End()` immediate save) is reliable in a way the DTO-side flag wasn't — `End()` always fires, `GetSettings` doesn't.

### Adding a New Telemetry Channel

Most new channels only need step 1 — set `simhub_property` (and optional `simhub_scale`) in `Telemetry.json` and the resolver pulls the value directly via `PluginManager.GetPropertyValue`. Steps 2–4 are only required when the channel should ALSO be readable via the legacy `SimHubField` snapshot path (e.g. for tests that don't have a `PluginManager`, or when the property needs pre-scaled access via `GameDataSnapshot`).

1. **`Data/Telemetry.json`** — Ensure the channel's URL, compression type, and package_level are defined. Add an optional `simhub_property` (full SimHub property path, e.g. `DataCorePlugin.GameData.OilPressure`) to pre-assign a default mapping, and `simhub_scale` (number) if the SimHub value's unit doesn't match the compression's expected range (e.g. `0.01` for 0–100 → 0–1; `57.2957795` for radians → degrees). Use `simhub_property: "@internal/<key>"` to lock the channel to a plugin-computed value (add the corresponding `case` in `MozaPlugin.ResolveInternalChannel`).
2. *(optional)* **`DashboardProfileStore.cs`** — Add the URL suffix → `SimHubField` mapping to `UrlFieldMap` (legacy fallback when `SimHubProperty` is cleared)
3. *(optional)* **`DashboardProfile.cs`** — Add the new value to the `SimHubField` enum
4. *(optional)* **`GameDataSnapshot.cs`** — Add a field, populate it in `FromStatusData()`, and add a case to `GetField()`

To add a new `@internal/` sentinel channel: set the sentinel in `Telemetry.json`, then extend the `switch` in `MozaPlugin.ResolveInternalChannel(string path)` to return the computed `double`. No UI changes — internal channels are filtered out of the mapping grid automatically.

### String-typed channels (out-of-band on sess=0x01)

Channels declared with `compression: "string"` in `Telemetry.json` (`TrackId`, `CarModel`, `SessionTypeName`, `Flag_Name`, `patch/TrackName`, `patch/DisplayTrackName`, vehicle tags, tyre compound types, etc. — 23 in total) **do not bit-pack into the value frame**. They ride a separate transport: `type=0x05` sub-msgs on session 0x01 with `[type=0x05][size_LE u32 = 2+strlen][channel_idx u8][0x80|strlen u8][ASCII strlen bytes]`. Full wire-format reference: [`protocol/sessions/session-0x01-channel-protocol.md`](protocol/sessions/session-0x01-channel-protocol.md); discovery details: [`protocol/findings/2026-05-14-sess01-channel-protocol-and-string-values.md`](protocol/findings/2026-05-14-sess01-channel-protocol-and-string-values.md).

Plumbing inside the plugin:

- **`DashboardProfileStore.BuildMultiStreamProfile` / `BuildProfileFromCatalog` / `BuildPerWidgetProfile`** route any URL whose Telemetry.json compression is `"string"` to `MultiStreamProfile.StringChannels` (separate from the bit-packed `Tiers` list). The wheel does NOT expect string-typed entries in any `type=0x01` tier-def, so a `string` compression must NEVER be assigned a `CompressionTable` entry — that would land the channel in a bit-packed slot the firmware refuses to bind.
- **`ChannelCatalogParser.FindIdxByUrl(url)`** maps a URL string to its 1-based catalog idx as announced by the wheel on b2h sess=0x01 `type=0x04` records. The wheel re-indexes URLs per dashboard, so this lookup is the authoritative source; idx values cannot be hardcoded per channel.
- **`StringValueBuilder.Build(channelIdx, value)`** builds the type=0x05 sub-msg body. Output is byte-exact against the captured PitHouse wire (verified against `imola`→idx7 and `ks_laguna_seca`→idx7 from `bridge-20260514-204307.jsonl`). Max strlen 127 — high bit of the flag byte is the string-type signal, low 7 bits are the length.
- **`TelemetrySender.TickEmitStringValues()`** runs immediately after `TickEmitValueFrames` in the steady-state tick loop. Per-URL state dict (`_stringChannelState`) tracks the last value + last-emit `Environment.TickCount` so the wire only carries a re-send when the value changes, with a 2-second keepalive floor when it doesn't. Records are chunked via `TierDefinitionBuilder.ChunkMessage(session: 0x01, seq: ref _session01OutboundSeq)` so the existing chunk CRC + framing path is reused.
- **Test mode** uses the channel's `TestSignal.StringValue` (assigned at dashboard-load by `TestSignalCatalog.Resolve` when `data_type == "string"`, defaulting to `"DEMO"`). The bit-packed test path's `TestSignalGenerator.Compute` returns `0.0` for string channels — the string is read directly off `TestSignal.StringValue`, not through the numeric pipeline.
- **Live SimHub property wiring is NOT yet implemented.** When the game is running but TestMode is off, the emitter currently re-emits the static `TestSignal.StringValue` ("DEMO" by default) — placeholder until per-channel SimHub property reads land. See the `TODO(2026-05-15)` in `TickEmitStringValues`.

Cadence observed from PitHouse (single-channel `TrackId` sample): bursts of 3–6 retransmits on value-change, 10–30 s silence at steady state. Plugin re-implements this approximately with the 2-second keepalive floor plus event-driven sends on value-change.

### Dashboard Switch State Machine

`DashboardBindingCoordinator.ApplyTelemetryDashboardFromProfile(MozaProfile)` (proxied via `MozaPlugin`) is the single entry point for binding the wheel's displayed dashboard to whatever the current SimHub game profile saved. It fires from `ApplyProfile` at plugin Init / game-switch and from `PollStatus`'s retry loop (driven by the coordinator's `_pending` snapshot — see below). Returns `true` once the apply has resolved (or been permanently dropped), `false` to defer.

**Inputs.** The saved `TelemetryDashboardKey` (`wheel:<id>` / `file:<filename>:<sha1-8>` / `builtin:<name>`) resolves to a *target name*; that name is then looked up in `WheelStateForDiagnostics.ConfigJsonList` (the wheel's alphabetical dashboard library) to find a *target slot* (uint32, the kind=4 frame index). All three key kinds funnel through the same slot lookup — previously only `wheel:` keys emitted kind=4, which silently left the wheel pointing at the prior game's dashboard when a `file:`/`builtin:` profile was activated.

**Readiness gate (defer if not met).** Apply defers and returns `false` when:
1. `sender == null` — sender hasn't been constructed yet (plugin Init has fired but `StartTelemetryIfReady` hasn't created the sender).
2. `!sender.IsActive` — sender exists but is in `Starting`/`Preamble`/`Idle`. A kind=4 emitted before preamble completes is silently dropped by the wheel.
3. `sender.IsInSilenceCooldown` — a previous kind=4 went out on the wire within the silence window (`SilenceGate.StopReopenSilenceMs` = 11 s, or `HotSwitchCooldownMs` = 200 ms in hot-renegotiation mode); the wheel's sess=0x09 binding-state timeout is still running and another emit would race the in-flight restart.
4. `state == null || state.ConfigJsonList.Count == 0` — `ConfigJsonClient.LastState` hasn't produced its first parse yet (or every chunk dropped). The `_cachedLastState` static fallback fills this in across plugin reloads in the same SimHub process so a cold-start chunk drop doesn't leave the apply blind.

`PollStatus` retries via the coordinator's `_pending` field — a `PendingApply` record (`Key` / `DeadlineUtc` / `RetryCount` / `LastRetryWarnUtc`) protected by `_stateLock` so the PollStatus, dispatcher, and profile-change threads can read/write it atomically. The deadline is `PendingProfileKeyTimeout = 5 min`; `_lastApplyDeferReason` (also lock-guarded) throttles the per-defer Debug log to one line per state transition AND surfaces via `PendingDashboardApplyDescription` so the UI status label reads e.g. `Switching dashboard… (sender=not-Active cooldown=False) — N frames sent` instead of a bare "Switching dashboard…". The retry path increments `RetryCount` each tick and emits a Warn line every `RetryWarnInterval = 30 s` so a silent multi-minute wait doesn't pass unnoticed; timeout escalates from the prior Info log to a Warn line including the retry count. All `_pending` mutations after `ApplyTelemetryDashboardFromProfile` returns use `ReferenceEquals(_pending, snap)` so a concurrent `SetPendingDashboardKey` / `ClearPendingDashboardKey` from another thread cannot be overwritten.

**Apply path (slot resolved, gate passed).** Two paths based on whether the wheel is already on the target slot:

1. **Slot already bound** (`sender.WheelReportedSlot == slot` OR `sender.LastEmittedKind4Slot == slot`). The wheel itself emits a `04 <slot:u32> ...` record on sess=0x02 b2h after every dashboard switch — that's wire-level ground truth, parsed by `MaybeUpdateWheelReportedSlot` and exposed as `WheelReportedSlot`. The static `LastEmittedKind4Slot` survives plugin reload, so game-switching between profiles that share a dashboard hits this path on the first apply. Skip: no kind=4 emit, no pipeline action. Logs `no wire action needed`.
2. **Slot mismatch** — emit kind=4 to the target slot via `OnDashboardSwitched((uint)slot)` → `SwitchToProfile` → `SendDashboardSwitch`. With `EnableHotRenegotiation` (default), no Stop+Start cycle; the tick handler emits a paced multi-emission tier-def burst echoing the wheel's END marker (see [`protocol/tier-definition/handshake.md`](protocol/tier-definition/handshake.md) § In-game dashboard switch).

**Wheel-initiated switch handling** (`MaybeUpdateWheelReportedSlot` + `TelemetrySender.WheelInitiatedSwitch` event). When the user presses a wheel-side dash-cycle control, the wheel emits its own b2h kind=4 with the new slot (preceded by a `b8 AA BB` event — see [`protocol/devices/wheel-0x17.md`](protocol/devices/wheel-0x17.md) § Group 0x43). The sender's `MaybeUpdateWheelReportedSlot` detects this (new slot ≠ `_lastEmittedKind4Slot`, `prevSlot >= 0`), updates `_lastEmittedKind4Slot` to suppress echo re-trigger, arms the hot-reneg burst WITHOUT emitting another kind=4 (wheel already sent its own), and raises `WheelInitiatedSwitch` with the new slot. `MozaPlugin.OnWheelInitiatedSwitch` resolves slot → dashboard name via `WheelStateForDiagnostics.ConfigJsonList`, updates `ActiveTelemetryProfileName`, calls `ApplyTelemetrySettings` (stages the new profile in the sender for the queued tier-def emissions), persists settings, raises `DashboardSelectionChanged` so the UI dropdown reflects the wheel's choice. Same `TickGrowSubscriptionIfCatalogStable` burst path runs the 4-emission tier-def burst.

**Catalog re-sync probe.** `TelemetrySender` schedules a probe (single kind=4 to the wheel's currently-bound slot, deferred 800 ms) whenever `BuildTierDefinitionMessage` finds unbound channels in the wheel's catalog — symptom of an incomplete cold-start advertisement. Throttle (~8 s between probes) and the has-fired memo live on `Lifecycle.CatalogResyncProbe`; `ScheduleCatalogResyncProbe` calls `IsThrottleClear(now)` early-out then `MarkFired(now)` only when it commits to emit, so the wheel-on-target shortcut leaves `HasFired` false. The probe nudges the wheel into re-advertising channels; the subsequent tier-def emission (with the wheel's freshly-pushed END marker) is what commits the binding.

**Silence gates** (in `Lifecycle.SilenceGate`, both `DateTime.UtcNow.Ticks` snapshots on static fields):
- `MarkStopped` — called at the end of every `Stop()`. `StartInner` reads `LastStopUtcTicks` and sleeps `RemainingStopReopenWaitMs(preStopTicks)` (= `max(0, StopReopenSilenceMs - elapsed)`, cap 11 s) before opening sessions. Load-bearing for the wheel's sess=0x09 handshake even on cold start (removing this empirically breaks the handshake — the wheel's internal session/binding-state timeout needs ~11 s of host quiet).
- `MarkSwitchEmitted` — called only when `SendDashboardSwitch` actually emits kind=4 on the wire. Drives `IsInSilenceCooldown` (the UI grayout signal and the `SendDashboardSwitch` double-click guard); the gate width is `HotSwitchCooldownMs` (200 ms) when hot-renegotiation is enabled, else `StopReopenSilenceMs` (11 s). UI deliberately doesn't gray out on every `Stop()` — only when a real switch is in flight.

**Per-plugin-instance tracking, reset semantics:**
- `_lastEmittedKind4Slot` (static, in `TelemetrySender`) — survives plugin reload within a SimHub process (game switch). Reset by `ResetBindingTracking()` (called from `MozaPlugin.ResetWheelDetection` on wheel hot-swap).
- `_wheelReportedSlot` (instance) — derived from wheel b2h type-04 records, accumulates from sess=0x02 acks during preamble + post-kind=4 echoes. Reset on hot-swap.
- `_cachedLastState` (static, in `ConfigJsonClient`) — the wheel-library cross-instance fallback. Cleared by `HardReset()` from `ResetBindingTracking`.
- `_lastAppliedDashboardKey` (instance, in `DashboardBindingCoordinator`, lock-guarded by `_stateLock`) — short-circuits repeated `ApplyTelemetryDashboardFromProfile` calls for the same key in one plugin instance (e.g., multiple `OnProfileChanged` events). Reset on `Init` (via `ClearLastAppliedDashboardKey`) and `ResetWheelDetection`. All reads go through `GetLastAppliedKey()`, all writes through `SetLastAppliedKey(key)`, so the coordinator never touches the field outside the lock.

**UI gating** (`MozaWheelSettingsControl.RefreshTelemetryStatus`). The dashboard combo + test buttons are enabled iff `sender != null && sender.IsActive && !sender.IsInSilenceCooldown && !_plugin.IsPendingDashboardApply`. The `IsPendingDashboardApply` check keeps the UI locked across the entire game-switch transient instead of flickering enabled during the brief moments between preamble end → probe kind=4 → cooldown clear → `RestartForSwitch`'s Stop. The status label surfaces the current state (`Connecting to wheel…`, `Switching dashboard… (sender=not-Active cooldown=False)`, `Sending — N frames sent`) — the parenthesised defer reason comes from `_plugin.PendingDashboardApplyDescription` so the user can see *why* a switch is stuck instead of staring at a bare "Switching dashboard…". `PopulateDashboardCombo` matches `state.ConfigJsonList[i]` against `selectedName` with `StringComparison.OrdinalIgnoreCase` to align with `DashboardBindingCoordinator`'s key/name lookups; an exact-case `==` previously left the combo stuck on the prior game's dashboard whenever the saved profile name's casing differed from the wheel's configJsonList entry. **Repopulation triggers** are (a) first arrival of wheel state (`!_dashComboFromWheelState && state.ConfigJsonList.Count > 0`) AND (b) `sender.WheelReportedSlot != _lastPopulatedWheelSlot`. The slot-change trigger catches the startup case where the plugin emits kind=4 to apply the saved profile's dashboard before the wheel's b2h type-04 echo lands — without it the dropdown shows the wheel's pre-switch dash forever even though the wheel switched correctly. `PopulateDashboardCombo` snapshots `sender.WheelReportedSlot` into `_lastPopulatedWheelSlot` at the end of each population pass.

**Total time, game switch → settled (observed):**
- Game switch: ~5-15 s — cold-start preamble + apply; depends on wheel handshake speed.
- In-game dashboard switch (hot-reneg default): ~1-2 s for the tier-def burst to land + wheel to bind. UI ungrayed after `HotSwitchCooldownMs` (200 ms).
- Legacy `EnableHotRenegotiation = false` path: ~35 s — Preamble + probe cooldown 11 s + apply `RestartForSwitch` 11 s + new preamble. Stop+Start path is only reached for game-switch, wheel hot-swap, or sess=0x02 engagement-watchdog exhaustion; the 11 s silence gates are the wheel's sess=0x09 interlock and can't be shortened without breaking the handshake.

**Diagnostic tools** (in `tools/`):
- `wire-dashboard-switches <jsonl>` — lists every host-emitted kind=4 + sess=0x09 lifecycle + Δ to next devinit. Flags rapid (Δ<1 s) emit clusters.
- `wire-sess-lifecycle <jsonl> [sess...]` — per-session frame chronology for the named sessions.
- `wire-kind4-response <jsonl>` — for each host kind=4, shows the wheel's b2h reply window so you can read the type-04 slot echoes directly.

### Implementation plan — FSR / RS21-D03 (group `0x42`) display-telemetry wheel

**Status: PLAN (not implemented).** This wheel variant uses a *fundamentally different* telemetry transport from every other display wheel the plugin supports. The entire pipeline documented above — catalog advertisement, tier-definition, session 0x02, bit-packed `0x43`/`7D 23` value frames, the dashboard-switch state machine — **does not apply**. Captured behaviour and the full wire decode are in [`protocol/devices/wheel-0x17.md`](protocol/devices/wheel-0x17.md) § Group `0x42` and [`../usb-capture/CAPTURES.md`](../usb-capture/CAPTURES.md) § `fsr1/`. Identified by model-name reply **`FSR`** (hw `RS21-D03-HW FW-C`, sw `RS21-D03-MC FW`), hub-attached behind an `S03 HUB`. This is the **FSR V1** — a distinct, older wheel from the **FSR V2** (`W13`), which uses the standard tier-definition path and is already modelled by the `W13` entry in `WheelModelInfo.KnownModels`.

What this firmware does instead: the host pushes **pre-computed display field values** to the wheel (`0x17`) as fixed-schema **group `0x42`** records — `7E <len> 42 17 [type] [b1] [b2] <fixed-layout data> <csum>` — at ~28 Hz. `type` (b0) selects a fixed-length record; PitHouse enumerates types `01`–`12` once at startup with zero payloads, then fills a subset live. There is **no channel catalog**, so each dashboard's field layout is baked, not negotiated. **The wheel has multiple built-in dashboards, and the active `0x42` record `type` tracks the displayed one** (`FSR1 with game` emits type `02` throughout; `FS1 multiple changes` shifts `0e`→`09`→`06` as dashboards are cycled — see [`protocol/devices/wheel-0x17.md`](protocol/devices/wheel-0x17.md) § Group 0x42). So the field set is *fixed per dashboard*, not globally — the user still picks which SimHub property feeds each field (their stated requirement), but the mapping is per-record-type, and the host must know which dashboard/type is currently shown to emit the right record. No dedicated dashboard-switch opcode was seen; switches coincide with a full group `0x40` config re-sweep and, in these captures, appear host-driven (no wheel→host active-dashboard signal observed).

#### Phase 0 — Finish the wire decode (PREREQUISITE — gates Phases 2–4)

Only the *structure* is proven. The **per-field channel semantics are not decoded** for the live record types (`04/06/09/0b/0d/0e`); for the gameplay-hot type `02` we have only *candidate* mappings derived by time-aligning fields against the group-`0x3F` RPM-LED bitmask within one capture (`[14]`≈RPM/load, `[15]`=engine-running flag 0/75, `[17]`=gear 0–5, `[6]/[8]/[10]/[12]`=a 4-equal-element wheel array). None of these is a confirmed channel identity, and no encoding (width/endianness/scale/units) is established. Before any frame can be *filled* meaningfully, nail down, per record type:

- each field's byte offset, width, endianness, signedness, and scale → SimHub quantity it drives;
- the meaning of `b1` (per-type, `0x00` on the zero frame, type-specific non-zero when populated — candidate: populated-field count/validity);
- whether `b2` selects sub-streams/modes (type `02`: `00`/`20`; type `0e`: `00`/`80`/`c0`);
- **which record `type` corresponds to which built-in firmware dashboard/page**, and whether the wheel signals the active page on b2h (check for any wheel→host `0xC2`/`0x42` frames — none were looked for in the current captures) — this determines *what the host should emit* at runtime.

Decode method (derive from PitHouse/captures, per project rules — no inventing): (a) a controlled capture where known telemetry is driven (fixed speed/rpm/gear ramps) and aligned to field bytes; (b) replay candidate values to the physical wheel and read the screen; or (c) binary RE of PitHouse's `0x42`-builder in `MOZA Pit House.exe` (anchors: strings `Steering Wheel/FSR-Official.json`, `Protocol - FSR`; a Ghidra workflow already exists — see `usb-capture/pithouse-re.md`). Known so far (from `MOZA Pit House/bin`): the channels are the standard `Telemetry.json` `v1/gameData/…` set, **but `0x42` uses a byte-aligned firmware encoding, not the tier-def bit-packing** (so reuse Telemetry.json channel *identities/scales* but not its bit widths); the FSR V1 LED preset (`default_preset_library.rcc` → `FSR-Official.json`) defines only LEDs/flags (drives the RPM bar from `CarSettings_CurrentDisplayedRPMPercent`), **not** the `0x42` screen layout — that is firmware-baked and must come from route (a)/(b)/(c). Until this is done the rest of the plan is unbuildable beyond scaffolding.

#### Phase 1 — Variant detection & routing

### Key Protocol Details

- Message format: `[0x7E] [length] [request_group] [device_id] [command_id...] [payload...] [checksum]`
- Response parsing: toggle bit 7 of request_group, swap nibbles of device_id, then match command_id (with 0xFF wildcards)
- Read messages use a zero-filled payload of the declared byte width (matches boxflat's `prepare_message`). Some wheels silently drop reads with a non-zero payload, even though the protocol nominally ignores payload bytes on reads
- All multi-byte integers are big-endian; floats are byte-reversed
- `MozaSerialConnection` uses two write lanes. **One-shot FIFO** (`_oneShotQueue`) carries session traffic, settings writes, and probes — ordered, paced at 4 ms between consecutive one-shots (boxflat's proven timing; Moza bases drop rapid settings writes like the 30+ `ApplyProfile` burst). Under bandwidth pressure the `WriteBudget` token-bucket extends the 4 ms gate further (still order-preserving FIFO, never drops) so the wheel can drain its receive buffer before we pile on more. Pacing uses `Stopwatch.GetTimestamp()` (monotonic 64-bit) so the gate stays correct across long uptime; the previous `Environment.TickCount` int math wraps every ~24.8 days. **Stream slots** (`_streamSlots` — one per `StreamKind`: tier dash 0–7, enable, sequence, mode, ab9-engine-vibration, ab9-engine-pulse, ab9-trigger-a, ab9-trigger-rpm, ab9-trigger-extra, ab9-low-rate) carry periodic telemetry — latest-wins coalescing, unpaced and **never bandwidth-gated**. `SendStream(kind, bytes)` overwrites any predecessor in the same slot via `Interlocked.Exchange`, so a stale tier-0 frame can never reach the wire if a newer one was already generated. `WriteLoop()` drains one-shots first (paced only when the previous write was also a one-shot), then sweeps stream slots back-to-back. A pooled stuff buffer emits each frame in a single `_port.Write()` call via `MozaProtocol.StuffFrame`. `FlushPendingWrites()` drops both lanes + calls `_port.DiscardOutBuffer()` — used by `TelemetrySender.Pause()` so Test-Stop halts wire traffic within ~1 ms instead of bleeding through the 16 KB OS buffer (~1.4 s at 115200 baud). Pre-2026-04-22 (queue-drift fix): single `ConcurrentQueue`, unconditional 4 ms pacing between every dequeue, per-byte `_port.Write(msg, i, 1)` for each stuffed body byte. Under steady telemetry (3+ frames/tick × 4 ms pacing) drain time hit the 30 ms tick budget, so any OS jitter grew backlog until the displayed dashboard values lagged the game by hundreds of ms over a single lap. **Stream-lane gating** (the `WriteBudget.MayDrainStreams()` check that suppressed stream writes when the rolling 1-s window crossed the budget) was removed 2026-05-09 — the gate held value frames for ~700 ms-1 s during routine bursts and starved the wheel into dropping configJson chunks, while latest-wins coalescing already provides natural backpressure (fresh `SendStream` overwrites pending slots before the writer drains them) and `SerialPort.Write` blocks when the OS write buffer fills, providing a hard physical-layer gate. **Read path** unchanged: bulk-read thread polls `BytesToRead` at 2 ms, pulls whatever's available via `Read(buf, 0, avail)` into a 64 KB in-memory rx buffer, parses as many complete frames as possible per pass. Per-byte `ReadByte()` was replaced 2026-04-22 because its ~100 µs per-call cost under Wine starved the reader during bursts and lost chunks. Wine's blocking `Read(buf, 0, full_count)` does NOT return early when fewer bytes are available — it hangs — so the BytesToRead guard is load-bearing
- 0x7E byte stuffing: every `0x7E` byte in a frame's body (group, device, payload, checksum) is doubled on the wire. `MozaProtocol.StuffFrame(frame, dest)` writes the start+len header raw then walks body bytes emitting each and appending an extra `0x7E` on match; `StuffedFrameSize(frame)` pre-computes the output size so `WriteLoop`'s pooled stuff buffer can be grown on demand. The single-shot stuffed write replaced per-body-byte `_port.Write(msg, i, 1)` calls in 2026-04-22 — the old path issued 10–32 syscalls per frame, compounding Wine's per-call overhead. `ReadLoop()` reverses stuffing: after reading the start+len header, it reads N+3 decoded bytes, collapsing any `0x7E 0x7E` pair back to a single `0x7E`; an unpaired `0x7E` marks the start of a new frame and the current frame is abandoned. Without this, payload 0x7Es (common in zlib-compressed session data and seq counters) desync the read stream. **Checksum must also account for stuffing** — the wire-level sum includes the duplicated `0x7E` bytes, so `CalculateWireChecksum()` adds `count(0x7E in body) × 0x7E` to the raw sum. Pre-2026-04-22 the plugin's `CalculateChecksum` omitted this term and dropped ~20% of zlib-bearing session chunks (configJson state, dashboard uploads) whose compressed payloads contain 0x7E bytes. See `docs/protocol/wire/checksum.md`
- Session close frame: the type=0x00 end marker carries a 6-byte payload `7C 00 [session] 00 [ack_lo] [ack_hi]` with `len=6`. A 4-byte payload advertised as len=6 causes the wheel/sim to over-read into the next frame and kill the read loop. `SendSessionClose()` zero-fills the ack_seq bytes when simply reclaiming a stale session

### Dependencies

- **NuGet:** `Microsoft.NETFramework.ReferenceAssemblies.net48`, `Newtonsoft.Json`, `log4net`
- **Runtime (Windows only):** `Microsoft.Win32.Registry` (in `mscorlib`, no extra reference needed) — used by `MozaPortDiscovery` to enumerate MOZA USB CDC ports from `HKLM\SYSTEM\CurrentControlSet\Enum\USB`. The serial-probe fallback consults the registry lookup per port: ports the registry already pinned to a non-matching MOZA PID are skipped (no serial bytes), unclassified ports are probed, and the probe is short-circuited entirely when every COM port has a classified non-matching PID. Hard opt-out: `DisableSerialProbeFallback` setting
- **SimHub DLLs** (checked into `libs/SimHub/`, reference-only, not packaged): `SimHub.Plugins.dll`, `GameReaderCommon.dll`, `SimHub.Logging.dll`, `SerialDash.dll`, `BA63Driver.dll`, `HidSharp.dll`. A daily GitHub Actions workflow automatically creates PRs when new SimHub versions are released.

**Important:** The SimHub DLLs in `libs/SimHub/` must match the runtime SimHub version. The PluginSdk ships older DLLs that may be missing interface members added in newer releases, causing `TypeLoadException` at runtime. Always update from the actual SimHub installation directory.
